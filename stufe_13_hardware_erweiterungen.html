1. RAM Expansion Unit (REU 1750) — 512 KB aus der Steckdose

Die REU 1750 ist die mächtigste Erweiterung für den C64 aus der Commodore-Ära. Sie steckt in den Expansion-Port und bringt bis zu 512 KB zusätzlichen RAM mit — und was noch wichtiger ist: einen eigenen DMA-Controller. Mit diesem Direct Memory Access kann die REU Daten zwischen sich selbst und dem C64-RAM übertragen, ohne dass der 6510 jeden einzelnen Byte selbst kopieren muss.

1.1 REU-Architektur: Was ist DMA und warum ist es so schnell?

Beim normalen Kopieren läuft jedes Byte durch den Prozessor: LDA $ADDR (2–4 Zyklen) + STA $DEST (3–5 Zyklen) = mindestens 5–9 Zyklen pro Byte. 256 Bytes zu kopieren kostet ca. 1792 Zyklen.

Die REU besitzt dagegen einen eigenen Adressbus. Beim DMA-Transfer nimmt der REU-Controller den C64-Bus in Beschlag und schaufelt Daten direkt zwischen C64-RAM und REU-RAM — ein Byte pro Bustakt, mit 2 Zyklen Overhead für Start und Stop. 256 Bytes: 256 + 2 = 258 Zyklen. Sieben Mal schneller als Software-Kopie.

Eine REU 1750 hat 512 KB. Die REU 1764 hat 256 KB, die REU 1700 nur 128 KB. Das Modell erkennt man beim Programmstart via Statusregister (mehr dazu in Abschnitt 1.4).

1.2 Register-Übersicht $DF00–$DF0B

Die REU wird über elf Register im I/O-Bereich angesprochen, sichtbar wenn kein Cartridge eingesteckt ist (oder wenn das Cartridge diese Adressen freigibt):

AdresseNameR/WFunktion
$DF00STATUSRStatus: Bit 7=IRQ-ausstehend, Bit 6=End-of-Block, Bit 5=Fault, Bit 4=Size; W: Befehl auslösen
$DF01COMMANDWBefehlsregister: $01=Stash (C64→REU), $02=Fetch (REU→C64), $03=Swap, $04=Verify; Bit 4=Autoload; Bit 7=Sofortausführung
$DF02C64BASE_LOR/WC64-Basisadresse niederwertiges Byte
$DF03C64BASE_HIR/WC64-Basisadresse hochwertiges Byte
$DF04REUBASE_LOR/WREU-Adresse Byte 0 (niedrigstes)
$DF05REUBASE_MIR/WREU-Adresse Byte 1 (mittleres)
$DF06REUBASE_HIR/WREU-Adresse Byte 2 (höchstes, Bits 0–2 für Bank 0–7)
$DF07LENGTH_LOR/WTransferlänge niederwertiges Byte
$DF08LENGTH_HIR/WTransferlänge hochwertiges Byte (0000 = 65536 Bytes)
$DF09INTMASKR/WInterrupt-Maske: Bit 7=IRQ freigeben, Bit 6=End-of-Block IRQ, Bit 5=Verify-Fehler IRQ
$DF0AADDRCTRLR/WAdress-Kontrolle: Bit 7=C64-Adresse fixieren, Bit 6=REU-Adresse fixieren

1.3 Transfertypen

Das COMMAND-Register ($DF01) kennt vier Transferrichtungen:

  • $01 — Stash (C64→REU): Daten aus C64-RAM in REU schreiben
  • $02 — Fetch (REU→C64): Daten aus REU in C64-RAM laden — der häufigste Fall
  • $03 — Swap: Inhalt von C64-RAM und REU tauschen (nützlich für Doppelpuffer)
  • $04 — Verify: Prüft ob C64-RAM und REU identisch sind (kein Transfer, nur Vergleich)

Durch Setzen von Bit 4 im COMMAND-Register aktiviert man den Autoload-Modus: Nach dem Transfer werden die Adressregister automatisch zurückgesetzt, sodass ein erneuter Transfer sofort vom Anfang beginnt. Bit 7 in $DF01 löst den Transfer sofort aus.

1.4 REU-Adressierung: 24-Bit-Adressraum

Die REU hat einen 24-Bit-Adressraum — 16 MB theoretisch, in der Praxis 128 KB–512 KB je nach Modell. Die drei Register $DF04/$DF05/$DF06 bilden zusammen die 24-Bit REU-Adresse:

REU 24-Bit Adressraum: $DF06 [Bits 2-0] = Bank (0-7) Bits 23-16 $DF05 = Middle-Byte Bits 15-8 $DF04 = Low-Byte Bits 7-0 REU 1700 (128KB): Bänke 0-1 ($00000-$1FFFF) REU 1764 (256KB): Bänke 0-3 ($00000-$3FFFF) REU 1750 (512KB): Bänke 0-7 ($00000-$7FFFF)

2. REU-DMA in der Praxis — Blöcke in Mikrosekunden

2.1 Einen Transfer starten: die sieben Schritte

Ein REU-Transfer wird immer gleich eingeleitet:

  1. C64-Basisadresse in $DF02/$DF03 setzen
  2. REU-Basisadresse in $DF04/$DF05/$DF06 setzen
  3. Transferlänge in $DF07/$DF08 setzen (0 = 65536 Bytes)
  4. Optional: Interrupt-Maske in $DF09 konfigurieren
  5. Optional: Adress-Kontroll-Register $DF0A konfigurieren
  6. COMMAND-Register $DF01 mit Befehl UND Bit 7 gesetzt schreiben
  7. Warten bis $DF00 Bit 6 (End-of-Block) gesetzt ist — oder via IRQ

2.2 Universelle REU-Transfer-Subroutine

; reu_transfer — Universelle DMA-Subroutine
; Parameter werden über Zero-Page übergeben:
;   REU_C64_LO  ($10): C64-Basisadresse Low
;   REU_C64_HI  ($11): C64-Basisadresse High
;   REU_ADDR_LO ($12): REU-Adresse Low
;   REU_ADDR_MI ($13): REU-Adresse Middle
;   REU_ADDR_HI ($14): REU-Adresse High (Bank in Bits 0-2)
;   REU_LEN_LO  ($15): Transferlänge Low
;   REU_LEN_HI  ($16): Transferlänge High
;   REU_CMD     ($17): Befehl ($81=C64→REU, $82=REU→C64, $83=Swap)
; Verändert: A

REU_STATUS  = $DF00   ; Status-Register
REU_COMMAND = $DF01   ; Befehl (Bit 7 = sofort auslösen)
REU_C64_LO  = $10    ; ZP: C64-Adresse Low
REU_C64_HI  = $11    ; ZP: C64-Adresse High
REU_ADDR_LO = $12    ; ZP: REU-Adresse Low
REU_ADDR_MI = $13    ; ZP: REU-Adresse Mid
REU_ADDR_HI = $14    ; ZP: REU-Adresse High/Bank
REU_LEN_LO  = $15    ; ZP: Länge Low
REU_LEN_HI  = $16    ; ZP: Länge High
REU_CMD     = $17    ; ZP: Befehl

reu_transfer:
    lda REU_C64_LO
    sta $DF02            ; C64-Adresse Low
    lda REU_C64_HI
    sta $DF03            ; C64-Adresse High
    lda REU_ADDR_LO
    sta $DF04            ; REU-Adresse Low
    lda REU_ADDR_MI
    sta $DF05            ; REU-Adresse Mid
    lda REU_ADDR_HI
    sta $DF06            ; REU-Adresse High (Bank)
    lda REU_LEN_LO
    sta $DF07            ; Länge Low
    lda REU_LEN_HI
    sta $DF08            ; Länge High
    lda REU_CMD
    ora #$80             ; Bit 7 = sofort ausführen
    sta $DF01            ; Transfer starten!
.wait:
    lda $DF00            ; Status lesen
    and #$40             ; Bit 6 = End-of-Block
    beq .wait            ; Warten bis fertig
    rts

Der DMA-Transfer blockiert den 6510 vollständig — er kann währenddessen keine Interrupts bearbeiten und keinen Code ausführen. Für kurze Transfers (<256 Bytes) ist das kein Problem. Für große Transfers (mehrere KB) sollte man den IRQ davor sperren (SEI) und danach wieder freigeben (CLI).

2.4 REU-Erkennung beim Programmstart

; detect_reu — Prüft ob REU vorhanden und ermittelt Kapazität
; Rückgabe: A = 0 (keine REU), 128 (128KB), 256 (=0 = 256KB), 2 (512KB)
; Methode: Schreibe Test-Byte in REU-RAM, lese zurück. Kein REU → Schreiben wirkungslos.

detect_reu:
    lda $DF00           ; Status lesen
    and #$10            ; Bit 4 = Size-Flag (0=512KB, 1=kleinere Modelle)
    sta reu_size_flag
    ; Test: Schreibe $55 ins REU-RAM an Adresse $00000
    lda #$55
    sta test_byte
    lda #<test_byte
    sta $DF02           ; C64-Adresse = &test_byte
    lda #>test_byte
    sta $DF03
    lda #$00 : sta $DF04  ; REU-Adresse = $000000
    lda #$00 : sta $DF05
    lda #$00 : sta $DF06
    lda #$01 : sta $DF07  ; Länge = 1
    lda #$00 : sta $DF08
    lda #$81            ; Stash (C64→REU) + sofort
    sta $DF01
    ; Ändere C64-Byte, dann Fetch zurück
    lda #$AA : sta test_byte
    lda #$82 : sta $DF01  ; Fetch (REU→C64) + sofort
    lda test_byte
    cmp #$55            ; $55 zurückgekommen? Dann REU vorhanden
    bne .no_reu
    lda #1              ; REU gefunden
    rts
.no_reu:
    lda #0
    rts

3. REU-Anwendungen — Screenramer, Musikdaten, Sprite-Frames

3.1 Screen-Buffer im REU — Sofortige Screen-Umschaltung

Der häufigste Anwendungsfall für die REU ist das blitzschnelle Sichern und Wiederherstellen des gesamten Bildschirminhalts. Ein vollständiger C64-Screen besteht aus 1000 Bytes Screen-RAM ($0400–$07E7) plus 1000 Bytes Color-RAM ($D800–$DBE7). Mit einer Software-Kopieschleife dauert das über 10.000 Zyklen — mit der REU sind es zwei DMA-Transfers, zusammen unter 2200 Zyklen.

; screen_save — Sichert Screen-RAM + Color-RAM in REU
; screen_restore — Lädt Screen+Color aus REU zurück
; REU-Layout: $00000-$003E7 = Screen-RAM (1000 Bytes)
;             $003E8-$007CF = Color-RAM  (1000 Bytes)

screen_save:
    ; Screen-RAM ($0400) → REU $00000
    lda #$00 : sta $DF02  ; C64: $0400
    lda #$04 : sta $DF03
    lda #$00 : sta $DF04  ; REU: $000000
    lda #$00 : sta $DF05
    lda #$00 : sta $DF06
    lda #$E8 : sta $DF07  ; Länge = 1000 = $03E8
    lda #$03 : sta $DF08
    lda #$81 : sta $DF01  ; Stash (C64→REU)
    ; Color-RAM ($D800) → REU $003E8
    lda #$00 : sta $DF02
    lda #$D8 : sta $DF03  ; C64: $D800
    lda #$E8 : sta $DF04  ; REU: $0003E8
    lda #$03 : sta $DF05
    lda #$00 : sta $DF06
    lda #$E8 : sta $DF07
    lda #$03 : sta $DF08
    lda #$81 : sta $DF01
    rts

screen_restore:
    ; REU $00000 → Screen-RAM ($0400)
    lda #$00 : sta $DF02
    lda #$04 : sta $DF03
    lda #$00 : sta $DF04
    lda #$00 : sta $DF05
    lda #$00 : sta $DF06
    lda #$E8 : sta $DF07
    lda #$03 : sta $DF08
    lda #$82 : sta $DF01  ; Fetch (REU→C64)
    ; REU $003E8 → Color-RAM ($D800)
    lda #$00 : sta $DF02
    lda #$D8 : sta $DF03
    lda #$E8 : sta $DF04
    lda #$03 : sta $DF05
    lda #$00 : sta $DF06
    lda #$E8 : sta $DF07
    lda #$03 : sta $DF08
    lda #$82 : sta $DF01
    rts

3.2 Mehrere SID-Tunes im REU

Musik-Daten sind typischerweise 4–16 KB groß. Vier verschiedene SID-Tunes gleichzeitig im C64-RAM zu halten ist nicht möglich, aber per REU sehr einfach. Beim Start werden alle Tunes per Disk-Loader in den REU-Speicher geladen (REU-Adresse $10000, $14000, $18000, $1C000). Beim Tune-Wechsel: ein einziger DMA-Fetch ersetzt die Musikdaten im C64-RAM in Millisekunden.

3.3 REU als Sprite-Animations-Puffer

Sprite-Animationen mit vielen Frames benötigen viel Speicher. Ein 12-Frame-Sprite braucht 12 × 63 = 756 Bytes. Für einen vollständigen 64-Frame-Zyklus wären das 4 KB — problemlos im REU unterzubringen. Im IRQ-Handler wird der nächste Animationsframe via DMA in das aktive Sprite-Daten-RAM übertragen.

Für alle REU-Transfers gilt: Die REU-Adresse wird nach dem Transfer automatisch erhöht, die C64-Adresse ebenso. Wenn man die Adresse fixieren möchte (z. B. um dieselbe Adresse wiederholt zu beschreiben), kann man Bit 7 ($C64-Adresse fixieren) oder Bit 6 (REU-Adresse fixieren) in $DF0A setzen.

4. EasyFlash-Cartridge — 1 MB Flash im Steckmodul

EasyFlash ist ein modernes Cartridge-Format für den C64, entworfen von Skoe. Es enthält 1 MB Flash-Speicher aufgeteilt in 64 Bänke à 16 KB (je 8 KB für ROML bei $8000 und 8 KB für ROMH bei $A000/$E000). Durch Bank-Switching kann ein Programm bis zu 1 MB Code und Daten im Cartridge-Speicher ablegen — deutlich mehr als im 64 KB RAM des C64.

4.1 EasyFlash-Architektur

EasyFlash Speicherlayout: 64 Bänke × 16 KB = 1 MB Flash Jede Bank: ROML 8 KB → sichtbar bei $8000-$9FFF ROMH 8 KB → sichtbar bei $A000-$BFFF (Ultimax: $E000-$FFFF) Bank 0: Pflicht-Startbank — enthält RESET/NMI-Vektoren in ROMH Bänke 1-63: Frei verfügbar für Code, Daten, Level-Inhalte

4.2 Bank-Switching-Register

AdresseNameFunktion
$DE00BANKBank-Nummer 0–63 (Bits 0–5). Schreibzugriff schaltet sofort um.
$DE02MODEBit 2: Onboard-LED. Bit 1: GAME. Bit 0: EXROM. Standardmodus: $04 (GAME=1,EXROM=0 = 16KB Cartridge Mode)
$DE04NV (optional)EasyFlash NV: nicht-volatile Speicherung der aktuellen Bankeinstellung über Reset hinaus

4.3 Bank-Switching in ACME

; bank_switch — EasyFlash-Bank aktivieren
; Eingabe: A = Banknummer (0-63)
; GAME/EXROM auf 16KB-Cartridge-Modus ($04)

EF_BANK  = $DE00   ; Bank-Register
EF_MODE  = $DE02   ; Mode-Register

bank_switch:
    sta EF_BANK          ; Bank 0-63 aktivieren
    lda #$04             ; GAME=1, EXROM=0 → 16KB Cartridge Mode
    sta EF_MODE          ; Modus setzen
    rts

; Beispiel: Code aus Bank 3 aufrufen
; In Bank 3 liegt Routine 'gfx_init' bei $8200
call_bank3_routine:
    lda #3
    jsr bank_switch      ; Bank 3 aktivieren
    jsr $8200            ; Routine in ROML aufrufen
    lda #0
    jsr bank_switch      ; Zurück zu Bank 0
    rts

4.4 .crt-Image-Erstellung

Ein EasyFlash-Cartridge-Image hat das CRT-Format. Es besteht aus einem 64-Byte-Header gefolgt von CHIP-Chunks, einen pro Bank und Bereich (ROML/ROMH). Das Tool cartconv (Teil des VICE-Pakets) erstellt CRT-Images aus Binärdateien.

Typischer Ablauf mit cartconv:

cartconv -t EasyFlash \ -b 0 -l bank0_roml.bin -r bank0_romh.bin \ -b 1 -l bank1_roml.bin -r bank1_romh.bin \ ... \ -o mein_cartridge.crt
EasyFlash eignet sich besonders für Programme über 50 KB, große Datentabellen (vollständige Sinustabellen, Level-Daten) oder Demos mit vielen Parts, die nicht alle gleichzeitig im C64-RAM Platz haben.

5. CIA2-Userport — 8-Bit-Parallel-I/O an $DD01

Der Userport des C64 ist ein 24-poliger Edge-Connector auf der Rückseite des Gehäuses. Über ihn kann man eigene Hardware anschließen — 8 Datenleitungen plus Handshaking-Leitungen, Stromversorgung und GND. Gesteuert wird er durch CIA2 bei $DD00–$DDFF.

5.1 Userport-Pinbelegung (relevante Pins)

PinNameCIA2-RegisterFunktion
C–JPB0–PB7$DD01 (Port B)8 bidirektionale Datenleitungen
B/FLAG2$DD0D Bit 4Handshake-Eingang (Edge-Trigger)
MPA2$DD00 Bit 2Handshake-Ausgang
A,NGND—Masse
2+5V—+5 Volt (max. 100mA entnehmen)

5.2 CIA2 konfigurieren: Direction-Register $DD03

Bevor man den Userport nutzen kann, muss man festlegen, welche Pins Ausgang und welche Eingang sind. Das Data-Direction-Register $DD03 steuert dies bitweise: Bit = 1 → Ausgang, Bit = 0 → Eingang.

5.3 LED-Ausgabe und Taster-Eingabe

; userport_leds — Schreibt LED-Muster an Userport-Pins PB0-PB7
; Eingabe: A = LED-Bitmuster (Bit 0 = LED 1, Bit 7 = LED 8)
; userport_read_buttons — Liest Tasterstatus mit 3-Frame-Debounce
; Rückgabe: A = gedrückte Tasten (0=gedrückt da Pull-Up)

CIA2_PORTB  = $DD01   ; Port B Datenregister
CIA2_DDRB   = $DD03   ; Port B Direction (1=Ausgang)

userport_init_output:
    lda #$FF             ; Alle Pins als Ausgang
    sta CIA2_DDRB
    lda #$00             ; Alle LEDs aus
    sta CIA2_PORTB
    rts

userport_leds:
    sta CIA2_PORTB       ; LED-Muster direkt schreiben
    rts

userport_init_input:
    lda #$00             ; Alle Pins als Eingang
    sta CIA2_DDRB
    rts

btn_prev:   !byte $FF  ; Vorheriger Zustand
btn_count:  !byte 0   ; Debounce-Zähler (Frames)

userport_read_buttons:
    lda CIA2_PORTB       ; Port lesen (0=gedrückt wegen Pull-Up)
    cmp btn_prev
    beq .same            ; Gleich wie vorher?
    sta btn_prev         ; Neuer Zustand
    lda #0
    sta btn_count        ; Debounce-Zähler reset
    lda #$FF             ; Noch kein stabiles Ergebnis
    rts
.same:
    inc btn_count
    lda btn_count
    cmp #3               ; 3 Frames stabil = entprellt
    bcc .not_stable
    lda btn_prev         ; Stabilen Zustand zurückgeben
    rts
.not_stable:
    lda #$FF
    rts
Die Taster-Eingänge am Userport verwenden interne Pull-Up-Widerstände. Ein gedrückter Taster zieht den Pin auf GND → gelesen wird 0. Nicht gedrückt → gelesen wird 1. Die Logik ist also invertiert!

6. VIC-II Banking — 64 KB Video-RAM voll nutzen

Der VIC-II Grafikchip sieht nur einen 16-KB-Bereich des Speichers — seinen Bank. Dieser Bank kann in einem von vier möglichen Bereichen des C64-RAMs liegen. Durch bewusstes Planen der Speicherbelegung kann man alle 64 KB für Video-Daten nutzen, indem man verschiedene Objekte (Charset, Bitmap, Sprites) in verschiedene VIC-Bänke legt und zwischen ihnen umschaltet.

6.1 Die vier VIC-Bänke

VIC-II Banking — CIA2 $DD00 Bits 1-0 (invertiert!): CIA2[1:0] VIC-Bank Bereich Besonderheit ───────────────────────────────────────────────────── 11 Bank 0 $0000-$3FFF DEFAULT — hier startet der C64 10 Bank 1 $4000-$7FFF Kein ROM-Overlay 01 Bank 2 $8000-$BFFF Kein ROM-Overlay 00 Bank 3 $C000-$FFFF Kein ROM-Overlay ACHTUNG: Bits sind INVERTIERT! 11 = $0000 (Bank 0), 00 = $C000 (Bank 3) VIC-Bank setzen: LDA CIA2_PORT ; $DD00 lesen AND #%11111100 ; Bits 0+1 löschen ORA #bank_bits ; Bank-Bits setzen (invertiert) STA CIA2_PORT

6.2 Charset-ROM-Fenster beachten

In Bank 0 und Bank 2 gibt es eine wichtige Einschränkung: Bei $1000–$1FFF (Bank 0) und $9000–$9FFF (Bank 2) sieht der VIC-II immer den eingeblendeten Zeichensatz-ROM, unabhängig davon was im RAM steht. Diese Bereiche eignen sich daher nicht für eigene Grafik-Daten.

6.3 $D018 — VIC-Bank-intern: Screen und Charset platzieren

Innerhalb des aktiven VIC-Banks steuert das Register $D018, wo Screen-RAM und Charset-ROM/RAM liegen:

$D018 BitsFunktionAdress-Formel
Bits 7–4Screen-RAM Basis (0–15)Bankbasis + Wert × $0400
Bits 3–1Charset/Bitmap Basis (0–7)Bankbasis + Wert × $0800
Bit 0Bitmap-Modus Offset0=$0000, 1=$2000 innerhalb der Bitmap-Basis

6.4 Vollständige VIC-Banking-Konfiguration

; vic_bank_setup — Vollständige VIC-II Bank-Konfiguration
; Ziel-Layout in Bank 1 ($4000-$7FFF):
;   Screen-RAM:   $4000 (Offset 0 × $0400 = $0000 in Bank)
;   Charset:      $4800 (Offset 1 × $0800 = $0800 in Bank) [kein ROM-Overlay in Bank 1]
;   Sprite-Daten: $4FC0 (Sprite-Pointer bei $4FF8: Wert $3F = $4000 + $3F*$40 = $4FC0)
;   Bitmap:       $6000 (Offset 4 × $2000 = Bitmap-Basis $4000+$2000 = $6000, Bit0=1)

CIA2_PORT  = $DD00
VIC_CTRL1  = $D011
VIC_MEMCTL = $D018

vic_bank_setup:
    ; CIA2: VIC-Bank 1 aktivieren ($4000-$7FFF)
    ; Bank 1 = CIA2-Bits 1:0 = 10 binär
    lda CIA2_PORT
    and #%11111100       ; Bits 0+1 löschen
    ora #%00000010       ; Bank 1 = 10 binär (invertiert!)
    sta CIA2_PORT

    ; $D018: Screen bei Offset 0 ($4000), Charset bei Offset 1 ($4800)
    ; Bits 7-4 = Screen-Offset = 0 → $0000 in Bank = $4000
    ; Bits 3-1 = Charset-Offset = 1 → $0800 in Bank = $4800
    lda #%00000010       ; Screen=0 ($4000), Charset=1 ($4800)
    sta VIC_MEMCTL

    ; Sprite-Pointer im Screen-RAM ($4FF8-$4FFF) setzen
    ; Sprite 0 Daten bei $4FC0: Pointer = ($4FC0-$4000)/$40 = $3F
    ldx #7
.sptr_loop:
    lda #$3F
    sta $4FF8,x          ; Alle 8 Sprites auf $4FC0
    dex
    bpl .sptr_loop
    rts
Beim Wechsel des VIC-Banks (CIA2 $DD00) muss auch $D018 entsprechend angepasst werden, da die Adressen relativ zur Bankbasis sind. Außerdem: Sprites werden immer aus dem aktuell aktiven VIC-Bank geladen — Sprite-Daten müssen also im gleichen Bank wie Screen-RAM liegen.

7. Timing, DMA-Interaktion und Geräteerkennung

7.1 DMA-Stalls und Interrupt-Handling

Während ein REU-DMA-Transfer läuft, hält der DMA-Controller den 6510-Bus-Request aktiv. Der Prozessor kann in dieser Zeit keine Interrupts bearbeiten. Für kurze Transfers (<1 KB) ist das unproblematisch — der Raster-IRQ wartet wenige Hundert Zyklen. Für längere Transfers (mehrere KB) sollte man:

  • Den Transfer im vertikalen Blanking (Rasterzeile > 250 PAL) starten
  • SEI vor dem Transfer ausführen, CLI danach
  • Oder den REU-End-of-Block-IRQ nutzen und darauf reagieren

7.2 EasyFlash und Raster-IRQ

Bankwechsel via $DE00 dauern genau einen Schreibzyklus. Das ist schnell genug, um im Raster-IRQ die Bank zu wechseln, Daten abzurufen und zurückzuschalten. Für zeitkritische Raster-Effekte gilt: Bankwechsel außerhalb der Badline-Region durchführen, da diese ohnehin 40 Zyklen stiehlt.

7.3 Universelle Erkennungsroutine für Erweiterungen

Ein robustes Programm erkennt beim Start, welche Hardware verfügbar ist, und aktiviert entsprechende Codepfade:

; hardware_detect — Erkennt verfügbare Erweiterungen
; Ergebnis in hw_flags:
;   Bit 0 = REU vorhanden
;   Bit 1 = EasyFlash vorhanden
;   Bit 2 = 1764/1750 (REU >= 256KB)

hw_flags: !byte 0

hardware_detect:
    lda #0 : sta hw_flags
    ; REU-Test: Schreibe Kennung in REU, lies zurück
    jsr detect_reu
    beq .check_ef
    lda hw_flags
    ora #%00000001       ; REU-Bit setzen
    sta hw_flags
    ; REU-Größe prüfen: $DF00 Bit 4 = 0 → 512KB
    lda $DF00
    and #$10
    bne .check_ef
    lda hw_flags
    ora #%00000100       ; 512KB-Bit setzen
    sta hw_flags
.check_ef:
    ; EasyFlash-Test: $DE00 schreiben und prüfen ob Freeze-Button reagiert
    lda $DE02            ; Mode-Register lesen
    cmp #$FF             ; $FF = kein EasyFlash (ungültiger Wert)
    beq .done
    lda hw_flags
    ora #%00000010       ; EasyFlash-Bit setzen
    sta hw_flags
.done:
    rts
Zeige dem Benutzer beim Programmstart eine Statusmeldung: "REU 512KB erkannt — Extrastufen verfügbar" oder "Kein REU — Standard-Modus". Das ist guter Stil und hilft beim Testen auf echter Hardware.

8. Walkthrough: REU-beschleunigte Bild-Slideshow

Vollständiges Beispielprojekt: Vier Bitmap-Screens wechseln sich automatisch ab — jeder Wechsel in unter 2 ms per DMA.

8.1 Konzept und REU-Layout

Vier Hires-Bitmap-Screens (je 8 KB Bitmap-Daten + 1 KB Color-RAM = 9 KB pro Screen) werden beim Start von Disk in den REU-Speicher geladen. Im Hauptprogramm läuft ein Frame-Counter. Nach jeweils 150 Frames (3 Sekunden PAL) wird der nächste Screen per DMA in das sichtbare C64-RAM transferiert.

REU-Speicherbelegung für 4 Screens: REU $00000-$01F3F Screen 0 Bitmap (8000 Bytes) REU $01F40-$023C7 Screen 0 Color (1000 Bytes) REU $04000-$05F3F Screen 1 Bitmap (8000 Bytes) REU $05F40-$063C7 Screen 1 Color (1000 Bytes) REU $08000-$09F3F Screen 2 Bitmap (8000 Bytes) REU $09F40-$0A3C7 Screen 2 Color (1000 Bytes) REU $0C000-$0DF3F Screen 3 Bitmap (8000 Bytes) REU $0DF40-$0E3C7 Screen 3 Color (1000 Bytes) Gesamtbedarf: ~36 KB (passt komfortabel in REU 1700)

8.2 Hauptschleife mit DMA-Screen-Wechsel

; slideshow_loop — Hauptschleife der REU-Slideshow
; Wechselt alle 150 Frames (3 Sek. PAL) zum nächsten Screen

frame_count:    !byte 0
current_screen: !byte 0

; REU-Basisadressen pro Screen (High-Byte und Mid-Byte)
reu_screen_hi: !byte $00,$04,$08,$0C  ; Bitmap High-Bytes
reu_color_hi:  !byte $01,$05,$09,$0D  ; Color-RAM High-Bytes (Offset $1F40)

slideshow_loop:
.wait_vbl:
    lda $D011
    bpl .wait_vbl        ; Warten bis Bit 7 = Rasterzeile > 255

    inc frame_count
    lda frame_count
    cmp #150
    bcc .no_switch

    ; Screen wechseln
    lda #0 : sta frame_count
    inc current_screen
    lda current_screen
    cmp #4
    bcc .do_fetch
    lda #0 : sta current_screen

.do_fetch:
    ldy current_screen

    ; Bitmap: REU → $2000 (Hires-Bitmap-Adresse in Bank 0)
    lda #$00 : sta $DF02  ; C64: $2000
    lda #$20 : sta $DF03
    lda #$00 : sta $DF04
    lda reu_screen_hi,y
    sta $DF05            ; REU-Adresse Mid
    lda #$00 : sta $DF06
    lda #$40 : sta $DF07  ; Länge $1F40 = 8000 Bytes
    lda #$1F : sta $DF08
    lda #$82 : sta $DF01  ; Fetch (REU→C64)

    ; Color-RAM: REU → $D800
    lda #$00 : sta $DF02
    lda #$D8 : sta $DF03
    lda #$40 : sta $DF04  ; Offset $1F40 innerhalb des Blocks
    lda reu_color_hi,y
    sta $DF05
    lda #$00 : sta $DF06
    lda #$E8 : sta $DF07  ; Länge $03E8 = 1000 Bytes
    lda #$03 : sta $DF08
    lda #$82 : sta $DF01

.no_switch:
    jmp slideshow_loop

8.3 Performance-Ergebnis

Ein vollständiger Screen-Wechsel (8 KB Bitmap + 1 KB Color-RAM) dauert mit REU-DMA:

  • Bitmap-Transfer: 8.000 + 2 = 8.002 Zyklen = ca. 8,1 ms PAL
  • Color-RAM-Transfer: 1.000 + 2 = 1.002 Zyklen = ca. 1,0 ms
  • Gesamt: ~9 ms gegenüber ~73 ms für eine Software-Kopierschleife

Der Transfer findet im VBlank statt und ist im ersten Frame vollständig abgeschlossen — kein Tearing, kein sichtbares Flackern.
