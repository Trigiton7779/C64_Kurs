Stufe 12 — Workflow

Demo-Entwicklung von A bis Z

★ Meister-Niveau ⏱ ca. 14–18 Stunden 📋 Voraussetzung: Stufen 09–11

★ DEMOSCENE — KUNST AUS CODE ★

Eine Demo ist kein Spiel, kein Werkzeug — sie ist eine Kunstform. Ihr einziger Zweck: zeigen, was möglich ist. Seit 1985 treibt die Demoscene die Grenzen der C64-Hardware weiter, und heute noch erscheinen Demos, die scheinbar Unmögliches zeigen. Diese Stufe erklärt, wie man selbst eine Demo von der Idee bis zum Release entwickelt.

Lernziele dieser Stufe

  • Ein Demo-Konzept entwickeln und Musik-first oder Effekt-first wählen
  • Timing-Tabellen erstellen und Effekte frame-genau mit der Musik synchronisieren
  • Part-Wechsel mit Fade- und Wipe-Übergängen implementieren
  • Die richtige Entwicklungsreihenfolge für eine Multi-Part-Demo kennen
  • PAL/NTSC-Kompatibilitätsprüfung und Anpassungsroutinen schreiben
  • Ein D64-Image mit AUTOSTART und korrekter Datei-Reihenfolge erstellen
  • Eine vollständige 3-Part-Demo von Konzept bis CSDb-Upload entwickeln

1. Demo-Konzeption — Thema, Stil und Struktur

Während ein Spiel durch seine Mechanik lebt, lebt eine Demo durch ihre visuelle und musikalische Wirkung. Das Konzept entscheidet darüber, ob die Demo kohärent wirkt oder wie eine zufällige Ansammlung von Effekten.

1.1 Musik-first vs. Effekt-first — zwei Philosophien

Es gibt zwei grundlegende Ansätze für die Demo-Entwicklung:

AnsatzAblaufVorteilRisiko
Musik-first Musik fertigstellen → Timing-Tabelle ableiten → Effekte passend entwickeln Effekte folgen der Musik natürlich; wirkt harmonisch und "geplant" Musik muss früh fertig sein; Änderungen an der Musik zwingen zur Überarbeitung aller Timings
Effekt-first Effekte entwickeln → Musik zum bestehenden Timing komponieren/beauftragen Effekte können frei experimentiert werden; Musik-Timing ist flexibler Musik muss auf fertige Effekte zugeschnitten werden; weniger Kohärenz wenn nicht sorgfältig geplant
Empfehlung für Soloprojekte
Wenn du Musik und Code selbst machst: Musik-first. Wenn du ein bestehendes SID-File verwendest (z.B. aus HVSC): ebenfalls Musik-first — analysiere das Stück, bevor du einen Effekt schreibst.

1.2 Part-Liste und Demo-Dramaturgie

Eine gute Demo hat eine Dramaturgie: sie beginnt mit einer starken Eröffnung, steigert sich zum Höhepunkt und endet mit einem befriedigenden Abschluss. Die Part-Liste ist das Drehbuch der Demo.

[INTRO] Kurz, prägnant. Logo + Musik-Einstieg. 15–30 Sekunden. | v [PART 1] Erster großer Effekt. Aufmerksamkeit etablieren. 30–60 Sek. | v [ÜBERGANG] Fade-to-Black. 1–2 Sekunden. | v [PART 2] Haupteffekt. Stärkster visueller Eindruck. 60–90 Sek. | v [ÜBERGANG] Wipe oder Screen-Scroll. | v [PART 3] Ruhigerer Effekt oder technisches Showpiece. 30–60 Sek. | v [CREDITS / OUTRO] Text + Musik-Outro. Greetings, Credits. 30–45 Sek. | v [ENDE] Zurück zu BASIC oder Endlosschleife.

1.3 Stilentscheidungen

Lege vor Entwicklungsbeginn fest, welchen Stil deine Demo hat. Diese Entscheidungen beeinflussen jede technische Wahl:

  • Farbpalette: Monochromatisch (eine Akzentfarbe) oder bunt (alle 16 C64-Farben)?
  • Grafik-Modus: Zeichenmodus (flexibel, schnell) oder Bitmap (mehr Pixel, weniger Farben)?
  • Musik-Stil: Elektronisch/Chippy oder melodisch? Tempo (BPM)?
  • Effekt-Ästhetik: Technisch-geometrisch oder organisch-fließend?
  • Schrift: Welcher Charset für Credits und Scroller?

1.4 Zeitplan und Gruppen-Rollen

Soloprojekt oder Gruppe macht einen großen Unterschied im Zeitplan:

RolleAufgabenWerkzeuge
CoderAlle Effekte, Loader, Part-Wechsel, TimingACME, VICE, eigene Tools
MusikerSID-Komposition, Player-OptimierungGoatTracker, SidTracker64
GrafikerLogos, Charset-Designs, FarbgebungCharPad, SpritePad, Pixelfolio

Als Soloentwickler nimmst du alle drei Rollen ein. Das ist machbar, aber platziere deine stärkste Fähigkeit im "Part 2" — dem Herzstück der Demo.

2. Musik-Synchronisation — Der kritische Pfad

Musik-Synchronisation ist das, was eine Demo von einer bloßen Effekt-Sammlung unterscheidet. Effekte, die exakt auf Beats, Melodie-Höhepunkte und Übergänge reagieren, lassen Musik und Grafik zu einem Erlebnis verschmelzen.

2.1 SID-Tickrate und Zeiteinheiten

Alles in einer C64-Demo dreht sich um den Frame als Zeiteinheit:

StandardFrames/SekundeTaktzyklen/FrameRasterzeilen/Frame
PAL (Europa)50 Hz19.656312
NTSC (USA)60 Hz17.030263

GoatTracker ruft seine Player-Routine einmal pro Frame auf. Das bedeutet: 50 Player-Aufrufe pro Sekunde (PAL). Ein Takt bei 125 BPM (2 Takte/Sekunde) dauert:

  • 50 Frames/Sekunde ÷ 2 Takte/Sekunde = 25 Frames pro Takt
  • Bei 4/4-Takt: 25 ÷ 4 = 6,25 Frames pro Beat (gerundet: 6 oder 7)

2.2 Song-Struktur analysieren und Timing-Tabelle erstellen

Bevor du eine einzige Zeile Effekt-Code schreibst, analysiere die Musik und erstelle eine Timing-Tabelle. Diese Tabelle ist das Herzstück der Demo-Synchronisation:

2.3 Sync-Tabelle in ACME implementieren

; ── Timing-Tabelle: Events als Frame-Nummer + Event-Code ─────
; Format: !word Frame, !byte Event-Code
; Abschluss: !word $FFFF (Sentinel)

EVENT_NONE       = 0
EVENT_BEAT       = 1     ; Kurzer Farb-Impuls
EVENT_PART2_INIT = 2     ; Zu Part 2 wechseln
EVENT_PART3_INIT = 3     ; Zu Part 3 wechseln
EVENT_FLASH      = 4     ; Bildschirm-Flash
EVENT_PLASMA_MAX = 5     ; Plasma-Geschwindigkeit Maximum
EVENT_SCROLLER   = 6     ; Scroller ein/aus
EVENT_END        = $FF   ; Demo endet

sync_table:
        !word 25  : !byte EVENT_BEAT
        !word 50  : !byte EVENT_SCROLLER
        !word 75  : !byte EVENT_BEAT
        !word 100 : !byte EVENT_FLASH
        !word 150 : !byte EVENT_PART2_INIT
        !word 200 : !byte EVENT_PLASMA_MAX
        !word 350 : !byte EVENT_PART3_INIT
        !word 500 : !byte EVENT_END
        !word $FFFF           ; Sentinel: Ende der Tabelle

; ── Event-System im IRQ ───────────────────────────────────────
sync_index:     !byte 0    ; Zeiger in sync_table (Byte-Index)
frame_counter:  !word 0    ; Globaler Frame-Zähler (16-Bit)

check_sync_events:
        ; Frame-Zähler erhöhen
        inc frame_counter
        bne .no_hi
        inc frame_counter+1
.no_hi:
        ; Aktuellen Tabelleneintrag lesen
        ldx sync_index
        lda sync_table,x      ; Low-Byte Frame-Nummer
        cmp frame_counter
        bne .no_event
        lda sync_table+1,x    ; High-Byte Frame-Nummer
        cmp frame_counter+1
        bne .no_event

        ; Event ausführen
        lda sync_table+2,x    ; Event-Code lesen
        jsr dispatch_event

        ; Zum nächsten Tabelleneintrag (3 Bytes pro Eintrag)
        clc
        txa
        adc #3
        sta sync_index
.no_event:
        rts

dispatch_event:
        ; Verzweigung anhand des Event-Codes
        cmp #EVENT_BEAT      : beq do_beat
        cmp #EVENT_FLASH     : beq do_flash
        cmp #EVENT_PART2_INIT: beq init_part2
        cmp #EVENT_PART3_INIT: beq init_part3
        cmp #EVENT_END       : beq demo_end
        rts

3. Part-Struktur und Übergänge

In einer Demo ist jeder "Part" eine eigenständige Sequenz mit eigenem Effekt, eigener Initialisierung und eigenem Speicher-Layout. Der Wechsel zwischen Parts muss sauber und ästhetisch ansprechend sein.

3.1 Part-Speicherlayout

Es gibt zwei Strategien für das Part-Speicherlayout:

  • Alles im RAM: Alle Parts gleichzeitig im Speicher. Nur möglich wenn die Demo klein genug ist. Vorteil: kein Laden nötig; Nachteil: 64 KB sind schnell voll.
  • Streaming: Parts werden von Diskette geladen wenn sie gebraucht werden. Benötigt einen Loader. Für große, mehrminütige Demos standard.
; Speicherlayout für eine kleine 3-Part-Demo (alles im RAM)

SHARED_CODE     = $0900  ; IRQ, Musik, Sync-Tabelle, Part-Wechsel (~2 KB)
MUSIC_DATA      = $1000  ; GoatTracker-Daten (~3 KB)
PART1_CODE      = $2000  ; Sinusscroller + Rasterbar-Effekt (~2 KB)
PART1_DATA      = $2800  ; Charset, Scroller-Text (~3 KB)
PART2_CODE      = $3800  ; Plasma-Effekt (~1.5 KB)
PART2_TABLES    = $4000  ; Sinustabellen für Plasma (2 × 256 Bytes)
PART3_CODE      = $4200  ; DYSP Credits-Scroller (~1.5 KB)
PART3_DATA      = $4800  ; Credits-Text, Sprite-Daten (~2 KB)

; Tabellen der Part-Funktionen (Dispatch-Tabelle)
part_init_table:
        !word init_part1
        !word init_part2
        !word init_part3

part_update_table:
        !word update_part1
        !word update_part2
        !word update_part3

current_part:   !byte 0    ; 0 = Part 1, 1 = Part 2, 2 = Part 3

call_part_update:
        lda current_part
        asl a                  ; × 2 für Word-Offset
        tax
        lda part_update_table,x
        sta ZP_PTR_LO
        lda part_update_table+1,x
        sta ZP_PTR_HI
        jmp (ZP_PTR_LO)        ; Indirekter Aufruf des aktuellen Parts

3.3 Fade-to-Black implementieren

; Fade-to-Black: 16 Frames, schrittweise alle Sprite-Farben auf 0
; und Hintergrundfarbe auf 0

fade_counter:   !byte 0

start_fade_out:
        lda #16              ; 16 Schritte
        sta fade_counter
        rts

update_fade:
        lda fade_counter
        beq .fade_done       ; Fertig: nichts mehr tun

        dec fade_counter

        ; Hintergrundfarbe abdunkeln (geht nur in Schritten, da C64
        ; keine RGB-Kontrolle hat — wir schalten einfach auf Schwarz)
        lda fade_counter
        cmp #8               ; Ab Mitte: auf Schwarz setzen
        bcs .bg_still_on
        lda #0               ; Schwarz
        sta $D021            ; Hintergrundfarbe
        sta $D020            ; Rahmenfarbe
.bg_still_on:
        ; Sprites deaktivieren bei 0
        lda fade_counter
        bne .fade_done
        lda #0
        sta $D015            ; Alle Sprites aus

        ; Nächsten Part initialisieren
        jsr init_next_part
.fade_done:
        rts

3.4 Shared-Ressourcen zwischen Parts

Einige Ressourcen liegen dauerhaft im RAM und werden von allen Parts geteilt — sie werden nur einmal beim Demo-Start initialisiert:

  • Musik-Player + Daten: GoatTracker-Routine läuft von Anfang bis Ende
  • IRQ-Handler: Bleibt immer derselbe, ruft den aktuellen Part auf
  • Sync-Tabelle + Frame-Counter: Einmalig initialisiert
  • Zero-Page-Variablen: Gemeinsam genutzte Pointer und temporäre Werte

Part-spezifische Ressourcen (Charset, Sprite-Daten, Effekt-Variablen) werden beim init_partX-Aufruf gesetzt und beim Part-Wechsel durch den Nachfolger überschrieben.

4. Entwicklungsreihenfolge — Loader zuerst, Credits zuletzt

Die Entwicklungsreihenfolge einer Demo unterscheidet sich von der eines Spiels: Du entwickelst nicht Feature für Feature, sondern Part für Part — aber in einer bestimmten Reihenfolge, die sicherstellt, dass du immer etwas Lauffähiges hast.

  • 1
    Loader und Bootstrap: Disk-Layout planen, Demo startet und zeigt schwarzen Bildschirm. Musik läuft. Frame-Counter zählt. Sync-Tabelle ist leer — wird später gefüllt.
  • 2
    Musik-Player testen: GoatTracker-Export einbinden, im IRQ aufrufen. Sicherstellen, dass die Musik stabil und ohne Knackser läuft. Zyklen-Budget messen: wie viel bleibt für die Effekte?
  • 3
    Stärksten Effekt zuerst: Part 2 (Haupteffekt) entwickeln, bevor Part 1 (Intro). Wenn der Haupteffekt nicht überzeugend ist, weißt du es früh genug um die Richtung zu ändern.
  • 4
    Intro + weiterer Part: Part 1 entwickeln. Die Part-Wechsel-Mechanik einbauen. Übergänge implementieren.
  • 5
    Credits/Outro: Letzter Part. Oft ein ruhigerer Scroller mit Gruppen-Namen und Greetings. Erst jetzt die finalen Texte eintragen.
  • 6
    Timing-Feinschliff: Sync-Tabelle mit echten Frame-Nummern füllen (vorher Platzhalter). Jedes Timing in VICE prüfen und anpassen. PAL/NTSC-Test.
Wichtig: Timing erst am Ende fixieren
Trage in die Sync-Tabelle erst dann echte Frame-Nummern ein, wenn alle Effekte fertig sind. Vorher kannst du die Demo auch ohne Sync laufen lassen — einfach Platzhalter-Werte weit in der Zukunft (z.B. $7FFF) eintragen.

5. Loader und Disk-Struktur

Für eine Demo, die in den RAM passt, reicht der Standard-KERNAL-Loader. Für größere, mehrteilige Demos ist ein schneller Custom-Loader sinnvoll — aber dessen Implementierung übersteigt den Scope dieser Stufe. Hier das Konzept und die KERNAL-Variante.

5.1 Standard-KERNAL-Loader für die Demo

; Demo-Loader: Alle Teile von Diskette laden
; Annahme: Demo startet von Disk, KERNAL aktiv

; Konstanten für KERNAL-Aufrufe
SETLFS          = $FFBA
SETNAM          = $FFBD
LOAD            = $FFD5

load_file:
        ; Parameter: .X = Lade-Adresse Low, .Y = High, .A = Datei-Nummer
        ; Dateiname in filename_ptr/filename_len
        pha                  ; Dateinummer merken

        ; SETLFS: Logische Datei = A, Gerät 8, Sekundäradresse 0
        lda filename_len
        ldx #<ldy #>filename
        jsr SETNAM           ; Dateiname setzen

        pla                  ; Dateinummer zurück
        ldx #8               ; Gerät 8 (1541)
        ldy #0               ; Sekundäradresse 0
        jsr SETLFS

        ; LOAD: A=0 (Laden), XY = Zieladresse (wenn Sek.-Adresse=0: ignoriert)
        ; Mit Sek.-Adresse=1: Ladeadresse aus PRG-Header verwenden
        lda #0               ; Modus: Laden (nicht Verifizieren)
        jsr LOAD

        ; Fehlerprüfung: Carry = 1 → Fehler
        bcs load_error
        rts

load_error:
        ; Fehlerbehandlung: Fehlermeldung anzeigen, Demo abbrechen
        brk                  ; Placeholder — im echten Code: Fehlermeldung

filename:       !text "BITWAVE.PART2"
filename_len:   !byte 13

5.2 D64-Datei-Reihenfolge optimieren

Die Ladereihenfolge beeinflusst, wie lange der Nutzer auf den nächsten Part warten muss. Auf einer 1541 dauert das Laden etwa 1 Sekunde pro 1 KB (KERNAL-Standard). Optimierungsregeln:

  • Musik zuerst: Musik-Daten sind klein (~3 KB) und müssen sofort beim Start verfügbar sein
  • Parts in Reihenfolge: Part 2 laden, während Part 1 läuft
  • Große Dateien zuletzt: Sinustabellen, Charset-Sets
# D64-Image für BITWAVE-Demo erstellen
# Reihenfolge: Loader, Musik, Part1, Part2, Part3

c1541 -format "bitwave,99" d64 bitwave.d64

# Loader zuerst (startet automatisch)
c1541 -attach bitwave.d64 -write loader.prg "loader"

# Musik (wird sofort nach Start geladen)
c1541 -attach bitwave.d64 -write musik.prg "musik"

# Parts in Laufzeit-Reihenfolge
c1541 -attach bitwave.d64 -write part1.prg "part1"
c1541 -attach bitwave.d64 -write part2.prg "part2"
c1541 -attach bitwave.d64 -write part3.prg "part3"

# Ergebnis prüfen
c1541 -attach bitwave.d64 -dir

6. PAL/NTSC-Kompatibilität

Eine Demo, die nur auf PAL läuft, schließt alle nordamerikanischen C64-Nutzer aus. NTSC-Kompatibilität ist ein Zeichen von Professionalität — und technisch gar nicht so schwierig, wenn man von Anfang an daran denkt.

6.1 Die wichtigsten Unterschiede

EigenschaftPAL (Europa)NTSC (USA/Kanada)Konsequenz
Rasterzeilen/Frame312263Raster-IRQ-Zeilennummern müssen angepasst werden
Frames/Sekunde50 Hz60 HzBewegungsgeschwindigkeiten 20% zu schnell auf NTSC
Taktfrequenz985.248 Hz1.022.727 HzSID-Frequenzen klingen auf NTSC ~4% höher
Sichtbarer Bereich284 sichtbare Zeilen235 sichtbare ZeilenNTSC hat oben/unten weniger sichtbaren Bereich

6.2 PAL/NTSC-Detektion im Code

; PAL/NTSC-Erkennung durch Zählen der Rasterzeilen
; PAL hat 312 Zeilen, NTSC hat 263 Zeilen pro Frame
; Methode: Warte auf Rasterzeile 0, zähle bis Rasterzeile 0 wieder erscheint

is_ntsc:        !byte 0    ; 0 = PAL, 1 = NTSC

detect_standard:
        ; Warte auf Rasterzeile 0 (Anfang eines Frames)
.wait_non_zero:
        lda $D012
        bne .wait_non_zero   ; Warte bis Zeile ≠ 0
.wait_zero:
        lda $D012
        bne .wait_zero       ; Warte bis Zeile = 0 (neuer Frame)

        ; Jetzt zählen wir die Rasterzeilen bis zum nächsten Frame-Start
        ; Wenn in 300 Zeilen noch kein neuer Rahmen-Beginn: PAL (312 Zeilen)
        ldx #$18             ; = 24 × 256 = 6144 Schleifendurchläufe
        ldy #0
.count:
        lda $D012
        cmp #263-20         ; NTSC: Rasterzeile 243 vor Frame-Ende
        bcc .not_ntsc_line   ; Noch nicht dort
        lda #1
        sta is_ntsc          ; NTSC erkannt!
        rts
.not_ntsc_line:
        iny
        bne .count
        dex
        bne .count
        ; Schleife abgelaufen ohne NTSC-Erkennung → PAL
        lda #0
        sta is_ntsc
        rts

; ── Raster-IRQ-Anpassung je nach Standard ───────────────────
setup_raster_irq:
        lda is_ntsc
        beq .pal_irq
        ; NTSC: Rasterzeile 240 (kurz vor Frame-Ende bei 263 Zeilen)
        lda #240
        sta $D012
        rts
.pal_irq:
        ; PAL: Rasterzeile 300 (kurz vor Frame-Ende bei 312 Zeilen)
        lda #300-256         ; = 44 (Bit 8 wird via $D011 gesetzt)
        sta $D012
        rts

6.3 Bewegungsgeschwindigkeiten anpassen

Auf NTSC laufen Effekte 20% schneller (60 statt 50 Hz). Du kannst das mit einer einfachen Strategie kompensieren: Lade einen geschwindigkeits-Faktor abhängig vom erkannten Standard:

; Geschwindigkeits-Tabelle: PAL vs. NTSC
; Auf NTSC: Effekte 5/6 so schnell bewegen wie auf PAL
; Einfachste Methode: jeden 6. Frame auf NTSC überspringen

ntsc_skip_counter: !byte 0

should_update:
        lda is_ntsc
        beq .always_update   ; PAL: immer updaten

        ; NTSC: jeden 6. Frame überspringen
        inc ntsc_skip_counter
        lda ntsc_skip_counter
        cmp #6
        bne .always_update
        lda #0
        sta ntsc_skip_counter
        clc                  ; Carry = 0: diesen Frame überspringen
        rts
.always_update:
        sec                  ; Carry = 1: normaler Frame
        rts

7.2 Raster-Jitter mit VICE diagnostizieren

Raster-Jitter — das leichte Wackeln von Rasterbars oder Raster-Effekten — ist der häufigste Demo-Bug. Er entsteht, wenn der IRQ nicht immer auf genau der gleichen Taktphase eintrifft. Die Diagnose ist einfach: Setze die Rahmenfarbe zu Beginn des IRQ auf Weiß — siehst du einen weißen Strich statt einer Linie, hast du Jitter.

; Jitter-Test: Wenn der weiße Strich unscharf oder gewellt ist → Jitter
raster_irq_test:
        lda #1              ; Weiß
        sta $D020

        ; ... IRQ-Arbeit ...

        lda #0              ; Schwarz
        sta $D020
        jmp $EA81

; Lösung: Double-IRQ-Technik aus Stufe 07 anwenden
; Erster IRQ NOP-Schleife bis zur stabilen Phase,
; Zweiter IRQ macht die eigentliche Arbeit

8. Release — NFO, Packaging und CSDb

8.1 Die NFO-Datei — Pflichtfelder und Format

9. Vollständiger Walkthrough — 3-Part-Demo "BITWAVE"

Von der Konzeptidee bis zum CSDb-tauglichen D64-Image — jeder Schritt dokumentiert.

9.1 Konzept und Part-Liste

BITWAVE — Demo-Konzept (eine halbe Seite)

Thema: Elektronische Wellen, Frequenzen, digitaler Klang
Ansatz: Musik-first (vorhandenes SID-File aus HVSC)
Stil: Monochrom (Blau auf Schwarz) mit akzentuiertem Grün für Beats
Laufzeit: ~3 Minuten
Parts: Intro (Rasterbars + Sinus-Scroller) → Plasma (Haupteffekt) → DYSP-Credits
Ziel: CSDb-Release, PAL-optimiert, NTSC-kompatibel

9.2 Timing-Tabelle aus der Musik ableiten

Zuerst das SID-File in VICE laden und die Musik analysieren. Notiere Frame-Nummern für wichtige Ereignisse:

  • Frame 0: Intro startet, Rasterbars langsam
  • Frame 50: Beat 1 → Rasterbars beschleunigen
  • Frame 100: Musik-Refrain → Scroller erscheint
  • Frame 175: Musik-Break → Fade-Out, Part 2 startet
  • Frame 225: Beat → Plasma-Geschwindigkeit Maximum
  • Frame 380: Outro-Beginn → Fade-Out, Part 3 (Credits)
  • Frame 500: Ende → Endlosschleife oder BASIC-Rückkehr

9.3 Part 1: Intro — Rasterbars + Sinusscroller

Part 1 kombiniert einfache Rasterbars (aus Stufe 09) mit einem Hardware-Scroller. Die Rasterbar-Farben reagieren auf Beat-Events aus der Sync-Tabelle.

; Part 1: Sinus-Scroller mit Rasterbars
; Scroll-Register $D016 Bits 0-2 für horizontales Soft-Scroll

scroll_x:       !byte 7    ; Aktuelle Scroll-Position
text_offset:    !byte 0    ; Offset in scroll_text
rasterbar_color: !byte 6   ; Aktuell Blau

update_part1:
        ; Scroll-Register dekrementieren
        dec scroll_x
        bpl .no_shift        ; Noch positiv: nur Pixel-Scroll

        ; Auf 7 zurücksetzen und Zeichenreihe verschieben
        lda #7
        sta scroll_x
        jsr shift_screen_left ; Alle 40 Zeichen in Zeile 24 um 1 links
        jsr append_next_char  ; Nächstes Zeichen des scroll_text rechts einfügen
.no_shift:
        ; Scroll-Register setzen
        lda scroll_x
        and #$07
        ora #$C8             ; Bits 7-4: Standard + 38 Spalten (schmaler Rand)
        sta $D016
        rts

scroll_text:
        !pet "     bitwave by yourgroup ... music: yourtune ... greetings to the world     "
        !byte 0             ; Terminierung

; Beat-Event-Handler: Rasterbar-Farbe kurz auf Grün (Blitz-Effekt)
do_beat:
        lda #5              ; Grün
        sta rasterbar_color
        lda #8              ; 8 Frames Grün bleiben
        sta beat_decay
        rts

9.4 Part 2: Plasma mit Musik-Synchronisation

Der Plasma-Effekt aus Stufe 09 wird hier mit der Musik verknüpft: die Plasma-Geschwindigkeit steigt mit dem Beat und sinkt danach wieder ab.

; Plasma-Geschwindigkeit wird durch Sync-Events gesteuert
plasma_speed:   !byte 1    ; Startet langsam
plasma_decay:   !byte 0    ; Countdown für Rückkehr zur Normalgeschwindigkeit

update_part2:
        ; Plasma-Geschwindigkeit abklingen lassen
        lda plasma_decay
        beq .no_decay
        dec plasma_decay
        bne .no_decay
        lda #1               ; Zurück auf Normalgeschwindigkeit
        sta plasma_speed
.no_decay:
        ; Plasma-Phase um plasma_speed erhöhen
        lda plasma_speed
        clc
        adc plasma_phase1
        sta plasma_phase1
        ; ... Phase 2 analog ...
        jsr render_plasma
        rts

; Beat-Event im Plasma: Geschwindigkeit kurz erhöhen
do_event_plasma_max:
        lda #4               ; 4× normale Geschwindigkeit
        sta plasma_speed
        lda #30              ; 30 Frames schnell, dann abklingen
        sta plasma_decay
        rts

9.5 Part 3: Credits-Scroller mit DYSP-Welle

Der Credits-Part zeigt die Gruppen-Infos in einem Sprite-Scroller. DYSP (Dynamic Sprite Positioning) lässt die Sprites in einer Sinus-Welle schwingen. Die Credits werden als Zeichenkette gespeichert und Buchstabe für Buchstabe zu Sprites konvertiert.

9.6 Integration und Timing-Feinschliff

Nach Fertigstellung aller Parts wird die Sync-Tabelle mit echten Frame-Nummern befüllt. Methode: Demo starten, VICE pausieren (Alt+P), Frame-Counter auslesen und mit dem Musik-Eindruck vergleichen.

; Sync-Tabelle finalisieren (reale Werte nach Messung)
sync_table:
        !word 50  : !byte EVENT_BEAT        ; Gemessen: Frame 50
        !word 100 : !byte EVENT_SCROLLER     ; Refrain: Frame 100
        !word 175 : !byte EVENT_PART2_INIT   ; Break: Frame 175
        !word 225 : !byte EVENT_PLASMA_MAX   ; Höhepunkt: Frame 225
        !word 380 : !byte EVENT_PART3_INIT   ; Outro: Frame 380
        !word 500 : !byte EVENT_END          ; Ende: Frame 500
        !word $FFFF                           ; Sentinel

9.7 Build-Skript und Release

#!/bin/bash
# build_bitwave.sh — Vollständiger Build der BITWAVE-Demo
set -e

echo "=== BITWAVE BUILD ==="

# Alle Teile kompilieren
acme --cpu 6510 -f cbm -o loader.prg src/loader.asm
acme --cpu 6510 -f cbm -o part1.prg  src/part1.asm
acme --cpu 6510 -f cbm -o part2.prg  src/part2.asm
acme --cpu 6510 -f cbm -o part3.prg  src/part3.asm
echo "Kompilierung abgeschlossen."

# D64-Image erstellen
rm -f bitwave.d64
c1541 -format "bitwave,99" d64 bitwave.d64 \
  -write loader.prg "loader" \
  -write musik.prg  "musik"  \
  -write part1.prg  "part1"  \
  -write part2.prg  "part2"  \
  -write part3.prg  "part3"
echo "D64-Image: bitwave.d64 ($(du -k bitwave.d64 | cut -f1) KB)"

# VICE starten
x64sc bitwave.d64 &
echo "VICE gestartet."

Übungsaufgaben

Übung 12.1 — Timing-Tabelle analysieren

Wähle ein SID-File aus der HVSC (High Voltage SID Collection). Lade es in VICE und analysiere die Musik: Notiere 10 Frame-Nummern für wichtige Ereignisse (Beats, Melodie-Einsätze, Breaks). Erstelle daraus eine vollständige ACME-Sync-Tabelle.

Übung 12.2 — Fade-Übergang

Implementiere einen Fade-to-Black-Übergang, der 24 Frames dauert. Verwende einen stufenlosen Ansatz: Fülle das Color-RAM Zeile für Zeile mit Schwarz (eine Zeile pro Frame), während der Hintergrund bereits nach 12 Frames schwarz wird. Teste den Übergang auf PAL und NTSC.

Übung 12.3 — PAL/NTSC-Kompatibilität

Implementiere die PAL/NTSC-Erkennungsroutine aus diesem Kapitel. Zeige auf dem Bildschirm an, welcher Standard erkannt wurde. Passe einen bestehenden Raster-IRQ-Effekt so an, dass er auf beiden Standards korrekt funktioniert.

Übung 12.4 — Vollständige BITWAVE-Demo

Entwickle die BITWAVE-Demo aus diesem Walkthrough vollständig: Intro mit Rasterbar + Scroller, Plasma-Effekt mit Musik-Sync, Credits-Part. Erstelle das finale D64-Image, schreibe die NFO und reiche die Demo auf CSDb ein. Das ist das ultimative C64-Mastership-Projekt.
