Abschnitt 1: Adressierungsarten — Das Fundament der Maschinenprogrammierung

Wer zum ersten Mal mit dem 6502- oder 6510-Prozessor in Berührung kommt, ist oft überrascht von der schieren Anzahl der Adressierungsarten. Während moderne RISC-Architekturen — etwa ARM oder MIPS — mit wenigen, uniformen Adressierungsmodi auskommen, bietet der 6502-Prozessor insgesamt dreizehn verschiedene Modi an. Diese Vielfalt ist kein historischer Zufall, sondern das Ergebnis einer bewussten Designentscheidung von Chuck Peddle und seinem Team bei MOS Technology im Jahr 1975. Der 6502 sollte ein kostengünstiger, aber mächtiger Prozessor für den aufkommenden Heimcomputermarkt sein — und das bedeutete, mit nur 8-Bit-Registern und einem 64-KB-Adressraum maximale Flexibilität zu bieten.

Die zentrale Herausforderung der damaligen Zeit war der begrenzte Speicher. Programme mussten kompakt sein, Daten mussten effizient adressiert werden, und häufig genutzte Variablen sollten mit minimaler Rechenzeit erreichbar sein. Die Lösung von MOS Technology war ein hierarchisches Adressierungssystem: Für besonders häufig zugegriffene Daten gab es die "Zero Page" (Seite Null), die mit einem einzigen Byte adressiert werden konnte und damit sowohl Code-Größe als auch Ausführungszeit minimierte. Für weniger häufige Zugriffe standen vollständige 16-Bit-Adressen zur Verfügung. Für strukturierte Daten boten indizierte Modi komfortablen Zugriff auf Arrays und Tabellen. Und für die Implementierung von Zeigern und dynamischen Datenstrukturen existierten die indirekten Modi.

Der 6510-Prozessor im Commodore 64 ist eine direkte Weiterentwicklung des 6502 mit einer entscheidenden Erweiterung: einem integrierten 6-Bit-I/O-Port, der über die Adressen $0000 und $0001 erreichbar ist. Über diesen Port wird unter anderem das Memory-Banking-System des C64 gesteuert, das es erlaubt, ROM- und RAM-Bereiche in denselben Adressraum ein- und auszublenden. Alle dreizehn Adressierungsarten des 6502 sind im 6510 unverändert erhalten — wer sie beherrscht, beherrscht das fundamentale Werkzeug der Assembler-Programmierung auf dem C64.

1.2 Implied — Implizite Adressierung

Die implizite Adressierung ist die einfachste aller Modi: Der Befehl arbeitet ausschließlich auf internen Registern der CPU — kein Operand im Speicher, kein zusätzliches Byte im Maschinencode. Das Opcode-Byte allein bestimmt vollständig, was zu tun ist. Diese Befehle sind immer genau 1 Byte lang und benötigen in der Regel 2 Taktzyklen. 

; Implied-Befehle: Typische Anwendungen
NOP             ; No Operation — 2 Zyklen Zeitverschwendung (oder Timing-Pad)
CLC             ; Clear Carry Flag — vor ADC zwingend!

CLI             ; Clear Interrupt Disable — IRQs erlauben
SEI             ; Set Interrupt Disable — IRQs sperren
CLD             ; Clear Decimal Mode (auf C64 nach Reset nötig)
SED             ; Set Decimal Mode (BCD-Arithmetik)
CLV             ; Clear Overflow Flag
INX             ; Increment X — X++
DEX             ; Decrement X — X--

DEY             ; Decrement Y — Y--

TAY             ; Transfer A → Y

TSX             ; Transfer SP → X

PHP             ; Push Processor-Flags auf Stack
PLP             ; Pull Processor-Flags vom Stack
RTS             ; Return from Subroutine — liest Adresse vom Stack
RTI             ; Return from Interrupt — liest Flags + Adresse vom Stack

1.3 Accumulator — Akkumulator-Adressierung

Eine Sonderform der impliziten Adressierung: Der Befehl wirkt direkt auf den Akkumulator A. Im Assembler-Quelltext kann das Ziel explizit als A angegeben werden, viele Assembler erlauben aber auch das Weglassen des Operanden (weil der Kontext eindeutig ist). Nur Schiebe- und Rotationsbefehle unterstützen diesen Modus. 

ASL A           ; Arithmetic Shift Left: A = A << 1, Bit 7 → Carry, 0 → Bit 0
LSR A           ; Logical Shift Right: A = A >> 1, Carry ← Bit 0, 0 → Bit 7
ROL A           ; Rotate Left: A = A << 1 (durch Carry), Bit 7 → Carry
ROR A           ; Rotate Right: A = A >> 1 (durch Carry), Carry → Bit 7

; Anwendungsbeispiel: Multiplikation × 2 und Division / 2
LDA #$18         ; A = 24
ASL A           ; A = 48 (× 2 durch Links-Shift)
ASL A           ; A = 96 (× 4 durch zweiten Links-Shift)
LSR A           ; A = 48 (÷ 2 durch Rechts-Shift)

1.4 Immediate — Unmittelbare Adressierung

Bei der Immediate-Adressierung folgt der Operand — eine Konstante — direkt auf das Opcode-Byte im Maschinencode. Das Präfix-Zeichen # im Assembler-Quelltext signalisiert dem Assembler, dass ein unmittelbarer Wert folgt. Ohne # würde derselbe Wert als Speicheradresse interpretiert — ein häufiger und schwer zu findender Fehler! 

; Verschiedene Literal-Formate für denselben Wert (65 dezimal = $41 = %01000001):
LDA #65          ; Dezimal: Wert 65
LDA #$41         ; Hexadezimal: $41 = 65
LDA #%01000001   ; Binär: %01000001 = 65
LDA #'A'         ; ASCII-Zeichen: 'A' = 65 (Assembler-abhängig)

; DER KRITISCHE UNTERSCHIED:
LDA #$42         ; Immediate: Lädt den WERT 66 in A
LDA $42          ; Zero Page: Lädt den INHALT von Adresse $0042 in A!
LDA $0042        ; Absolute: Lädt den INHALT von Adresse $0042 in A (3 Bytes!)

; Typische Verwendung: Vergleiche und Initialisierungen
CMP #$0D         ; Vergleiche A mit Return-Zeichen (13)
LDX #$00         ; X auf 0 setzen (Zähler initialisieren)
LDA #%00110111   ; Banking-Register: Bitmuster direkt als Binär
ORA #%10000000   ; Bit 7 setzen ohne andere Bits zu verändern
AND #%01111111   ; Bit 7 löschen ohne andere Bits zu verändern

1.5 Zero Page — Die Nullseite als Schnellzugriffs-Ressource

Die Zero-Page-Adressierung ist eine der fundamentalsten Optimierungen des 6502/6510-Designs. Der Adressraum $0000–$00FF wird als "Seite Null" (Zero Page) bezeichnet, und für Adressen innerhalb dieses Bereichs genügt ein einziges Byte als Operand: Die CPU setzt das High-Byte der 16-Bit-Adresse intern automatisch auf $00. Daraus ergeben sich zwei entscheidende Vorteile gegenüber der absoluten Adressierung: Der Maschinencode-Befehl ist 1 Byte kürzer (2 statt 3 Bytes), und die Ausführung ist 1 Taktzyklus schneller (3 statt 4 Zyklen). Bei einem Programm, das tausende Male denselben Zero-Page-Zugriff ausführt, summiert sich das zu messbarer Performance. 

; Zero Page Adressierung: Syntax und Maschinencode-Größe
LDA $42          ; 2 Bytes: [A5] [42] — Lädt Inhalt von $0042, 3 Zyklen
STA $42          ; 2 Bytes: [85] [42] — Speichert A in $0042, 3 Zyklen
INC $42          ; 2 Bytes: Erhöhe Inhalt von $0042 um 1, 5 Zyklen

; Zum Vergleich: Absolute Adressierung derselben Stelle
LDA $0042        ; 3 Bytes: [AD] [42] [00] — Lädt Inhalt von $0042, 4 Zyklen

; Praktisches Beispiel: Schleifenzähler in Zero Page
LDX #$00
MainLoop:
    INC $FB         ; ZP-Variable $FB inkrementieren (5 Zyklen)
    LDA $FB         ; Wert lesen (3 Zyklen)
    CMP #100        ; Mit 100 vergleichen
    BNE MainLoop    ; Schleife bis 100 erreicht

1.6 Zero Page, X — Nullseiten-Indizierung mit X

Diese Adressierungsart kombiniert Zero-Page-Effizienz mit der Flexibilität eines Index-Registers. Die CPU addiert den 1-Byte-Operanden und den X-Wert, wobei das Ergebnis auf 8 Bit beschränkt bleibt — es gibt einen Wraparound innerhalb der Zero Page! Ein Zeiger von $F0 mit X = $20 ergibt nicht $0110, sondern $10 ($F0 + $20 = $110, modulo 256 = $10). Dieses Verhalten ist kein Bug, sondern intendiertes Design und muss beim Einsatz von Zero-Page,X berücksichtigt werden. 

; Zero Page,X: Effektive Adresse = ($Operand + X) & $FF (nur in ZP!)
LDA $20,X        ; Adresse: ($20 + X) mod 256  — immer in Zero Page
STA $20,X        ; Schreibt A an ($20 + X) mod 256

; Anwendung: 8 Spieler-Scores in Zero Page $30-$37
; X = Spieler-Nummer (0-7)
LDX #3           ; Spieler 3
LDA $30,X        ; Lädt Score[3] aus $0033
CLC
ADC #10          ; +10 Punkte
STA $30,X        ; Score[3] speichern

; WRAPAROUND DEMONSTRATION:
LDX #$F0
LDA $20,X        ; Effektive Adresse: ($20 + $F0) = $110 → $10 (ZP-Wraparound!)

1.7 Zero Page, Y — Nullseiten-Indizierung mit Y

Analog zur Zero-Page,X-Adressierung, jedoch mit dem Y-Register als Offset. Ein wichtiger Unterschied: Nur die Befehle LDX und STX unterstützen Zero-Page,Y — kein anderer Befehl! Der Wraparound-Mechanismus ist identisch mit ZP,X. Diese Einschränkung spiegelt die Hardware-Architektur des 6502 wider: Das X-Register ist primär der Index für horizontale (X-Richtung) Zugriffe, das Y-Register für vertikale. 

; Zero Page,Y: NUR für LDX und STX verfügbar!
LDX $20,Y        ; Korrekt: Lädt X aus ZP-Adresse ($20 + Y) mod 256
STX $20,Y        ; Korrekt: Speichert X an ZP-Adresse ($20 + Y) mod 256

; Häufiger Fehler-Versuch (COMPILIERT NICHT KORREKT):
; LDA $20,Y → Dies wird als Absolute,Y $0020,Y übersetzt! Nicht ZP,Y!

1.8 Absolute — Vollständige 16-Bit-Adressierung

Die absolute Adressierung erlaubt den Zugriff auf jeden beliebigen Byte im 64-KB-Adressraum des C64. Der Maschinencode-Befehl enthält die vollständige 16-Bit-Adresse als zwei Bytes nach dem Opcode, gespeichert im Little-Endian-Format: das niedrigwertige Byte (Low-Byte) zuerst, dann das höherwertige (High-Byte). So wird die Adresse $D020 im Maschinencode als [20] [D0] gespeichert. Dieser Modus ist unverzichtbar für den Zugriff auf Hardware-Register wie VIC-II, SID, CIA, das Screen-RAM und das Color-RAM. 

; Absolute Adressierung: 3 Bytes, 4 Zyklen
; Maschinencode: [Opcode] [Low-Byte] [High-Byte]

LDA $D020        ; Maschinencode: [AD] [20] [D0] — VIC-II Rahmenfarbe lesen
STA $D020        ; Maschinencode: [8D] [20] [D0] — Rahmenfarbe setzen
STA $0400        ; Erstes Zeichen des Screen-RAM beschreiben
LDA $FFE4        ; Aus KERNAL-ROM lesen (Adresse im ROM-Bereich)

; Symbolische Namen statt rohe Adressen (bessere Lesbarkeit):
VICBORDER = $D020
VICBGCOL  = $D021
SCREENRAM = $0400

LDA #2           ; Farbe 2 = Rot
STA VICBORDER   ; Rahmen rot
STA VICBGCOL    ; Hintergrund ebenfalls rot

1.9 Absolute, X — Absolute Adressierung mit X-Offset und Page-Crossing

Der Modus Absolute,X addiert den Wert des X-Registers zur 16-Bit-Basisadresse im Befehl. Das Ergebnis ist eine dynamische effektive Adresse, die sich mit jedem X-Wert ändert — ideal für den Zugriff auf Arrays und Lookup-Tabellen. Normalerweise benötigt dieser Modus 4 Zyklen. Es gibt jedoch einen besonderen Fall: die Page-Crossing-Penalty.

Eine "Speicherseite" (Page) ist ein 256-Byte-Block mit einer gemeinsamen High-Byte-Adresse: $0100–$01FF ist Page 1, $0200–$02FF ist Page 2, und so weiter. Wenn die Addition von Basisadresse und X-Offset eine Seitengrenze überschreitet — das High-Byte der effektiven Adresse also größer ist als das High-Byte der Basisadresse — benötigt die CPU einen zusätzlichen Taktzyklus (insgesamt 5 statt 4). Dieser Strafzyklus entsteht, weil die CPU intern zuerst nur das Low-Byte addiert und dann beim Übertrag eine Korrektur des High-Bytes vornehmen muss. 

; Absolute,X: Effektive Adresse = Basisadresse + X

LDA $C000,X      ; Effektive Adresse = $C000 + X
                 ; X=$00 → $C000 (4 Zyklen, keine Page-Crossing)
                 ; X=$FF → $C0FF (4 Zyklen, noch auf Seite $C0)
LDA $C0FF,X      ; X=$01 → $C100 (5 Zyklen! Seitengrenze $C0/$C1 überschritten)

; PAGE CROSSING BEISPIEL:
; Basis: $C080, X = $90 → Effektiv: $C110 → Page-Crossing! (+1 Zyklus)
; Basis: $C080, X = $7F → Effektiv: $C0FF → Kein Page-Crossing (4 Zyklen)

; Tabellen-Lookup: Sinus-Tabelle
SinusLookup:
    ; 256 Einträge, X = Winkel (0-255)
    LDA SinusTable,X  ; Sinus[X] lesen

; Optimierung: Tabelle an Seitenanfang platzieren → kein Page-Crossing möglich
; SinusTable bei $1200 → $1200 + $FF = $12FF (immer Seite $12)
        *= $1200   ; Tabelle erzwingen auf Seitenbeginn
SinusTable:
        !byte 128,131,134...  ; 256 Werte

1.10 Absolute, Y — Absolute Adressierung mit Y-Offset

Identisch mit Absolute,X, jedoch wird das Y-Register als Offset verwendet. Wichtig: Der Y-Modus wird vom VIC-II-Renderer und von Spieleroutinen intensiv genutzt, weil Y das bevorzugte Register für vertikale Iteration ist. Dieselbe Page-Crossing-Penalty gilt wie bei Absolute,X. Ein häufiges Entwurfsmuster ist die Verwendung von X für eine Tabellenauswahl und Y für den Index innerhalb der Tabelle. 

; Absolute,Y: Effektive Adresse = Basisadresse + Y
LDA $D800,Y      ; Color-RAM: Farbe an Position Y lesen
STA $D800,Y      ; Color-RAM: Farbe an Position Y schreiben

; Zeile des Screen-RAM mit Farbe füllen (Y = Spalte 0-39)
LDY #39          ; Von Spalte 39 rückwärts
LDA #7           ; Gelbe Farbe
@ColorRow:
    STA $D800,Y    ; Farbe in Color-RAM Zeile 0
    DEY
    BPL @ColorRow  ; Bis Y < 0

1.11 Indirect — Indirekter Sprung (nur JMP)

Die Indirect-Adressierung ist auf einen einzigen Befehl beschränkt: JMP. Der Befehl liest aus der angegebenen Adresse ein 16-Bit-Sprungziel und springt dorthin. Das erlaubt dynamische Sprungtabellen: Im Speicher steht nicht direkt die Zieladresse, sondern ein Zeiger auf das Ziel. Indem man den Zeiger-Wert zur Laufzeit ändert, kann man den Kontrollfluss dynamisch umlenken — ein mächtiges Werkzeug für Plugin-Systeme, Betriebssystem-Vektoren und Dispatch-Tabellen.

Berühmter 6502-Hardware-Bug: JMP ($xxFF)

Wenn das Low-Byte der Zeiger-Adresse gleich $FF ist (also die Zeiger-Adresse am Ende einer Speicherseite liegt), tritt ein Hardware-Bug auf: Das High-Byte des Sprungziels wird nicht aus $xxFF + 1 = $xx00 + $0100 gelesen, sondern aus $xx00 (Seitengrenze wird nicht überschritten!). Beispiel:

JMP ($10FF) liest Low-Byte aus $10FF, aber High-Byte aus $1000 (nicht $1100)!

Vermeidung: Zeiger-Adressen nie an Seitengrenzen platzieren ($xxFF vermeiden).

; Indirect JMP: Sprungtabelle für Unterprogramm-Dispatch
JumpTable:
    !word RoutineA          ; $0: Adresse von RoutineA (Low, High)
    !word RoutineB          ; $2: Adresse von RoutineB
    !word RoutineC          ; $4: Adresse von RoutineC

; Dispatch: A = Routine-Nummer (0, 1, 2)
Dispatch:
    ASL A               ; × 2 (weil jeder Eintrag 2 Bytes)
    TAX                 ; Offset in X
    LDA JumpTable,X     ; Zieladresse Low-Byte
    STA $FB             ; In ZP-Zeiger $FB
    LDA JumpTable+1,X   ; Zieladresse High-Byte
    STA $FC             ; In ZP-Zeiger $FC
    JMP ($FB)           ; Indirekter Sprung über ZP-Zeiger (KEIN $FF-Problem wenn $FB)

1.12 (Indirect, X) — Pre-Indexed Indirect

Dies ist die komplexeste Adressierungsart des 6510. Der 3-stufige Prozess läuft wie folgt ab: Zunächst wird der 1-Byte-Operand und das X-Register addiert (mit Wraparound in der Zero Page). Dann wird aus dieser Zero-Page-Adresse ein 16-Bit-Zeiger gelesen (Low-Byte aus der berechneten Adresse, High-Byte aus der nächsten Adresse). Schließlich wird der Datenbyte aus dieser 16-Bit-Adresse gelesen oder dorthin geschrieben. Der gesamte Prozess kostet 6 Taktzyklen. 

; (Indirect,X): Effektive Adresse = Inhalt von [($Operand + X) mod 256]
; Schritt 1: ZP-Zeiger-Adresse = ($20 + X) & $FF
; Schritt 2: Low-Byte = [$ZP-Zeiger], High-Byte = [$ZP-Zeiger + 1]
; Schritt 3: Effektive Adresse = High*256 + Low → Daten lesen/schreiben

LDA ($20,X)         ; 6 Zyklen

; Praktisches Beispiel: Dispatch-Tabelle für 4 Aktionen
; In Zero Page ab $20: 4 × 2-Byte-Zeiger auf Handler-Routinen
; $20/$21 = Zeiger auf Handler 0
; $22/$23 = Zeiger auf Handler 1
; $24/$25 = Zeiger auf Handler 2
; $26/$27 = Zeiger auf Handler 3

; Setup:
    LDA #<Handler0     ; Low-Byte von Handler0
    STA $20
    LDA #>Handler0     ; High-Byte von Handler0
    STA $21

; Aufruf mit Action-ID in X (0, 2, 4, 6 — da 2-Byte-Zeiger):
    LDX #0              ; Action 0 auswählen
    JMP ($20,X)         ; Springt zu Handler0 (liest Zeiger aus $20/$21)
    LDX #2              ; Action 1: Zeiger bei $22/$23
    JMP ($20,X)         ; Springt zu Handler1

1.13 (Indirect), Y — Post-Indexed Indirect: Der mächtigste Modus

Der Modus (Indirect),Y ist ohne Zweifel der wichtigste und am häufigsten verwendete der indirekten Adressierungsarten. Anders als (Indirect,X) ist der Prozess zweistufig: Zuerst wird aus der Zero Page ein 16-Bit-Basiszeiger gelesen (Low-Byte aus der angegebenen Adresse, High-Byte aus der nächsten). Dann wird Y zu dieser Basisadresse addiert, und aus der resultierenden Adresse werden die Daten gelesen. Dieses Muster entspricht exakt dem Konzept eines C-Zeigers mit Array-Index: *(ptr + y).

Die entscheidende Stärke: Der Zeiger in der Zero Page kann zur Laufzeit beliebig geändert werden, sodass dieselbe Code-Routine auf beliebige Datenblöcke im gesamten 64-KB-Adressraum zeigen kann. Dies ist das Fundament für allgemeine String- und Speicher-Routinen. Wie bei Absolute,Y gilt: Wenn Y + Low-Byte des Zeigers eine Seitengrenze überschreitet, kostet das einen zusätzlichen Taktzyklus. 

; (Indirect),Y: Effektive Adresse = [Inhalt($ZP, $ZP+1)] + Y
; Schritt 1: Base = {$ZP+1} * 256 + {$ZP}  (16-Bit-Zeiger aus Zero Page lesen)
; Schritt 2: Effektive Adresse = Base + Y
; Schritt 3: Daten lesen/schreiben von/nach effektiver Adresse

LDA ($FB),Y         ; 5 Zyklen (6 wenn Page-Crossing)

; === VOLLSTÄNDIGES BEISPIEL: NULL-TERMINIERTER STRING AUSGEBEN ===
; KERNAL CHROUT ($FFD2) gibt Zeichen in A aus.
; Konvention: $FB/$FC = 16-Bit-Zeiger auf String im Speicher
; String endet mit Byte $00 (Null-Terminator)

PrintString:
        ldy #0              ; Y = Index, beginnt bei 0
@loop:
        lda ($fb),y         ; Zeichen bei [Zeiger + Y] laden
        beq @done           ; Wenn Null-Byte → Ausgabe beendet
        jsr $ffd2           ; KERNAL: Zeichen aus A ausgeben
        iny                 ; Nächstes Zeichen (Y++)
        bne @loop           ; Solange Y nicht auf 0 zurückrollt (max 255 Zeichen)
@done:
        rts

; Aufruf der Routine:
Aufruf:
        lda #<MeinText      ; Low-Byte der Textadresse
        sta $fb             ; In ZP-Zeiger Low-Byte
        lda #>MeinText      ; High-Byte der Textadresse
        sta $fc             ; In ZP-Zeiger High-Byte
        jsr PrintString     ; Routine aufrufen
        rts

MeinText:
        !text "HALLO C64 MASTERY!"
        !byte 0              ; Null-Terminator

; Zweiter Aufruf — völlig andere Adresse, dieselbe Routine:
        lda #<AndererText
        sta $fb
        lda #>AndererText
        sta $fc
        jsr PrintString     ; Dieselbe Routine, anderer Zeiger!

1.14 Relative — Bedingte Sprünge mit Reichweitenbeschränkung

Alle Branch-Befehle des 6510 verwenden relative Adressierung. Der Operand ist ein vorzeichenbehaftetes 8-Bit-Offset relativ zur Adresse des nächsten Befehls (also PC + 2, da jeder Branch-Befehl 2 Bytes belegt). Das Offset reicht von -128 bis +127, was eine Sprungreichweite von -126 bis +129 Bytes relativ zur Branch-Instruktion selbst bedeutet. Der Assembler berechnet das Offset automatisch aus dem Label — man muss es nicht manuell ermitteln. 

; Branch-Befehle und ihre Bedingungen:
BEQ Label        ; Branch if Equal (Zero-Flag gesetzt)
BNE Label        ; Branch if Not Equal (Zero-Flag nicht gesetzt)
BCS Label        ; Branch if Carry Set
BCC Label        ; Branch if Carry Clear
BMI Label        ; Branch if Minus (Negative-Flag gesetzt)
BPL Label        ; Branch if Plus (Negative-Flag nicht gesetzt)
BVS Label        ; Branch if Overflow Set
BVC Label        ; Branch if Overflow Clear

; REICHWEITENPROBLEM: Wenn Label zu weit entfernt → Assembler-Fehler!
; Lösung: Logik umkehren + absoluter JMP

; Problematisch (wenn @weit_entfernt > 127 Bytes entfernt):
;   BEQ @weit_entfernt   ← Assembler-Fehler!

; Lösung: Branch umkehren + JMP
        bne @skip           ; NICHT gleich → überspringen
        jmp WeitEntfernt    ; Gleich → absoluter Sprung (3 Bytes, 3 Zyklen)
@skip:

; Nicht gesprungen (Zero-Flag gesetzt):      2 Zyklen
; Gesprungen, kein Page-Crossing:            3 Zyklen
; Gesprungen, Page-Crossing (selten!):       4 Zyklen

Abschnitt 2: Die C64-Speicherkarte im Detail

Das Speichersystem des Commodore 64 gehört zu den faszinierendsten und gleichzeitig komplexesten Designs aller 8-Bit-Heimcomputer. Auf den ersten Blick bietet der C64 einen normalen 64-KB-Adressraum, wie er vom 6510-Prozessor adressiert werden kann. Hinter dieser scheinbar einfachen Oberfläche verbirgt sich jedoch ein ausgeklügeltes Bank-Switching-System, das ROM- und RAM-Bereiche dynamisch ein- und ausblenden kann, einen separaten I/O-Bereich bereitstellt und dem Programmierer damit eine ungewöhnlich hohe Flexibilität bietet.

Die Notwendigkeit für dieses komplexe System entstand aus einem grundlegenden Dilemma: Der C64 sollte sowohl ein benutzerfreundlicher Heimcomputer (mit BASIC-Interpreter, Betriebssystem und Zeichensatz-ROM) als auch eine leistungsfähige Spiele- und Programmiermaschine sein. Beides gleichzeitig in 64 KB unterzubringen war 1982 eine erhebliche Herausforderung. Die Lösung war das Overlapping-Memory-System: ROM und RAM belegen dieselben Adressen, aber je nach Konfiguration des 6510-I/O-Ports ist entweder das ROM oder das darunterliegende RAM sichtbar. So hat der C64 physisch deutlich mehr als 64 KB — nämlich 64 KB RAM plus ROMs plus I/O — aber nur ein Fenster von 64 KB, durch das die CPU zu einem Zeitpunkt schaut.

2.2 Die Zero Page — Heiliger Boden der C64-Programmierung

Die Zero Page ist der wertvollste Speicherbereich für den Assembler-Programmierer. Jedes Byte, das hier platziert wird, ist mit der schnellstmöglichen und kompaktestmöglichen Adressierungsart erreichbar. Professionelle C64-Programmierer legen deshalb alle Performance-kritischen Variablen, Zeiger und Schleifenzähler in der Zero Page ab und benutzen den übrigen RAM für Daten und Code.

bullet_x     = $54  ; Schuss-Position X
bullet_y     = $55  ; Schuss-Position Y

2.3 Der Hardware-Stack ($0100–$01FF)

Der Stack-Bereich ist fest verdrahtet und kann nicht verlegt werden. Der Stack-Pointer (SP) ist ein 8-Bit-Register, das als Offset in den Bereich $0100–$01FF dient. Beim Reset wird SP auf $FF initialisiert, sodass der Stack bei $01FF beginnt und nach unten wächst. Jedes PHA oder jeder JSR-Rücksprungadress-Push verringert SP, jedes PLA oder jedes RTS erhöht SP wieder. 

; Stack-Mechanismus im Detail:
; PHA: Schreibe A nach [$0100 + SP], dann SP--
; PLA: SP++, dann lese A von [$0100 + SP]
; JSR: Push (PC+2) Low dann High, dann PC = Zieladresse (SP -= 2)
; RTS: Pop High dann Low, PC = Wert + 1 (SP += 2)

; Typische Verwendung: Register sichern/restaurieren
MeineRoutine:
        pha                 ; A sichern
        txa
        pha                 ; X sichern (via A)
        tya
        pha                 ; Y sichern (via A)

        ; ... eigentliche Arbeit ...

        pla
        tay                 ; Y restaurieren
        pla
        tax                 ; X restaurieren
        pla                 ; A restaurieren
        rts
Warnung: Stack-Überlauf

Der Stack hat nur 256 Bytes Kapazität. Zu viele verschachtelte JSR-Aufrufe (ohne entsprechende RTS) oder zu viele PHA ohne PLA führen dazu, dass SP auf $00 wrapat und danach in den negativen Bereich ($FF, $FE, ...) wächst — was bedeutet: Der Stack überschreibt die Zero Page! Dieser Fehler äußert sich in mysteriösem, nicht-deterministischem Programmverhalten. Maximale Schachtelungstiefe ohne zusätzliche PHA-Befehle: 128 JSR-Ebenen.

2.4 Das Bank-Switching-System — Register $0001

Das Herzstück des C64-Speicherverwaltungssystems ist das I/O-Port-Register des 6510 an Adresse $0001. Über die unteren drei Bits dieses Registers wird gesteuert, welche ROM- und I/O-Bereiche im Adressraum sichtbar sind. Dies ist eine einzigartige Eigenschaft des 6510 gegenüber dem normalen 6502: Der integrierte I/O-Port ermöglicht das Bank-Switching ohne zusätzliche externe Hardware.

; KONFIGURATION 1: Standard nach Reset ($37)
; BASIC + KERNAL + I/O sichtbar
        lda #$37
        sta $01

; KONFIGURATION 2: Nur RAM (alle ROMs ausgeblendet)
; Maximaler freier RAM: 64 KB minus Stack, ZP, I/O-Bereich
        lda #$30
        sta $01
        ; ACHTUNG: Ab hier kein KERNAL mehr! JSR $FFD2 geht in eigenen Code!

; KONFIGURATION 3: Zeichensatz-ROM lesen ($33)
; Nützlich um den ROM-Zeichensatz in eigenes RAM zu kopieren
        lda $01
        and #%11111100     ; Bits 2-0: I/O aus, Charset-ROM ein
        ora #%00000011     ; HIRAM+LORAM setzen für BASIC+KERNAL ROM
        ; Ergebnis: %xxxxxx11 aber mit CHAREN=0 → $33
        lda #$33           ; Einfacher: direkt setzen
        sta $01
        ; Jetzt: Charset-ROM bei $D000 lesbar, I/O NICHT verfügbar!
        ; VIC-II, SID, CIA sind jetzt NICHT erreichbar bis...
        lda #$37           ; ...I/O wieder einblenden
        sta $01

; SAFE BANKING: Nur Bits 0-2 ändern, andere Bits (Datenleitungen) bewahren
SetBankingConfig:
        ; A = gewünschte Konfiguration (Bits 0-2)
        pha
        lda $01
        and #%11111000     ; Bits 0-2 löschen
        sta $fb             ; Aktuellen Wert (maskiert) speichern
        pla
        and #%00000111     ; Nur Bits 0-2 von Eingabe
        ora $fb             ; Kombinieren
        sta $01
        rts

2.5 VIC-II Speicher-Banking via CIA 2

Zusätzlich zum CPU-Banking gibt es ein separates Speicher-Banking für den VIC-II-Grafikprozessor. Der VIC-II kann nur einen 16-KB-Bereich gleichzeitig "sehen" — seine VIC-Bank. Die Auswahl dieser Bank erfolgt über die unteren zwei Bits von CIA 2 Port A ($DD00). Wichtig: Die VIC-Bank-Bits sind invertiert — Bit-Wert 0 bedeutet die Bank ist aktiv.

$DD00 Bits 1-0VIC-BankAdressbereichCharset-ROM
%11Bank 0$0000–$3FFFJa, bei $1000–$1FFF und $1800–$1FFF
%10Bank 1$4000–$7FFFNein (nur RAM)
%01Bank 2$8000–$BFFFJa, bei $9000–$9FFF
%00Bank 3$C000–$FFFFNein (nur RAM)
 
; VIC-Bank 2 auswählen ($8000-$BFFF):
        lda $dd00           ; CIA2 Port A lesen
        and #%11111100     ; Bits 1-0 löschen
        ora #%00000001     ; %01 → Bank 2
        sta $dd00

; VIC-II Zeiger-Register $D018 setzen:
; Screen-RAM in Bank 2 bei $8C00 → Offset = $0C00 → $0C00/1024 = 3 → Bits 7-4 = 0011
; Charset in Bank 2 bei $9000 → Offset = $1000 in der Bank → $1000/2048 = 2 → Bits 3-1 = 010
        lda #%00110100     ; Screen=3 (Bits 7-4=0011), Charset=2 (Bits 3-1=010), Bit0=0
        sta $d018

; Zur Bank 0 zurück (Standard):
        lda $dd00
        and #%11111100
        ora #%00000011     ; %11 → Bank 0 (Standard)
        sta $dd00

; Standard-Wert $D018 für Bank 0, Screen=$0400, Charset=$1000:
; Screen-Offset: $0400/1024 = 1 → Bits 7-4 = 0001
; Charset-Offset: $1000/2048 = 2 → Bits 3-1 = 010 (ROM-Zeichensatz!)
; $D018 = %00010100 = $14 ← Standardwert nach Reset
