Spielentwicklung von A bis Z

★ Meister-Niveau ⏱ ca. 12–16 Stunden 📋 Voraussetzung: Stufen 00–10

Lernziele dieser Stufe

  • Ein vollständiges Game Design Document (GDD) für ein C64-Spiel erstellen
  • Speicher-Layout und Modul-Struktur eines Projekts professionell planen
  • Asset-Pipeline von Grafik-Editoren bis zur ACME-Integration beherrschen
  • Die richtige Entwicklungsreihenfolge kennen und anwenden
  • Systematisch mit VICE debuggen und Cycle-Budgets überwachen
  • Ein fertiges Spiel als D64-Image verpacken und veröffentlichen
  • Einen vollständigen Plattformer von der Idee bis zum Release entwickeln

1. Vom Konzept zum Game Design Document

Der häufigste Fehler angehender C64-Entwickler: sofort mit dem Code anfangen. Das Ergebnis sind unfertige Projekte, die an der fehlenden Planung scheitern — nicht am fehlenden technischen Wissen. Professionelle Spielentwicklung beginnt immer auf Papier, und das gilt für den C64 genauso wie für moderne Plattformen.

1.1 Die Spielidee konkretisieren

Eine gute C64-Spielidee berücksichtigt die Hardware-Grenzen als kreative Kraft, nicht als Hindernis. Die Grenzen des C64 zwingen zu Designentscheidungen, die Spiele oft prägnanter und spielbarer machen als ihre aufgeblasenen modernen Gegenstücke.

Stelle dir diese vier Fragen, bevor du eine Zeile Code schreibst:

  • Was ist der Kern-Spielspaß? Beschreibe ihn in einem einzigen Satz. Wenn du ihn nicht in einem Satz ausdrücken kannst, ist die Idee noch nicht reif.
  • Welches Genre? Plattformer, Shoot'em'up, Puzzle, Adventure, Arcade — jedes Genre hat bewährte C64-Muster, von denen man lernen kann.
  • Wieviele Sprites brauchst du? 8 Hardware-Sprites sind dein Limit (ohne Multiplexer). Das bestimmt die Komplexität der Grafik-Engine.
  • Wie groß wird das Spiel? Ein Level? Mehrere? Disk-Laden nötig? Das entscheidet über die Architektur.
Tipp: Kleine Spiele fertigstellen
Dein erstes vollständiges C64-Spiel sollte in 4–8 Wochen fertig sein. Wähle eine Idee, die du in dieser Zeit umsetzen kannst. Ein fertiges einfaches Spiel ist wertvoller als ein unfertiges ambitioniertes Projekt.

1.2 Das Game Design Document (GDD) für den C64

Ein GDD für den C64 muss keine 50-seitige Abhandlung sein. Für ein Soloprojekt reicht ein einziges Dokument — aber es muss schriftlich vorliegen. Das zwingt dich, vage Ideen in konkrete Entscheidungen zu übersetzen.

Hier ist eine bewährte GDD-Vorlage, die auch als Kommentarblock in der Haupt-ASM-Datei leben kann:

; ================================================================
; GAME DESIGN DOCUMENT — "JUMPMASTER"
; Version: 1.0 | Status: In Entwicklung | Ziel: CSDb-Release
; ================================================================
;
; KONZEPT:
;   Ein 1-Screen-Plattformer. Der Spieler springt durch
;   automatisch generierende Plattformen nach oben.
;   Je höher, desto schneller. Kein Ende — nur ein Score.
;
; GENRE:        Endless Plattformer
; SPIELER:      1 (Joystick Port 2)
; GRAFIK:       Zeichenmodus (40x25), benutzerdefinierter Charset
; SOUND:        1 Melodie (3 Stimmen), 2 Sound-Effekte
; LEVEL:        Endlos, prozedural (kein Disk-Laden nötig)
; SPEICHER:     64 KB, kein Bank-Switching benötigt
;
; SPIELMECHANIK:
;   - Joystick: Links/Rechts bewegen, Feuer = Sprung
;   - Plattformen bewegen sich von oben nach unten
;   - Spieler fällt durch den unteren Rand = Game Over
;   - Score: +10 Punkte pro übersprungene Plattform
;
; TECHNISCHE ZIELE:
;   - 50 Hz stabil (kein Frame-Overrun)
;   - Max. 8 Sprites (Spieler + 7 Plattform-Highlights)
;   - Kein Multiplexer benötigt
;
; MEILENSTEINE:
;   Woche 1: Framework + Spieler-Bewegung
;   Woche 2: Plattform-Logik + Kollision
;   Woche 3: Score + Game Over + Musik
;   Woche 4: Polish, Test, Release
; ================================================================

1.3 Feature-Priorisierung: MoSCoW auf 64K anwenden

Die MoSCoW-Methode hilft, den Scope zu begrenzen. Jedes Feature landet in einer von vier Kategorien:

Must
Das Spiel ist ohne dieses Feature nicht spielbar: Spielerbewegung, Kollision, Game Over, Score-Anzeige
Should
Wichtig für Spielspaß, aber technisch verzichtbar: Musik, Titel-Screen, Highscore-Speicherung
Could
Nette Extras, wenn Zeit bleibt: Animierte Sprites, Zwischen-Screens, Bonus-Items
Won't
Explizit ausgeschlossen: Disk-Laden, mehr als 1 Level, Multi-Player, Cheats
Warnung: Scope Creep
Jede Woche willst du ein neues "Must-Have"-Feature hinzufügen. Schreib deine ursprüngliche Won't-Liste auf und halte sie ein. Fertigstellen ist wichtiger als Feature-Vollständigkeit.

1.4 Scope-Realismus — Was ist allein machbar?

ProjektgrößeZeitaufwandTypisches BeispielTechniken
Micro1–2 Wochen1-Screen-Arcade, PuzzleKeine Sprites-Engine, 1 Screen
Klein1–2 MonatePlattformer, Shoot'em'upSprites, einfache Tilemap
Mittel3–6 MonateScrolling-Spiel, AdventureMultiplexer, Disk-I/O, mehrere Level
Groß1–2 Jahrevollständiges RPG, StrategieAlles + Custom-Loader, riesige Daten

Als Einzelentwickler ist ein "Klein"-Projekt das realistische Ziel für den Anfang. Die meisten legendären C64-Spiele entstanden in Teams von 2–5 Personen über 6–18 Monate.

2. Technische Planung — Bevor die erste Zeile Code geschrieben wird

Technische Planung ist die Übersetzung des GDD in konkrete Speicher-Adressen, Dateistrukturen und Modulabhängigkeiten. Dieser Schritt ist langweilig, aber er entscheidet darüber, ob du später einfach neue Features hinzufügen kannst oder ob du ständig Code umstrukturieren musst.

2.1 Die Memory-Map als Projektplan

Plane das Memory-Layout, bevor du Code schreibst. Konflikte zwischen Modulen entdeckst du am besten auf Papier, nicht zur Laufzeit. Die Memory-Map sollte als ACME-Konstanten-Datei im Projekt existieren:

; ================================================================
; memory_map.asm — Zentrale Speicher-Konstantendatei
; Alle Module !source diese Datei als erstes
; ================================================================

; ── System-Bereiche (fix) ────────────────────────────────────
SCREEN_RAM      = $0400      ; 40x25 Zeichen
COLOR_RAM       = $D800      ; Farb-RAM (fix)
BASIC_START     = $0801      ; BASIC-Stub startet hier

; ── Zero Page — Spielvariablen ($30–$7F für uns) ─────────────
ZP_PLAYER_X     = $30        ; Spieler X-Position (Pixel)
ZP_PLAYER_Y     = $31        ; Spieler Y-Position (Pixel)
ZP_PLAYER_VY    = $32        ; Vertikalgeschwindigkeit (signed)
ZP_SCORE_LO     = $33        ; Score Low-Byte (BCD)
ZP_SCORE_HI     = $34        ; Score High-Byte (BCD)
ZP_FRAME_CNT    = $35        ; Frame-Zähler (0–49, für 50 Hz)
ZP_GAME_STATE   = $36        ; Aktueller Spielzustand
ZP_PTR_LO       = $FC        ; Universalzeiger Low (Standard-ZP)
ZP_PTR_HI       = $FD        ; Universalzeiger High

; ── Spielzustände ────────────────────────────────────────────
STATE_TITLE     = 0
STATE_GAME      = 1
STATE_GAMEOVER  = 2

; ── Modul-Adressen ───────────────────────────────────────────
MODULE_GAME     = $0900      ; Game-Loop
MODULE_INPUT    = $0A00      ; Input-Handler
MODULE_PLATFORM = $0B00      ; Plattform-Engine
MODULE_SID      = $1000      ; SID-Player (extern)

; ── Asset-Bereiche ────────────────────────────────────────────
CHARSET_DATA    = $2000      ; Zeichensatz (2 KB)
SPRITE_DATA     = $2800      ; Sprites (64*8 = 512 Bytes)
MUSIC_DATA      = $3000      ; SID-Musik-Daten (bis $3FFF)

2.2 Modul-Aufteilung und Multi-File-Projekte mit ACME

Ein professionelles Projekt besteht aus mehreren ASM-Dateien, die per !source zusammengebaut werden. Das ermöglicht unabhängige Entwicklung einzelner Module und schnellere Kompilierung durch modulares Denken:

; ================================================================
; main.asm — Einstiegspunkt des Projekts
; Kompilieraufruf: acme --cpu 6510 -f cbm -o jumpmaster.prg main.asm
; ================================================================

; Zieladresse: BASIC-Programmstart
        * = $0801

; 1. Konstanten und Speicher-Map zuerst
        !source "memory_map.asm"

; 2. BASIC-Stub (SYS 2062)
        !byte $0B,$08,$0A,$00,$9E,$32,$30,$36,$32,$00,$00,$00

; 3. Initialisierung
        * = $080E
        !source "init.asm"

; 4. Game Loop
        * = MODULE_GAME
        !source "gameloop.asm"

; 5. Einzelne Module
        * = MODULE_INPUT
        !source "input.asm"

        * = MODULE_PLATFORM
        !source "platforms.asm"

; 6. Asset-Daten (binär eingebettet)
        * = CHARSET_DATA
        !bin "assets/charset.bin"

        * = SPRITE_DATA
        !bin "assets/sprites.bin"

; 7. Musik zuletzt (kann groß sein)
        * = MUSIC_DATA
        !source "assets/musik.asm"

2.3 Meilensteinplan: Vertikal vs. horizontal

Es gibt zwei Ansätze für den Entwicklungsplan:

  • Horizontal: Erst alle Systeme aufbauen, dann Inhalte einfügen. Risiko: Du baust Systeme, die du nie brauchst.
  • Vertikal (empfohlen): Zuerst ein spielbares Mini-Spiel in voller Tiefe, dann erweitern. Du hast immer etwas Spielbares.

3. Die Asset-Pipeline — Grafik und Sound produzieren

Assets — Sprites, Zeichensätze, Tilemaps, Musik — entstehen in spezialisierten Werkzeugen und müssen in ein Format überführt werden, das ACME in die PRG-Datei einbetten kann. Diese Pipeline zu kennen und zu beherrschen ist entscheidend für professionelle Ergebnisse.

+──────────────+ Export +──────────────+ !bin +──────────────+ │ CharPad │ ─────────────► │ charset.bin │ ─────────── ► │ PRG-Datei │ │ (Charset) │ .bin-Datei │ (2048 Bytes)│ │ (ACME out) │ +──────────────+ +──────────────+ │ │ │ │ +──────────────+ Export +──────────────+ !bin │ │ │ SpritePad │ ─────────────► │ sprites.bin │ ─────────── ► │ │ │ (Sprites) │ .bin-Datei │ (N*64 Bytes)│ │ │ +──────────────+ +──────────────+ │ │ │ │ +──────────────+ Export+Asm +──────────────+ !source │ │ │ GoatTracker │ ─────────────► │ musik.asm │ ─────────── ► │ │ │ (Musik) │ .asm + Daten │ (Player+Dat)│ +──────────────+ +──────────────+ +──────────────+

3.1 Sprite-Design: SpritePad → ACME-Integration

SpritePad (von Spritemate) ist das Standardwerkzeug für C64-Sprites. Es ermöglicht das Zeichnen aller 8 Hardware-Sprites mit Farb-Vorschau und exportiert als binäre .bin-Datei oder direkt als ACME-!byte-Sequenz.

Export-Schritte in SpritePad:

  1. Sprites zeichnen (24×21 Pixel, Hires oder Multicolor)
  2. File → Export → Binary → Datei als assets/sprites.bin speichern
  3. Größe prüfen: N Sprites × 64 Bytes. SpritePad fügt Padding auf 64 Bytes auf.
; Sprite-Daten einbinden (z.B. 4 Sprites = 256 Bytes)
        * = SPRITE_DATA         ; = $2800
        !bin "assets/sprites.bin", 256   ; Genau 256 Bytes einlesen

; Sprite-Pointer setzen (Pointer-Seite: $07F8–$07FF)
; Sprite 0 liegt bei $2800 → Block $2800/64 = $A0 = 160
init_sprites:
        lda #160                ; Sprite-Block-Nummer
        sta $07F8              ; Sprite 0 Pointer
        sta $07F9              ; Sprite 1 Pointer
        lda #%00000011         ; Sprites 0 und 1 einschalten
        sta $D015
        rts

3.2 Charset-Design: CharPad → Export

CharPad (von Subchrist Software) ist der Standard für C64-Zeichensätze und Tilemaps. Es ermöglicht das Zeichnen von bis zu 256 eigenen Zeichen mit C64-Farb-Preview und exportiert Charset-Daten, Tilemap-Daten und Screen-RAM getrennt.

Wichtigste Export-Formate aus CharPad:

Export-TypDateiGrößeVerwendung
Charset (Zeichensatz)charset.bin2048 Bytes (256 Zeichen × 8)Wird in VIC-II-Charset-Bereich geladen
Map (Tilemap)map.binBreite × Höhe BytesLevel-Daten
Screen RAMscreen.bin1000 Bytes (40×25)Direkte Screen-RAM-Initialisierung
 
; Charset in VIC-II-Bank laden (hier: $2000)
load_charset:
        lda #$36              ; Kernal aus, RAM an $D000
        sta $01               ; Charset-ROM ausblenden
        ldy #0
.loop:
        lda charset_src,y    ; Quelle (eingebettete Daten)
        sta CHARSET_DATA,y   ; Ziel: $2000
        iny
        bne .loop
        lda #$37              ; Kernal wieder einschalten
        sta $01
        rts

; Charset-Daten eingebettet
charset_src:
        !bin "assets/charset.bin"   ; 2048 Bytes

; VIC-II auf Charset-Position $2000 zeigen lassen:
; $D018: Bits 3-1 = Charset-Block → $2000 = Bank 0 + Block 4
; Wert: Screen-RAM $0400 (Bits 7-4 = 1) + Charset $2000 (Bits 3-1 = 4)
; = %00010100 = $14
        lda #$14
        sta $D018

3.3 Musik-Pipeline: GoatTracker → SID-Daten einbinden

GoatTracker ist ein moderner SID-Tracker, der präzise Kontrolle über alle SID-Parameter ermöglicht. Er exportiert Musik als Player + Daten in einem Assembly-Format, das direkt mit ACME eingebunden werden kann.

Export-Workflow in GoatTracker:

  1. Musik fertig komponieren und speichern (.sng-Format)
  2. File → Export Song → Format: ACME → Datei: assets/musik.asm
  3. GoatTracker exportiert: Musik-Player-Code + Daten in einer ASM-Datei
  4. Die exportierte Datei definiert drei Labels: SID_INIT, SID_PLAY, SID_DATA
; Musik-Player in den IRQ einbinden
; Schritt 1: Beim Start initialisieren (Lied 0 starten)
init_music:
        lda #0               ; Lied-Nummer 0
        jsr SID_INIT         ; GoatTracker Init-Routine
        rts

; Schritt 2: Im Raster-IRQ jeden Frame aufrufen
irq_handler:
        lda #$01
        sta $D019            ; IRQ quittieren

        jsr SID_PLAY         ; Musik einen Frame vorspielen
        jsr game_update      ; Spiellogik

        jmp $EA81            ; Weiter mit Standard-KERNAL-IRQ

; Schritt 3: GoatTracker-Code einbinden
        * = MUSIC_DATA       ; = $3000
        !source "assets/musik.asm"  ; Definiert SID_INIT, SID_PLAY

4. Entwicklungsreihenfolge — Das Gerüst zuerst

Die Reihenfolge, in der du Features entwickelst, ist nicht beliebig. Die falsche Reihenfolge führt dazu, dass du bereits fertigen Code wiederholt umschreiben musst. Die richtige Reihenfolge bedeutet: jedes neue Feature baut sauber auf dem vorherigen auf.

1

Phase 1 — Technisches Gerüst

IRQ aufsetzen, VIC-II konfigurieren, leere Game-Loop implementieren. Ziel: VICE zeigt den korrekten Hintergrundfarbe und der IRQ läuft stabil. Kein Gameplay, noch kein Sprite.

2

Phase 2 — Audio-System

Musik-Player einbinden und im IRQ aufrufen. Ziel: GoatTracker-Musik spielt. Früh testen, weil die Musik-Routine erhebliche Zyklen verbraucht — du musst wissen, wie viel dir für das Gameplay bleibt.

3

Phase 3 — Grafik-Grundlage

Charset laden, Screen-RAM initialisieren, Sprites einrichten. Ziel: Spieler-Sprite erscheint an einer festen Position auf korrekt gezeichnetem Hintergrund.

4

Phase 4 — Spielmechanik

Input, Bewegung, Physik. Ziel: Spieler bewegt sich flüssig, springt realistisch, landet korrekt auf Plattformen. Das ist das Herzstück — hier verbringt man am meisten Zeit.

5

Phase 5 — Inhalte

Level-Daten, Feinde, Items, Score. Jetzt gibt es ein Framework, in das man Inhalte einfüllen kann. Iteration: Feature hinzufügen → testen → nächstes Feature.

6

Phase 6 — Polish & Release

Titel-Screen, Game-Over-Screen, Highscore, Soundeffekte, Bugfixes. Erst jetzt optimieren. Danach: D64-Image erstellen, auf echter Hardware testen, veröffentlichen.

4.1 Phase 1 im Detail: Das leere Framework

Das Framework ist das Fundament, auf dem alles andere aufbaut. Es enthält: BASIC-Stub, IRQ-Setup, VIC-II-Grundkonfiguration und die Game-Loop-Schleife. Erst wenn dieses Gerüst stabil läuft, kommt Schicht für Schicht dazu.

; ================================================================
; init.asm — System-Initialisierung
; ================================================================

start:
        ; CPU-Interrupts deaktivieren während Setup
        sei

        ; VIC-II: Hintergrundfarbe, Rahmenfarbe
        lda #0               ; Schwarz
        sta $D020            ; Rahmen
        sta $D021            ; Hintergrund

        ; Raster-IRQ einrichten
        lda #<sta $0314            ; IRQ-Vektor Low
        lda #>irq_handler
        sta $0315            ; IRQ-Vektor High
        lda #252             ; Rasterzeile für IRQ
        sta $D012
        lda $D011
        and #$7F             ; Bit 8 von $D012 = 0
        sta $D011
        lda #$01             ; Raster-IRQ aktivieren
        sta $D01A

        ; Startzustand setzen
        lda #STATE_TITLE
        sta ZP_GAME_STATE

        ; Musik initialisieren
        jsr init_music

        ; IRQ wieder erlauben
        cli

main_loop:
        ; Game-Loop wartet auf IRQ-Flag
        lda ZP_FRAME_CNT     ; Wurde ein neuer Frame gezeichnet?
        beq main_loop        ; Nein: warten
        lda #0
        sta ZP_FRAME_CNT     ; Frame-Flag zurücksetzen
        jsr game_dispatch    ; State Machine aufrufen
        jmp main_loop

irq_handler:
        lda #$01
        sta $D019            ; IRQ quittieren
        jsr SID_PLAY
        lda #1
        sta ZP_FRAME_CNT     ; Neuen Frame signalisieren
        jmp $EA81            ; KERNAL-IRQ-Ende

4.2 Phase 4: Sprung-Physik mit Gravitation

Gute Sprungphysik ist das Herzstück jedes Plattformers. Auf dem C64 simulieren wir Gravitation mit einem signed Velocity-Byte und einem Frame-for-Frame Update:

; ── Physik-Konstanten ────────────────────────────────────────
GRAVITY         = 1          ; Pixel/Frame² nach unten
JUMP_VELOCITY   = $F2       ; -14 als Zweierkomplement (signed)
MAX_FALL_SPEED  = 8          ; Maximale Fallgeschwindigkeit
GROUND_Y        = 220        ; Y-Pixel-Koordinate des Bodens

; ── Spieler-Physik pro Frame ─────────────────────────────────
update_physics:
        ; Gravitation anwenden (VY += GRAVITY)
        clc
        lda ZP_PLAYER_VY
        adc #GRAVITY
        sta ZP_PLAYER_VY

        ; Fallgeschwindigkeit begrenzen (signed, daher: VY < MAX_FALL_SPEED)
        cmp #MAX_FALL_SPEED
        bmi .no_cap         ; Noch unter Max: ok
        lda #MAX_FALL_SPEED
        sta ZP_PLAYER_VY
.no_cap:
        ; Neue Y-Position berechnen (Y += VY)
        clc
        lda ZP_PLAYER_Y
        adc ZP_PLAYER_VY    ; signed Addition: VY kann negativ sein
        sta ZP_PLAYER_Y

        ; Boden-Kollision prüfen
        cmp #GROUND_Y
        bcc .in_air         ; Noch in der Luft
        lda #GROUND_Y
        sta ZP_PLAYER_Y     ; Auf Boden setzen
        lda #0
        sta ZP_PLAYER_VY    ; Geschwindigkeit auf 0
.in_air:
        rts

; ── Sprung auslösen (nur wenn am Boden) ─────────────────────
do_jump:
        lda ZP_PLAYER_Y
        cmp #GROUND_Y       ; Steht Spieler auf dem Boden?
        bne .no_jump        ; Nein: kein Doppelsprung
        lda #JUMP_VELOCITY   ; Negative Geschwindigkeit = nach oben
        sta ZP_PLAYER_VY
.no_jump:
        rts

5. Testing und Debugging — Systematisch vorgehen

Debugging auf dem C64 ist eine eigene Disziplin. Ohne die richtigen Werkzeuge und Techniken verbringt man mehr Zeit mit der Fehlersuche als mit dem eigentlichen Code. Der VICE-Monitor ist dabei dein mächtigstes Werkzeug.

5.1 VICE-Monitor: Die wichtigsten Befehle

Den Monitor öffnest du in VICE mit Alt+M oder von BASIC aus mit SYS 1 (nach dem Laden deines Programms). Die wichtigsten Befehle:

5.2 Cycle-Budget überwachen

Ein PAL-Frame hat 63 × 312 = 19.656 Taktzyklen. Davon stehen für deinen Code effektiv etwa 15.000 frei (der Rest geht für VICE/DMA). Wenn dein Game-Loop zu viele Zyklen braucht, werden Frames übersprungen und das Spiel ruckelt.

Die eleganteste Methode zur Cycle-Messung: Den Rahmen als Stoppuhr nutzen. Erster IRQ setzt die Rahmenfarbe auf Weiß, zweiter IRQ auf Schwarz zurück. Die Breite des weißen Streifens im Rahmen zeigt direkt die Laufzeit an:

; Cycle-Budget-Messung — Debugging-Tool
; IRQ 1: Messung starten
irq_measure_start:
        lda #1              ; Rahmen = Weiß: Messung beginnt
        sta $D020
        jsr game_update     ; ← HIER wird der Code gemessen
        lda #0              ; Rahmen = Schwarz: Messung endet
        sta $D020
        jmp $EA81

; Schmaler weißer Streifen = wenige Zyklen = gut
; Breiter weißer Streifen = viele Zyklen = Frame-Overrun droht
; Streifen reicht bis zum unteren Rand = Frame-Overrun: Problem!

5.3 Playtest-Checkliste vor jedem Release

Teste jeden Game-State explizit durch. Was offensichtlich erscheint, wird überraschend oft übersehen:

  • Spiel startet ohne Probleme nach dem Laden
  • Titel-Screen zeigt sich korrekt, alle Texte lesbar
  • Spielstart funktioniert (Joystick-Feuer auf Titel-Screen)
  • Alle Bewegungsrichtungen des Spielers reagieren korrekt
  • Sprung funktioniert, Doppelsprung ist nicht möglich
  • Kollision mit Plattformen von oben korrekt
  • Spieler fällt durch Plattformen von unten/seitlich korrekt durch
  • Score erhöht sich korrekt
  • Game Over wird korrekt ausgelöst
  • Game Over Screen erscheint und ist verlassbar
  • Musik spielt während des gesamten Spiels ohne Unterbrechung
  • Kein Frame-Ruckeln bei vollem Bildschirm (Plattformen + Spieler)
  • Auf NTSC (VICE: Settings → Machine → Video Standard → NTSC) getestet

6. Distribution — Vom PRG zur Veröffentlichung

Das fertige Spiel als D64-Image zu verpacken und auf CSDb zu veröffentlichen ist der letzte Schritt — und wird von vielen Entwicklern unterschätzt. Ein schlecht verpacktes Spiel wird seltener gespielt und weniger wahrgenommen.

6.1 PRG-Datei prüfen

Überprüfe deine PRG-Datei, bevor du sie in ein D64-Image packst:

# Linux/macOS: PRG-Datei-Informationen
# Die ersten zwei Bytes der PRG sind die Lade-Adresse:
xxd jumpmaster.prg | head -1
# Erwartete Ausgabe: 01 08 ... (= $0801 = BASIC-Start)

# Dateigröße in VICE testen:
x64sc jumpmaster.prg
# Dann in VICE: LOAD"*",8,1 → RUN
# Alternativ: Drag & Drop in VICE-Fenster

6.2 D64-Image erstellen

Ein D64-Image (1541-Diskette) enthält bis zu 664 KB Nutzdaten auf 35 Spuren. Das Tool c1541 (Teil von VICE) erstellt und verwaltet D64-Images:

# Neues D64-Image erstellen und Dateien hinzufügen
c1541 -format "jumpmaster,01" d64 jumpmaster.d64

# Haupt-PRG auf der Diskette ablegen
c1541 -attach jumpmaster.d64 -write jumpmaster.prg "jumpmaster"

# Disketten-Verzeichnis prüfen
c1541 -attach jumpmaster.d64 -dir

# Erwartete Ausgabe:
# 0 "JUMPMASTER       01" 2A 2A
# 12 "JUMPMASTER"         PRG
# 649 BLOCKS FREE.

# AUTOSTART einrichten: Wenn die Diskette eingelegt wird, soll
# das Spiel automatisch starten. Erstelle eine AUTOSTART-Datei:
# (Dateiname: "autostart" mit BASIC-Inhalt: LOAD"JUMPMASTER",8,1:RUN)

7.3 Woche 1: Framework + Spielerbewegung

Tag 1–2: ACME-Projektstruktur anlegen (main.asm, memory_map.asm, init.asm). BASIC-Stub schreiben, Bildschirm auf Schwarz setzen, Raster-IRQ einrichten. Kompilieren: läuft und zeigt schwarzen Bildschirm.

Tag 3: Charset laden, Spieler-Sprite einrichten. Spieler erscheint als weißes Rechteck in der Bildschirmmitte.

Tag 4–5: Input-Handler schreiben. Joystick links/rechts bewegt Sprite. Frame-Counter im IRQ.

Tag 6–7: Sprung-Physik (Code von oben). Spieler springt und fällt zurück auf den Boden.

Meilenstein M1 erreicht wenn: Spieler springt und landet auf dem Boden-Tile, Bewegung links/rechts flüssig.

7.4 Woche 2: Plattform-Logik + Kollision

Die Plattform-Engine verwaltet 16 Plattformen in einem Ring-Puffer. Jede Plattform hat X-Position, Y-Position und Geschwindigkeit (alle nach unten). Wenn eine Plattform den unteren Bildschirmrand verlässt, wird sie oben neu positioniert.

; Plattform-Datenstuktur (16 Bytes pro Plattform)
; Offset 0: X-Position (Zeichen-Koordinate, 0–35)
; Offset 1: Y-Position (Pixel, 0–255)
; Offset 2: Breite (2–6 Zeichen)
; Offset 3: Flags (Bit 0 = aktiv)

NUM_PLATFORMS   = 8
PLAT_STRIDE     = 4          ; Bytes pro Plattform-Eintrag

platforms:      ; 8 Plattformen × 4 Bytes
        !fill NUM_PLATFORMS * PLAT_STRIDE, 0

update_platforms:
        ldx #0               ; Plattform-Index
.loop:
        lda platforms+3,x   ; Flags lesen
        and #$01
        beq .next           ; Nicht aktiv: überspringen

        ; Y-Position um 1 erhöhen (nach unten)
        inc platforms+1,x

        ; Untere Grenze prüfen
        lda platforms+1,x
        cmp #220
        bcc .next           ; Noch im Bild

        ; Plattform oben neu platzieren
        lda #20              ; Neue Y-Position oben
        sta platforms+1,x
        ; Neue X-Position zufällig (Pseudo-RNG via Uhr)
        lda $DC04            ; CIA1 Timer A Low = Zufallswert
        and #$1F             ; Modulo 32 → Wert 0–31
        sta platforms,x
.next:
        txa
        clc
        adc #PLAT_STRIDE
        tax
        cpx #(NUM_PLATFORMS * PLAT_STRIDE)
        bcc .loop
        rts

7.5 Woche 3: Score, HUD, Game Over, Musik

Score wird im BCD-Modus addiert und im Dezimalmodus auf dem Bildschirm angezeigt. Sechs BCD-Ziffern = bis zu 999.999 Punkte.

; Score um 10 erhöhen (BCD-Modus)
add_score:
        sed                  ; Dezimal-Modus an
        clc
        lda ZP_SCORE_LO
        adc #$10             ; +10 im BCD-Format
        sta ZP_SCORE_LO
        lda ZP_SCORE_HI
        adc #0
        sta ZP_SCORE_HI
        cld                  ; Dezimal-Modus aus
        rts

; Score auf Bildschirm ausgeben (Position: Zeile 0, Spalte 32)
display_score:
        lda ZP_SCORE_HI
        lsr a
        lsr a
        lsr a
        lsr a               ; High-Nibble
        ora #$30             ; → PETSCII Zifferzeichen
        sta SCREEN_RAM + 32 ; Bildschirmposition
        ; ... (weitere Nibbles analog)
        rts

7.6 Woche 4: Polish, Test, Release-Build

Schritt 1 — Bugs beheben: Alle Playtest-Checkliste-Punkte durchgehen. Jeden gefundenen Bug sofort beheben und danach erneut vollständig testen.

Schritt 2 — Build-Skript erstellen: Ein Shell-Skript automatisiert den Build:

#!/bin/bash
# build.sh — JUMPMASTER Build-Skript
set -e  # Abbrechen bei Fehler

# Kompilieren
acme --cpu 6510 -f cbm -o jumpmaster.prg main.asm
echo "Kompilierung erfolgreich: $(wc -c < jumpmaster.prg) Bytes"

# D64-Image erstellen
rm -f jumpmaster.d64
c1541 -format "jumpmaster,01" d64 jumpmaster.d64 \
      -write jumpmaster.prg "jumpmaster"
echo "D64-Image erstellt: jumpmaster.d64"

# Starten
x64sc jumpmaster.d64 &

Schritt 3 — Auf echter Hardware testen: Übertrage das D64-Image auf echte Hardware (1541Ultimate, SD2IEC) und spiele das Spiel auf einem echten C64. Abweichungen vom Emulator-Verhalten sind hier gut zu finden.

Schritt 4 — NFO schreiben, CSDb einreichen. Glückwunsch: das erste C64-Spiel ist fertig!

Übungsaufgaben

Übung 11.1 — Das Micro-Spiel

Erstelle ein komplettes GDD für einen einfachen Pong-Clone (1 Seite). Implementiere das Spiel in einer einzigen Woche. Das Ziel ist nicht Perfektion, sondern Fertigstellung: mit Titel-Screen, zwei Paddle-KIs und Score.

Übung 11.2 — Asset-Pipeline Übung

Lade CharPad und SpritePad herunter. Zeichne einen 16-Zeichen-Tileset (Plattform-Tiles: Boden, Wand, Decke, Ecken). Exportiere als .bin und binde die Datei mit !bin in ein ACME-Projekt ein. Zeige den Tileset auf dem Bildschirm an.

Übung 11.3 — Cycle-Budget-Analyse

Implementiere die Rahmenfarben-Messmethode in einem bestehenden Projekt. Messe, wie viele Zyklen deine Game-Loop-Routine verbraucht. Optimiere sie um mindestens 20%, und messe erneut. Dokumentiere, welche Änderungen den größten Effekt hatten.
