1. Der IEC-Seriell-Bus

Der C64 kommuniziert mit Laufwerken (und Druckern) über den IEC-Bus (International Electrotechnical Commission) — eine serielle Datenleitung mit drei aktiven Signalen, alle auf dem CIA2-Chip ($DD00).

CIA2 $DD00 — IEC-Bus-Leitungen:

Bit 7: DATA IN (Lesen, ob Data-Leitung Low ist)
Bit 6: CLK IN (Lesen, ob Clock-Leitung Low ist)
Bit 5: DATA OUT (0 schreiben → Leitung auf Low ziehen)
Bit 4: CLK OUT (0 schreiben → Leitung auf Low ziehen)
Bit 3: ATN OUT (Attention — Geräteauswahl/Kommando-Start)
Bit 0–1: VIC-II Bank-Select (Bits invertiert)

Alle drei Signale sind Open-Collector mit Pull-Up: Mehrere Geräte können die Leitung auf Low ziehen, niemand kann sie aktiv auf High treiben — High entsteht, wenn niemand zieht. Daher "wired AND".

1.1 Bustopologie

Am IEC-Bus können bis zu 15 Geräte mit eigenen Gerätenummern (8–15) hängen. Die 1541 Floppy hat Standard-Adresse 8. Drucker typischerweise Adresse 4. Der C64 als Controller sendet ATN um ein Gerät anzusprechen.

SignalLeitungProtokoll-Rolle
ATNAttentionC64 zieht ATN auf Low um Kommando zu senden (Geräteadresse, Sekundäradresse)
CLKClockSender steuert Clock: Low = bereit, High = Bit liegt an
DATADatenEmpfänger bestätigt Empfang; Bit-Übertragung auf DATA

1.2 Serielles Protokoll — Bit für Bit

Ein Byte überträgt der IEC-Bus in 8 einzelnen Bit-Schritten, LSB zuerst. Pro Bit:

  1. Sender zieht CLK auf Low (Bit liegt an)
  2. Sender setzt DATA auf den Bit-Wert
  3. Sender lässt CLK High (Takt-Flanke: Empfänger sampelt DATA)
  4. Empfänger signalisiert Bereitschaft über DATA

Das Timing ist software-gesteuert auf dem C64 — es gibt keinen Hardware-UART. Jede Flanke ist eine einzelne Lese/Schreib-Operation auf $DD00. Das ist der Hauptgrund für die geringe Übertragungsrate.

2. Standard KERNAL-Disk-Zugriff

Der einfachste Weg zu Disk-I/O ist über die KERNAL-Routinen. Die typische Sequenz für einen Datei-Load:

; ─── Robuste KERNAL-Laderoutine ────────────────────────────────
; Lädt eine Datei von Disk ins RAM
; Parameter: filename_ptr/$FB (Low/High), filename_len/$FD,
;            load_addr/$FE/$FF (Zieladresse — ignoriert wenn Sekundäradresse=1)
; Rückgabe: Carry clear = OK, Carry set = Fehler in A

kernal_load_file:
    ; 1. SETLFS: Logische Dateinummer, Gerätenummer, Sekundäradresse
    lda #1               ; Logische Dateinummer (1–255)
    ldx #8               ; Gerätenummer (8 = erste Floppy)
    ldy #0               ; Sekundäradresse 0 = Ladeadresse aus Datei-Header
    jsr $FFBA            ; SETLFS

    ; 2. SETNAM: Dateiname
    lda $FD              ; Dateinamen-Länge
    ldx $FB              ; Namens-Adresse Low
    ldy $FC              ; Namens-Adresse High
    jsr $FFBD            ; SETNAM

    ; 3. LOAD: Datei laden
    lda #0               ; 0=LOAD (1=VERIFY)
    ldx $FE              ; Zieladresse Low (nur wenn SA=1)
    ldy $FF              ; Zieladresse High
    jsr $FFD5            ; LOAD

    ; 4. Fehlerprüfung
    bcs .load_error      ; Carry set = KERNAL-Fehler
    jsr $FFB7            ; READST — Status-Byte lesen
    and #$BF             ; Bit 6 (EOI) ist normal, ignorieren
    beq .load_ok         ; Status 0 = alles gut

.load_error:
    sec                  ; Fehler signalisieren
    rts
.load_ok:
    clc
    rts

; Hilfsmakro: Dateiname als String-Literal übergeben
; Verwendung:
;   lda #<levelname : sta $FB
;   lda #>levelname : sta $FC
;   lda #levelname_end-levelname : sta $FD
;   jsr kernal_load_file
levelname:      !text "LEVEL1"
levelname_end:

2.1 Datei speichern mit SAVE

; SAVE: Speicherbereich auf Disk schreiben
; $FB/$FC = Startadresse, $AE/$AF = Endadresse+1
kernal_save_file:
    lda #1
    ldx #8
    ldy #1               ; SA=1: Zieladresse aus X/Y-Register
    jsr $FFBA            ; SETLFS
    lda #savename_end - savename
    ldx #<savename
    ldy #>savename
    jsr $FFBD            ; SETNAM

    lda #<$C000          ; Zeiger auf Startadresse (als ZP-Adresse übergeben!)
    sta $FB
    lda #>$C000
    sta $FC
    lda $FB              ; SAVE erwartet ZP-Adresse der Startadresse in A
    ldx #<$CFFF          ; Endadresse+1 Low
    ldy #>$CFFF          ; Endadresse+1 High
    jsr $FFD8            ; SAVE
    rts
savename: !text "SAVEGAME"
savename_end:
Achtung: KERNAL-ROM muss eingeblendet sein
KERNAL-Routinen sind nur erreichbar wenn das KERNAL-ROM bei $E000–$FFFF eingeblendet ist ($01 = $37, Standardzustand). Wer das Banking verändert hat, muss vor KERNAL-Calls lda #$37 / sta $01 aufrufen und danach wiederherstellen.

3. DOS-Kommandos

Über den Kommandokanal 15 können DOS-Befehle direkt an das Laufwerk gesendet werden: Dateien löschen, umbenennen, Disk formatieren, Direktzugriff auf Tracks und Sektoren.

; DOS-Kommando senden (Kanal 15 = Kommandokanal)
; cmd_ptr/$FB/$FC = Zeiger auf Kommando-String
; cmd_len/$FD = Länge
send_dos_cmd:
    lda #15              ; Logische Dateinummer 15
    ldx #8               ; Gerätenummer
    ldy #15              ; Sekundäradresse 15 = Kommandokanal
    jsr $FFBA            ; SETLFS
    lda $FD              ; Kommandolänge
    ldx $FB
    ldy $FC
    jsr $FFBD            ; SETNAM (Kommando ist der "Dateiname")
    jsr $FFC0            ; OPEN (sendet Kommando)
    lda #15
    jsr $FFC3            ; CLOSE
    rts

; Datei löschen: "S0:DATEINAME"
dos_delete:
    !text "S0:LEVEL1"
dos_delete_end:

; Datei umbenennen: "R0:NEUNAME=ALTNAME"
dos_rename:
    !text "R0:LEVEL2=LEVEL1"
dos_rename_end:

; Disk formatieren (mit ID): "N0:DISKNAME,ID"  — VORSICHT!
dos_format:
    !text "N0:MY DISK,MD"
dos_format_end:

; Fehlerkanal lesen — nach kritischen Operationen immer prüfen
read_error_channel:
    lda #15
    ldx #8
    ldy #15
    jsr $FFBA
    jsr $FFBD            ; SETNAM mit Länge 0 (kein Name)
    jsr $FFC0            ; OPEN — liest Fehlerkanal
    lda #15
    jsr $FFC6            ; CHKIN — Input auf Kanal 15 umlenken
    ; Jetzt per CHRIN ($FFCF) Fehlerstring lesen bis CR
    ldx #0
.read_loop:
    jsr $FFCF            ; CHRIN — ein Byte lesen
    cmp #$0D             ; CR = Ende des Fehlerstrings
    beq .done
    sta error_buf,x
    inx
    bne .read_loop
.done:
    lda #0
    sta error_buf,x      ; String null-terminieren
    jsr $FFCC            ; CLRCHN
    lda #15
    jsr $FFC3            ; CLOSE
    rts
error_buf: !fill 40, 0

3.1 Direktzugriff auf Track/Sektor

Für fortgeschrittene Anwendungen (eigenes Dateiformat, Direktlesung) kann man einzelne Sektoren per Pufferkommando lesen:

; Sektor direkt lesen via Puffer-Kanal (U1)
; track = Track (1–35), sector = Sektor (0–20 je nach Track)
read_sector:
    ; 1. Pufferkanal öffnen (SA=2)
    lda #2 : ldx #8 : ldy #2
    jsr $FFBA
    lda #0 : ldx #0 : ldy #0
    jsr $FFBD            ; kein Name
    jsr $FFC0            ; OPEN

    ; 2. U1-Kommando: "U1:2 0 18 1" (Kanal, Drive, Track, Sektor)
    lda #u1cmd_end - u1cmd
    ldx #<u1cmd : ldy #>u1cmd
    jsr $FFBD
    ; ... (SETLFS 15/8/15, OPEN, CLOSE wie in send_dos_cmd)

    ; 3. Pufferkanal lesen: 256 Bytes = 1 Sektor
    lda #2
    jsr $FFC6            ; CHKIN — Kanal 2 als Input
    ldx #0
.read256:
    jsr $FFCF            ; CHRIN
    sta sector_buffer,x
    inx
    bne .read256
    jsr $FFCC            ; CLRCHN
    lda #2 : jsr $FFC3   ; CLOSE
    rts

u1cmd: !text "U1:2 0 18 1"    ; Track 18, Sektor 1 (Directory)
u1cmd_end:
sector_buffer: !fill 256, 0

4. Das D64-Dateiformat

Eine Standard-1541-Diskette hat 35 Tracks mit je 17–21 Sektoren, abhängig von der Zone (Track-Gruppe). Jeder Sektor enthält genau 256 Bytes. Das ergibt 664 Sektoren = 170.496 Bytes Gesamtkapazität (von denen 664 = 166 KB nutzbar sind, nach Overhead).

4.1 Track/Sektor-Struktur

4.2 Spezielle Tracks

TrackInhalt
18, Sektor 0BAM (Block Availability Map): Belegungsbitmap aller Sektoren + Disk-Name + ID
18, Sektor 1+Directory: Dateiverzeichnis, 8 Einträge à 32 Bytes pro Sektor
Alle anderenDateidaten — als verkettete Liste von Sektoren

4.3 Sektor-Verkettung

Jeder Daten-Sektor beginnt mit zwei "Link"-Bytes:

  • Byte 0: Track des nächsten Sektors (0 = letzter Sektor)
  • Byte 1: Sektor des nächsten Sektors (oder Anzahl gültiger Bytes wenn letzter)
  • Bytes 2–255: Nutzdaten (254 Bytes)

Ein File lässt sich also als verkettete Liste lesen: BAM → Directory-Eintrag → erster Sektor → nächster Sektor → … → letzter Sektor (Track-Link = 0).

4.4 D64-Datei-Layout auf dem PC

Eine .d64-Datei ist ein lineares Image: Track 1 Sektor 0, Track 1 Sektor 1, …, Track 1 Sektor 20, Track 2 Sektor 0, … — in der Reihenfolge der Zones. Gesamtgröße: 174.848 Bytes (35 Tracks, Standard ohne Error-Map).

; D64-Offset-Berechnung für Track/Sektor (auf dem PC/Host)
; track: 1–35, sector: 0–20
; Offset = (Summe aller Sektoren in Tracks 1..track-1) * 256 + sector * 256
track_offsets:
    !word 0        ; Track 1 (Zone 1: 21 Sektoren)
    !word 5376     ; Track 2
    ; ... bis Track 35
    ; Track 18 (Directory) beginnt bei Offset 107520 ($1A400)

5. Warum der Standard-Load so langsam ist

Ein einfaches KERNAL-LOAD dauert oft 5–10 Sekunden für 10 KB. Die Ursache ist eine Kombination aus Protokoll-Overhead und einem Timing-Bug im Original-KERNAL:

5.1 Das Bit-Rauschen-Problem (1984)

Das serielle Protokoll des IEC-Bus wurde für den VC-20 entwickelt, der eine Hardware-Serielle hatte. Beim C64 fehlt diese Hardware — der Bus wird vollständig in Software emuliert. Der ursprüngliche C64-KERNAL hatte zudem einen Timing-Bug: Aus Kompatibilitätsgründen mit dem CBM-8032-Bus-Timing wurde das seriell implementierte Protokoll auf ~300 Bit/s effektiv gedrosselt, obwohl ~3.000 Bit/s möglich wären.

Das ergibt sich aus:

  • 1 Byte = 8 Bits, davon jedes Bit ca. 70–100 µs Übertragungszeit
  • Plus Handshake-Zyklen zwischen jedem Byte
  • Plus 1541-interne Verarbeitung (~2 ms pro Sektor-Header-Read)
  • Effektivrate: ~350–400 Bytes/Sekunde

5.2 Warum Software-Fastloader möglich ist

Die 1541 hat eine eigene 6502-CPU mit 1 MHz. Normaler KERNAL-Code läuft nur auf dem C64 — das Laufwerk führt eigene ROM-Firmware aus. Ein Software-Fastloader überträgt via M-W (Memory Write) neuen Drive-Code ins Laufwerk-RAM und lässt diesen dort ausführen. Der neue Code kann das Protokoll auf 8 Bit parallel (wenn Parallelkabel vorhanden) oder zeitoptimiertes 2-Bit seriell umschalten.

6. Fastloader-Konzept

6.1 Zwei Ansätze

Ansatz 1 — Protokoll-Optimierung (Software-only):
Neuer Drive-Code sendet Bytes mit präzisem Timing über die existierenden IEC-Leitungen. Kein Hardware-Eingriff nötig. Die 1541 sendet 2 Bits gleichzeitig (CLK + DATA) statt 1 — doppelte Datenrate. Mit optimalem Timing 8–10× so schnell wie Standard.

Ansatz 2 — Parallel-Übertragung (Hardware):
Zusätzliches Kabel zwischen 1541 User-Port und C64 User-Port überträgt 8 Bits simultan. Das Laufwerk hält 8 Datenbits auf eigene Ausgänge, ein Handshake-Signal löst das Lesen aus. Theoretisches Maximum: ~16 KB/s. Benötigt Hardware-Modifikation an Laufwerk und Kabel.

6.2 Drive-Code hochladen mit M-W

Der DOS-Kommandokanal nimmt nicht nur Disk-Kommandos entgegen, sondern auch direkte Speicherzugriffe über M-W (Memory Write) und M-E (Memory Execute):

; Drive-Code in die 1541 hochladen
; Der Code landet im 1541-RAM ab Adresse $0300 (freier RAM-Bereich)
; M-W Format: "M-W" + Low + High + Länge + Daten

upload_drive_code:
    ; Kommandokanal öffnen
    lda #15 : ldx #8 : ldy #15
    jsr $FFBA
    lda #0 : ldx #0 : ldy #0
    jsr $FFBD
    jsr $FFC0            ; OPEN

    lda #15
    jsr $FFC9            ; CHKOUT — Kanal 15 als Output

    ; M-W-Header senden: "M-W" + Adresse Low/High + Länge
    lda #'M' : jsr $FFD2
    lda #'-' : jsr $FFD2
    lda #'W' : jsr $FFD2
    lda #<$0300           ; 1541-Zieladresse Low
    jsr $FFD2
    lda #>$0300           ; 1541-Zieladresse High
    jsr $FFD2
    lda #drive_code_end - drive_code
    jsr $FFD2            ; Länge (max. 32 Bytes pro M-W-Kommando!)

    ; Drive-Code-Bytes senden
    ldx #0
.send_loop:
    lda drive_code,x
    jsr $FFD2            ; CHROUT
    inx
    cpx #drive_code_end - drive_code
    bcc .send_loop

    jsr $FFCC            ; CLRCHN
    lda #15 : jsr $FFC3  ; CLOSE
    rts

; M-E: Drive-Code ausführen
; "M-E" + Adresse Low/High (ohne Längenbyte!)
execute_drive_code:
    ; Gleiches Schema: SETLFS/OPEN/CHKOUT, dann senden:
    ; 'M', '-', 'E', <$0300, >$0300, CLRCHN, CLOSE
    rts

; Hinweis: M-W überträgt max. 32 Bytes pro Kommando.
; Für größere Drive-Programme: mehrere M-W mit ansteigender Zieladresse.

drive_code:
    ; Hier liegt der 1541-6502-Assembler-Code
    ; (auf dem C64 als Datenbytes abgelegt — nicht direkt ausführbar)
    !byte $EA, $EA, $60  ; Beispiel: 2× NOP + RTS
drive_code_end:
M-W: Nur für 1541/1571
M-W und M-E sind 1541-spezifisch. SD2IEC (SD-Karten-Laufwerk) und viele andere Emulations-Drives ignorieren sie oder verhalten sich anders. Eigenen Drive-Code immer auf echter Hardware oder in VICE mit echter 1541-Emulation testen.

7. Fertige Fastloader integrieren

In der Praxis schreibt man keinen eigenen Fastloader von Grund auf — stattdessen integriert man eine bewährte Lösung. Die bekannteste für C64-Homebrew und Demo-Szene:

7.1 Krill's Loader

Krill's Loader ist ein Software-Fastloader ohne Hardware-Anforderungen — funktioniert auf jeder 1541/1571/1581 und auf SD2IEC mit entsprechender Firmware. Eigenschaften:

  • ~8–10× schneller als KERNAL (ca. 3.500 B/s)
  • Kein User-Port-Kabel nötig
  • Kompatibel mit 1541, 1571, 1581, SD2IEC
  • Optionaler Raster-IRQ-freundlicher Betrieb (läuft teilweise im Hintergrund)
  • Quellcode frei verfügbar (ACME-Assembler)
  • Aktiv gewartet: unterstützt PAL und NTSC automatisch

7.2 Krill's Loader integrieren

; ─── Krill's Loader Integration (vereinfacht) ──────────────────
; Voraussetzung: Krill's Loader Quellcode im Projekt-Verzeichnis
; Source: https://csdb.dk/release/?id=<krill-loader-id>
;
; 1. Loader-Code einbinden
!source "krills_loader/loader.asm"

; und einige Zero-Page-Adressen. Genau im loader.asm prüfen!

; 2. Loader initialisieren (einmalig beim Programmstart)
init_loader:
    jsr LOADER_INSTALL    ; Drive-Code hochladen + Loader aktivieren
    ; Gibt zurück wenn Drive bereit ist
    ; Fehler: Carry set wenn kein Laufwerk gefunden
    bcs .no_drive

.no_drive:
    ; Fallback: KERNAL-Load verwenden

; 3. Datei laden
; Parameter: A = Dateinummer (aus Datei-Index), X/Y = Zieladresse
load_with_krill:

    lda #1               ; Datei-Index (1-basiert, aus Disk-Directory-Reihenfolge)

    ; X/Y enthalten Endadresse+1 nach dem Laden

; 4. Fortschritts-Callback (optional)
; Krill's Loader ruft eine Callback-Routine während des Ladens auf.
; Ideal für Lade-Bildschirm / Fortschrittsbalken.
krill_progress_cb:
    ; A = Anzahl geladener Blöcke (0 bis max)
    ; Routine muss alle Register erhalten (kein Clobber)
    pha
    jsr update_progress_bar
    pla
    rts

8. Eigenen Fastloader verstehen und schreiben

Um einen eigenen Fastloader zu verstehen, bauen wir einen minimierten 2-Bit-Software-Fastloader. Dieser überträgt 2 Bits gleichzeitig (ein Bit auf CLK, ein Bit auf DATA), was die Übertragungsrate theoretisch verdoppelt.

8.1 Protokoll-Design

C64 empfängt 1 Byte (8 Bits, 4 Schritte à 2 Bits):

1541: CLK=H DATA=H → Bereit-Signal
C64: liest ATN=L → bestätigt Empfangsbereitschaft

1541: setzt CLK=Bit1 DATA=Bit0 → Bits 0+1 gleichzeitig
C64: liest $DD00, maskiert CLK+DATA → 2 Bits extrahiert

1541: setzt CLK=Bit3 DATA=Bit2 → Bits 2+3
C64: liest $DD00 → 2 Bits extrahiert

... (4 Schritte für 8 Bits) ...

1541: CLK=H DATA=H → nächstes Byte bereit

8.2 C64-Empfänger-Code (vereinfacht)

; Mini-Fastloader: 2-Bit-Empfang von 1541
; Dieses Beispiel zeigt das Prinzip — echter Fastloader benötigt
; genaues Timing, PAL/NTSC-Kompensation und Fehlerbehandlung

; CIA2-Bit-Masken
IEC_CLK_IN  = %01000000   ; $DD00 Bit 6: CLK von Laufwerk
IEC_DAT_IN  = %10000000   ; $DD00 Bit 7: DATA von Laufwerk
IEC_CLK_OUT = %00010000   ; $DD00 Bit 4: CLK an Laufwerk
IEC_ATN     = %00001000   ; $DD00 Bit 3: ATN

; Ein Byte empfangen (stark vereinfacht)
receive_byte:
    sei                  ; IRQ sperren — Timing kritisch!

    ; Warten bis 1541 Bereit-Signal sendet (CLK + DATA = High)
.wait_ready:
    lda $DD00
    and #(IEC_CLK_IN | IEC_DAT_IN)
    cmp #(IEC_CLK_IN | IEC_DAT_IN)
    bne .wait_ready

    lda #0
    sta $03              ; Akkumulator für Byte-Aufbau

    ; 4 Mal: 2 Bits gleichzeitig lesen
    ldx #4
.receive_pair:
    ; Warten bis CLK auf Low geht (Bit liegt an)
.wait_clk_low:
    bit $DD00
    bvs .wait_clk_low   ; Bit 6 (CLK IN) = 1 → noch High

    ; DATA-Bit lesen (Bit 7)
    lda $DD00
    and #IEC_DAT_IN
    clc
    ror                  ; DATA-Bit in Carry
    ror $03              ; in Empfangsbyte schieben
    ; CLK-Bit (Bit 6) lesen und schieben
    lda $DD00
    and #IEC_CLK_IN
    clc
    ror
    ror
    ror $03

    ; Warten bis CLK wieder High (Bit-Ende)
.wait_clk_high:
    bit $DD00
    bvc .wait_clk_high   ; Bit 6 = 0 → noch Low

    dex
    bne .receive_pair

    cli
    lda $03              ; Empfangenes Byte in A
    rts

; Wichtig: Der Drive-Code (6502-Code FÜR die 1541) muss separat
; in ACME für CPU-Adressraum der 1541 assembliert und dann
; als Datenbytes im C64-Code eingebettet werden.
Drive-Code — Ein separates Programm
Der Drive-Code ist echtes 6502-Assembler-Programm, das auf der CPU der 1541 läuft. Es greift auf Register der 1541 zu (z.B. $1800 für VIA-Port des User-Ports, $1C0C für GCR-Track-Puffer). Ein vollständiger Fastloader besteht aus zwei Assemblier-Schritten: C64-Code und 1541-Code — letzterer wird als Byte-Array in den C64-Code eingebettet.

9. Praxis: Mehrteiliges Spiel mit Fastloader

Ein typisches Mehrteil-Spiel auf C64-Disk hat folgende Lade-Strategie:

  1. LOADER (erstes File, startet automatisch): Installiert Krill's Loader, zeigt Lade-Screen
  2. MAIN: Spielkern, Grafik-Routinen, Sprite-Daten, Musik-Player
  3. LEVEL1–8: Tilemap + Feind-Daten pro Level (100–500 Bytes je)
  4. MUSIC: SID-Musikdaten (1–5 KB)

9.1 Loader-Einstieg mit Fortschrittsanzeige

; ─── loader.asm — Wird zuerst von Disk gestartet ───────────────

*= $0801                 ; BASIC-Startadresse
!byte $0B,$08,$01,$00,$9E,$32,$30,$36,$31,$00,$00,$00
                         ; BASIC-Stub: 10 SYS 2061

*= $080D
    jsr init_screen       ; Schwarzer Hintergrund, Lade-Screen aufbauen
    jsr LOADER_INSTALL    ; Krill's Loader einrichten
    bcs .fallback_kernal

    ; MAIN laden (Datei-Index 2 auf Disk)
    ldx #<$1000
    ldy #>$1000
    lda #2               ; MAIN.PRG
    jsr LOADER_LOAD
    bcs .load_error

    ; MUSIC laden
    ldx #<$E000
    ldy #>$E000
    lda #6               ; MUSIC.PRG
    jsr LOADER_LOAD
    bcs .load_error

    ; Kontrolle an MAIN übergeben
    jmp $1000

.fallback_kernal:
    ; KERNAL-Fallback wenn Krill's fehlschlägt
    jsr kernal_load_main
    jmp $1000

.load_error:
    ; Fehlermeldung anzeigen (z.B. "LOAD ERROR")
    jmp *                ; Einfrieren

; Fortschrittsbalken-Update (Krill's Callback)
update_progress_bar:
    ; A = geladene Blöcke (jeder Block = 254 Bytes)
    ; Balken in Bildschirmzeile 12: $04C0 + Offset
    pha
    cmp #38              ; Max 38 Zeichen breit
    bcs .full
    tax
    lda #$A0             ; Gefülltes Quadrat (Bildschirmcode)
.bar_loop:
    sta $04C0,x          ; Bildschirmspeicher Zeile 12
    dex
    bpl .bar_loop
.full:
    pla
    rts

9.2 Level-Streaming während des Spiels

; Level laden während das Spiel läuft (zwischen Spieler-Tod und Respawn)
; Krill's Loader ist noch aktiv — erneuter LOADER_INSTALL nicht nötig!

current_level = $C0      ; 0-basierter Level-Index

load_next_level:
    ; Level-Daten-Index auf Disk: Levels starten bei Datei-Index 3
    lda current_level
    clc
    adc #3               ; Krill-Index (3=Level0, 4=Level1, ...)
    ldx #<level_data_ptr
    ldy #>level_data_ptr
    jsr LOADER_LOAD

    ; Zeige kurze "LEVEL N STARTING"-Meldung während Laden
    ; (Krill ruft progress_cb auf, kann dort auch Text-Update machen)
    rts

level_data_ptr = $5000   ; Festes RAM-Ziel für Level-Daten

Übungsaufgaben

Übung 17.1 — KERNAL-Load mit Fehlerbehandlung

Schreibe eine robuste load_file-Subroutine die: (1) eine Datei per KERNAL lädt, (2) READST nach dem Laden prüft, (3) den DOS-Fehlerkanal (Kanal 15) ausliest und als Zeichenkette im RAM speichert, (4) bei Fehler eine Fehlermeldung im Bildschirmspeicher ausgibt. Teste mit einer existierenden Datei, einer nicht-existierenden Datei, und einer Disk ohne Laufwerk (Gerätenummer 9).

Übung 17.2 — Disk-Directory einlesen

Öffne das Disk-Directory ("$") als Datei (SETLFS/SETNAM/OPEN/CHKIN), lese alle Einträge und zeige Dateiname und Blockgröße im Bildschirmspeicher an. Stoppe das Lesen beim ersten EOI oder Fehler. Formatiere die Ausgabe wie das BASIC-LOAD"$" Directory-Listing. Bonus: Sortiere die Einträge alphabetisch im RAM, dann anzeigen.

Übung 17.3 — Krill's Loader mit Lade-Screen

Erstelle ein Mini-Projekt mit zwei Dateien auf einer D64: (1) ein Loader-PRG der Krill's Loader installiert und eine zweite Datei lädt, (2) das Hauptprogramm mit einem einfachen Effekt. Implementiere eine Fortschrittsanzeige: Während des Ladens füllt sich ein Balken in Bildschirmzeile 24. Teste in VICE mit echter 1541-Emulation (True Drive Emulation aktiviert) um die tatsächliche Ladezeit zu sehen.

Übung 17.4 — D64-Analyse auf dem PC

Schreibe ein Python-Skript das eine .d64-Datei öffnet und ausgibt: (1) Disk-Name und ID aus dem BAM (Track 18, Sektor 0), (2) alle Directory-Einträge (Dateiname, Typ, Blockgröße), (3) welche Tracks/Sektoren belegt sind (aus der BAM-Bitmap). Teste mit der D64 aus deinem C64-Mastery-Projekt (beispiele/-Verzeichnis).
