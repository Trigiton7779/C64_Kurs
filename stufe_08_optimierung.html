1. Cycle Counting — Jeder Takt zählt

Der 6510-Prozessor im C64 arbeitet mit exakt 985.248 Hz (PAL) bzw. 1.022.727 Hz (NTSC). Ein Taktzyklus dauert damit ca. 1,015 Mikrosekunden. Der VIC-II zeichnet 312 Rasterzeilen pro Frame (PAL), wobei 63 Taktzyklen pro Rasterzeile zur Verfügung stehen:

; PAL-Timing:
; 985248 Hz / 50 Frames = 19704,96 Zyklen pro Frame
; 19705 Zyklen / 312 Zeilen = 63,15 ≈ 63 Zyklen pro Zeile
; Sichtbarer Bereich: 200 Zeilen × 63 Zyklen = 12600 Zyklen
; VBlank: 112 Zeilen × 63 Zyklen = 7056 Zyklen

; NTSC-Timing:
; 1022727 Hz / 60 Frames = 17045,45 Zyklen pro Frame
; 17045 Zyklen / 263 Zeilen = 64,8 ≈ 65 Zyklen pro Zeile
Warum Cycle Counting?
Für Rastereffekte, Musik-Player und zeitkritischen Code musst du genau wissen, wie viele Zyklen jede Routine verbraucht. Ein Raster-Handler muss garantiert innerhalb von 63 Zyklen (PAL) fertig sein — sonst „verrutscht" der Effekt.

1.1 Zyklen pro Opcode

Jeder 6510-Befehl verbraucht eine bestimmte Anzahl Taktzyklen. Zusätzlich kosten Page-Crossing (wenn Adressberechnung eine 256-Byte-Grenze überschreitet) einen Extrazyklus:

1.2 Cycle Counting in der Praxis

Um eine Routine zu analysieren, zählt man jeden Befehl durch. Beispiel: eine einfache Kopierschleife:

; Naive Kopierschleife — Analyse
;                    Zyklen   Gesamt (×256)
          LDX #$00   ; 2      512
CopyLoop:
          LDA $C000,X ; 4      1024
          STA $D000,X ; 5      1280
          INX         ; 2      512
          BNE CopyLoop ; 3/2   768/512 (letzter Durchlauf: 2)
; Total (Schleife): 14 Zyklen × 256 = 3584 Zyklen für 256 Bytes
; = ca. 57 Rasterzeilen!

; Optimiert mit unrolled loop (×8) — siehe Abschnitt 3
Page-Crossing-Falle
LDA $C0FF,X mit X=1 greift auf $C100 zu — das ist eine andere Seite! Das kostet einen Extrazyklus. Beim Durchlaufen von Tabellen, die eine 256-Byte-Grenze überschreiten, entstehen so unsichtbare Zeitfresser. Tabellen immer page-aligned anlegen (mit !align $100 in ACME).

2. Zero Page Strategie — Das goldene Gebiet

Die Zero Page ($0000–$00FF) ist das wertvollste RAM des C64. Zero-Page-Befehle benötigen nur 1 Byte Adresse statt 2, sind um 1 Zyklus schneller als absolute Adressen, und ermöglichen die indirekten Adressierungsarten (zp,X) und (zp),Y.

2.1 Belegte Zero-Page-Adressen

$10–$1FTeils KERNAL-VariablenNur wenn KERNAL abgeschaltet
$20–$7FTeils BASIC/KERNALNur wenn abgeschaltet

Empfohlene ZP-Strategie
Schalte BASIC (und ggf. KERNAL) aus: LDA #$35 / STA $01 (nur RAM, KERNAL an I/O). Dann stehen dir $0002–$00FF fast vollständig zur Verfügung. Lege alle kritischen Variablen, Pointer und Zähler dort ab.

2.3 Geschwindigkeitsvergleich

; Variable in absolutem RAM: $1000
          LDA $1000     ; 4 Zyklen, 3 Bytes
          STA $1001     ; 4 Zyklen, 3 Bytes
; Summe: 8 Zyklen, 6 Bytes

; Gleiche Variable in Zero Page: $02
          LDA $02       ; 3 Zyklen, 2 Bytes
          STA $03       ; 3 Zyklen, 2 Bytes
; Summe: 6 Zyklen, 4 Bytes → 25% schneller, 33% kleiner!

; Bei 1000 Zugriffen pro Frame:
; Absolut: 8000 Zyklen
; Zero Page: 6000 Zyklen → 2000 Zyklen gespart = ~32 Rasterzeilen!

3. Loop Unrolling — Schleifen auffalten

Eine Schleife hat Overhead: INX (2), BNE (3 genommen / 2 nicht genommen), zusammen 4–5 Zyklen Overhead pro Iteration. Bei engen Schleifen kann das 30–50 % des Gesamtaufwands ausmachen. Loop Unrolling eliminiert diesen Overhead durch manuelles Wiederholen des Schleifenkörpers.

3.1 Einfaches Unrolling

; Original: 256 Bytes kopieren
; Zyklen pro Iteration: LDA(4) + STA(5) + INX(2) + BNE(3) = 14
; Gesamt: 14 × 256 = 3584 Zyklen

CopyNaive:
          LDX #$00
.loop:    LDA $C000,X
          STA $D000,X
          INX
          BNE .loop
          RTS

; Unrolled: ×8 — 8 Kopieroperationen ohne Schleifentest
; Overhead: nur noch DEX + BNE alle 8 Bytes = 5/32 Iterations
; Zyklen pro Gruppe: (4+5)×8 + 2 + 3 = 77 statt 14×8=112
; Gesamt: 77 × 32 = 2464 Zyklen → 31% schneller!

CopyUnrolled:
          LDX #32      ; 32 Gruppen à 8 Bytes
          DEX          ; X = 31 (letzter Index für 0-basiert)
.loop:
          LDA $C000+(7*1),X ; Trick: Wir kopieren rückwärts
; Einfacher: Vorwärts mit Offset-Tabelle

; Praktischste Form: Vollständig unrolled für kleine Größen
Copy32:
!for i, 0, 31 {
          LDA $C000 + i
          STA $D000 + i
}
          RTS
; 32 × (4+4) = 256 Zyklen für 32 Bytes! (ohne STA absolut,X)
; Aber: 32 × 6 Bytes = 192 Bytes Code-Größe
Kompromiss: Speed vs. Größe
Vollständiges Unrolling ist das Schnellste, aber kostet viel Speicher. Für den C64 mit 64 KB RAM ist das ein echter Kompromiss. Unrolling ×4 oder ×8 ist oft ein guter Mittelweg: deutlicher Speed-Gewinn bei moderaten Kosten.

3.2 Duff's Device für 6510

Duff's Device (bekannt aus C) kombiniert einen switch-Jump mit einem unrolled Loop. In 6510 Assembly setzen wir das mit einem Dispatch-Sprung auf die richtige Position im unrolled Code um:

; Kopiere N Bytes ($FB = Zähler), Quelle $FC/$FD, Ziel $FE/$FF
; Technik: Spring direkt in unrolled Loop basierend auf (N mod 8)

CopyN:
          LDA ByteCount     ; Anzahl Bytes
          BEQ .done         ; 0 Bytes: fertig

          ; Berechne Start-Position im unrolled Loop
          SEC
          SBC #1
          LSR               ; /2
          LSR               ; /4
          LSR               ; /8 → Anzahl voller 8er-Gruppen
          STA GroupCount

          LDA ByteCount
          AND #$07          ; Reste 0-7
          ASL               ; × 2 (Bytes pro LDA+STA Paar = 6, aber vereinfacht)
          TAX
          JMP (JumpTable,X) ; Spring zur richtigen Position

JumpTable:
          !word Copy0, Copy1, Copy2, Copy3
          !word Copy4, Copy5, Copy6, Copy7

Copy7:    LDA (SrcPtr),Y : STA (DstPtr),Y : INY
Copy6:    LDA (SrcPtr),Y : STA (DstPtr),Y : INY
Copy5:    LDA (SrcPtr),Y : STA (DstPtr),Y : INY
Copy4:    LDA (SrcPtr),Y : STA (DstPtr),Y : INY
Copy3:    LDA (SrcPtr),Y : STA (DstPtr),Y : INY
Copy2:    LDA (SrcPtr),Y : STA (DstPtr),Y : INY
Copy1:    LDA (SrcPtr),Y : STA (DstPtr),Y : INY
Copy0:    ; Hier landen wir für Rest=0 (nach vollem 8er-Block)
          ; GroupCount verbleibende volle 8er-Gruppen weiter kopieren...
.done:    RTS

3.3 Schleife mit DEX statt INX+CMP

; Falsch (ineffizient):
          LDX #0
.loop:    LDA Data,X
          ; ... verarbeiten ...
          INX
          CPX #128   ; 2 Zyklen extra pro Iteration!
          BNE .loop

; Richtig — rückwärts zählen, BNE als Zero-Test:
          LDX #128
.loop:    LDA Data-1,X   ; Offset -1 wegen DEX vorher
          ; ... verarbeiten ...
          DEX            ; 2 Zyklen
          BNE .loop      ; 3/2 Zyklen — setzt Z-Flag direkt!
; Erspart CPX #N (2 Zyklen) pro Iteration → bei 128 Iterationen: 256 Zyklen gespart

4. Lookup-Tabellen — Mathematik durch Nachschlagen

Multiplikation, Sinus, Kosinus und Quadratwurzeln sind auf dem 6510 extrem teuer zu berechnen. Lookup-Tabellen (LUT) ersetzen Berechnungen durch einfache Array-Zugriffe und sind die wichtigste Optimierungstechnik für Grafikroutinen.

4.1 Sinustabelle

; 256-Einträge Sinustabelle (0-255, skaliert auf 0-127)
; Winkel: 0-255 = 0°-360° (1/256 Kreis Auflösung)
; Werte: 0-255, wobei 128 = Mitte (sin≈0), 0 = min, 255 = max

SinTable:

4.2 Multiplikations-LUT (a²/4 Methode)

Der Trick: a × b = ((a+b)² - (a-b)²) / 4. Damit kann man jede Multiplikation auf Quadrat-Tabellen-Lookups reduzieren:

; Quadrat/4 Tabelle: sq4[i] = (i*i)/4 für i = 0..511
; Benötigt 512 Bytes, spart aber VIEL Rechenzeit

; Multiplikation: A × B → Ergebnis in $10/$11 (16-Bit)
; A in Akku, B in X-Register bei Aufruf

Multiply:
          STX TmpX          ; B sichern
          STA TmpA          ; A sichern

          ; Berechne (A+B)/4 → Sq4Lo/Hi
          CLC
          ADC TmpX          ; A+B
          TAX
          LDA Sq4Lo,X
          STA $10
          LDA Sq4Hi,X
          STA $11

          ; Berechne (A-B)/4 → subtrahieren
          LDA TmpA
          SEC
          SBC TmpX          ; A-B (kann negativ sein → 256+Ergebnis nutzen)
          TAX
          LDA $10
          SEC
          SBC Sq4Lo,X
          STA $10
          LDA $11
          SBC Sq4Hi,X
          STA $11
          RTS

; sq4-Tabelle: 512 Einträge à 2 Bytes (Lo/Hi getrennt)
; sq4[i] = (i*i)/4
Sq4Lo:
!for i, 0, 255 { !byte <((i*i)/4) }
!for i, 256, 511 { !byte <((i*i)/4) }

Sq4Hi:
!for i, 0, 255 { !byte >((i*i)/4) }
!for i, 256, 511 { !byte >((i*i)/4) }

4.3 Row-Offset-Tabellen für Bitmap

; Für schnellen Bitmap-Zugriff: Zeilen-Offset vorberechnen
; Y-Pos → Adresse in der Bitmap
; Bitmap bei $2000, 320 Bytes pro 8-Pixel-Block-Zeile
; Für Y: Zeilen-Block = Y/8, Offset = Block*320 + (Y AND 7)

RowLoTable: ; Low-Byte der Zeilen-Startadresse
!for row, 0, 199 {
    !byte <($2000 + (row/8)*320 + (row & 7))
}

RowHiTable: ; High-Byte der Zeilen-Startadresse
!for row, 0, 199 {
    !byte >($2000 + (row/8)*320 + (row & 7))
}

; Verwendung: Pixel bei (X, Y) setzen
SetPixelFast:
          LDY PixelY        ; Y-Koordinate
          LDA RowLoTable,Y
          STA DstPtr
          LDA RowHiTable,Y
          STA DstPtr+1      ; Zeilen-Startadresse → DstPtr

          LDA PixelX
          LSR               ; /8 → Byte-Spalte (Quotient)
          LSR
          LSR
          TAY               ; Y = Byte-Offset in der Zeile

          LDA PixelX
          AND #$07          ; Bit-Position in Byte
          TAX
          LDA BitMask,X     ; Vorberechnete Bitmasken
          ORA (DstPtr),Y
          STA (DstPtr),Y
          RTS

BitMask: !byte $80,$40,$20,$10,$08,$04,$02,$01

5. Selbstmodifizierender Code

Selbstmodifizierender Code (Self-Modifying Code, SMC) ändert zur Laufzeit die eigenen Maschinenbefehle. Auf dem 6510 ist dies eine der mächtigsten Optimierungstechniken: Man schreibt direkt in den Operanden eines Befehls, um Adressberechnung zu eliminieren.

Nur im RAM!
Selbstmodifizierender Code funktioniert nur, wenn der Code im RAM liegt — nicht im ROM. Der C64-Code muss also im Bereich $0800–$CFFF oder nach $D800 (nach dem I/O-Bereich) im RAM liegen.

5.1 Adress-Patching

; Statt Index-Register für Quelladresse zu verwenden:
; Direkte Modifikation des LDA-Operanden

CopyWithSMC:
          LDA #<Source      ; Low-Byte der Quelladresse
          STA SMC_Src+1     ; Patch: Operand des LDA-Befehls unten
          LDA #>Source
          STA SMC_Src+2     ; High-Byte des LDA-Operanden

          LDA #<Dest
          STA SMC_Dst+1
          LDA #>Dest
          STA SMC_Dst+2

          LDX #$00
.loop:
SMC_Src:  LDA $0000,X       ; ← Operand wird zur Laufzeit gepatcht!
SMC_Dst:  STA $0000,X       ; ← Dieser auch
          INX
          BNE .loop
          RTS

; Vorteil: Keine Zero-Page-Pointer nötig → spart (zp),Y Overhead
; LDA abs,X = 4+1 Zyklen (aber kein 2-Zyklus-Indirekt-Overhead)
; LDA (zp),Y = 5+1 Zyklen

5.2 SMC für Konstanten-Substitution

; Anpassbare Zeichensatz-Ausgabe ohne Parameterübergabe

DrawChar:
; Patch: Zeichennummer direkt im LDA-Immediate eintragen
SMC_Char: LDA #$41           ; Byte 1 = LDA-Opcode ($A9), Byte 2 = Wert
          ASL                ; × 8 (8 Bytes pro Zeichen im Charset)
          ASL
          ASL
          ; ... Zeichendaten ausgeben
          RTS

; Aufrufer patcht die Zeichennummer:
          LDA CharCode
          STA SMC_Char+1     ; Byte nach dem Opcode = Immediate-Wert
          JSR DrawChar

5.3 Unrolled SMC-Kopierer (Maximum Speed)

; Ultra-schneller Kopier-Block: komplett unrolled, SMC-Adressen
; 256 Bytes in 512 Zyklen (2 Zyklen pro Byte → theoretisches Minimum)

FastCopy256:
; Patch Source und Dest Adressen
          LDA SrcAddrLo : STA FC_S0+1 : STA FC_S1+1 ; ... etc. für alle
          LDA SrcAddrHi : STA FC_S0+2 : STA FC_S1+2

; Unrolled: 256 LDA abs + STA abs Paare
FC_S0:    LDA $0000 : STA $0000   ; Beide Operanden werden gepatcht
FC_S1:    LDA $0001 : STA $0001   ; +1 addiert ACME im !for-Block
; ... (255 weitere Paare — typisch via ACME !for-Makro)
!for i, 2, 255 {
    LDA SrcBase + i
    STA DstBase + i
}
          RTS
; Ergebnis: 256 × (4+4) = 2048 Zyklen
; Verglichen mit Schleifen-Variante: 3584 → 43% schneller!

6.1 LAX — Lade A und X gleichzeitig

; Ohne LAX: Pointer laden für (zp),Y Zugriff
          LDA PointerLo    ; 3 Zyklen
          STA $FB          ; 3 Zyklen
          LDA PointerHi    ; 3 Zyklen
          STA $FC          ; 3 Zyklen
          LDX PointerLo    ; 3 Zyklen → X für anderes
; Total: 15 Zyklen

; Mit LAX: A und X gleichzeitig laden
          LAX PointerLo    ; 3 Zyklen → A=X=PointerLo
          STA $FB          ; 3 Zyklen
          LDA PointerHi    ; 3 Zyklen
          STA $FC          ; 3 Zyklen
; Total: 12 Zyklen → 20% schneller, X hat PointerLo als Bonus

; Praktisches Beispiel: Sprite-Daten kopieren mit LAX
CopySpriteData:
          LDY #62
.loop:
          LAX SpriteData,Y   ; A = X = SpriteData[Y]
          STA $0340,Y        ; In Sprite-Block schreiben (Absolut)
          ; X hat denselben Wert für anderweitige Nutzung
          DEY
          BPL .loop
          RTS

6.2 DCP — Decrement und Compare

; Standard: Zähler dekrementieren und prüfen
          DEC Counter      ; 5 Zyklen (Absolut)
          LDA Counter      ; 4 Zyklen
          CMP #5           ; 2 Zyklen
          BEQ Match        ; 3/2 Zyklen
; Total: 14 Zyklen

; Mit DCP (Zero Page!)
          DCP CounterZP    ; 5 Zyklen: DEC ZP + CMP A → Flags gesetzt!
          BEQ Match        ; 3/2 Zyklen
; Total: 8 Zyklen → 43% schneller!
; Achtung: DCP vergleicht mit dem Akkumulator-Wert VOR dem DEC

6.3 SLO — Shift Left und OR

; Bit-Manipulation: Byte shiften und in Akku mergen
; Standard:
          ASL $FB          ; 5 Zyklen
          LDA Akku_save    ; 3 Zyklen
          ORA $FB          ; 3 Zyklen
; Total: 11 Zyklen

; Mit SLO:
          SLO $FB          ; 5 Zyklen: ASL $FB + ORA $FB → Akku
; Total: 5 Zyklen → Hälfte! Useful für Pixel-Bit-Packing
Undokumentierte Opcodes und Emulator-Kompatibilität
Auf echter Hardware (6510-Chips) sind die oben genannten stabilen Opcodes zuverlässig. Im VICE-Emulator werden sie unterstützt. Jedoch: Auf CMOS-Varianten (65C02, 65C816) und einigen späten NMOS-Chips können sie sich anders verhalten. Für maximale Kompatibilität sparsam einsetzen.

7. DMA-Delays, Bad Lines und Sprite-Fetch

Der VIC-II „stiehlt" dem 6510 Taktzyklen, um Daten aus dem RAM zu lesen. Das passiert automatisch und muss im Cycle Counting berücksichtigt werden:

7.1 Bad Lines — Die teuersten Zeilen

Eine Bad Line tritt auf, wenn der VIC-II den Zeichensatz-Daten lesen muss. Das passiert auf jeder Rasterzeile, bei der YSCROLL (Bit 0–2 von $D011) == (Rasterzeile AND 7) gilt — also alle 8 Zeilen.

; Auf Bad Lines stiehlt der VIC-II 40 zusätzliche Zyklen!
; Normale Zeile:   63 Zyklen für CPU verfügbar
; Bad Line:        63 - 40 = 23 Zyklen für CPU verfügbar

; YSCROLL-Trick: Bad Lines verschieben
; Normalerweise: YSCROLL=3 → Bad Lines auf Zeilen 3,11,19,27,35...

; IRQ-Handler auf einer Bad Line ist deutlich eingeschränkt!
; Lösung: IRQ auf Nicht-Bad-Lines legen

; Prüfen ob aktuelle Zeile eine Bad Line ist:
          LDA $D011        ; YSCROLL
          AND #$07         ; Bits 0-2
          STA TmpA
          LDA $D012        ; Aktuelle Rasterzeile
          AND #$07
          CMP TmpA
          BEQ IsBadLine    ; Gleich → Bad Line!

7.2 Sprite-DMA-Kosten

Pro aktivem Sprite stiehlt der VIC-II 2 Zyklen beim Sprite-Daten-Fetch (jeweils 3 Bytes = 3 Lesezugriffe à 2 Takte). Bei 8 aktiven Sprites:

; Sprite-DMA Kosten pro Zeile im Sprite-Bereich:
; 1 Sprite:  2 Zyklen Diebstahl pro Zeile
; 4 Sprites: 8 Zyklen Diebstahl
; 8 Sprites: 16 Zyklen Diebstahl

; Aber: Sprite-Fetch geschieht NUR in den Zeilen wo das Sprite sichtbar ist!
; Bei Sprite-Y-Pos = 100: Fetch von Zeile 100 bis 120 (21 Zeilen)

; Optimierung: Sprites in vertikalen Clustern anordnen
; Statt 8 Sprites gleichmäßig verteilt → 8 Sprites eng beieinander
; Damit ist der DMA-Overhead auf wenige Zeilen konzentriert,
; und der Rest des Bildschirms hat volle 63 Zyklen (minus Bad Lines)

7.3 Refresh-Zyklen

; Der VIC-II benötigt 5 DRAM-Refresh-Zyklen pro Zeile
; Diese 5 Zyklen sind immer weg, auch ohne Sprites oder Bad Lines
; Tatsächlich verfügbare Zyklen pro normaler Zeile:
; 63 - 5 (Refresh) = 58 Zyklen maximal
; Minus Bad Line: 58 - 40 = 18 Zyklen!
; Minus 8 Sprites: 58 - 16 = 42 Zyklen (normale Zeile, kein Bad Line)

; Rasterzeilen-Budget-Rechner:
; Verfügbare Zyklen = 63 - 5 (Refresh)
                         - (IsBadLine ? 40 : 0)
                         - (NumActiveSprites * 2)
Praktischer Tipp: VBlank nutzen
Die meisten cycle-intensiven Routinen (Sprite-Multiplexer-Sortierung, Map-Update, KI) sollten im vertikalen Blanking-Interval laufen: Rasterzeilen 250–311 (PAL). Hier zeichnet der VIC-II nichts — keine Bad Lines, keine Sprite-Fetches in den oberen Randbereichen.

8. Stack-Tricks und weitere Techniken

8.1 Stack als schneller Puffer

; Der Stack ($0100–$01FF) ist eine Zero-Page-ähnliche Region
; Direkter Zugriff über TSX + LDA $0100,X ist möglich!

; Stack als temporären Datenpuffer nutzen
SaveRegisters:
          PHA              ; A sichern: 3 Zyklen
          TXA : PHA        ; X: 2+3=5 Zyklen
          TYA : PHA        ; Y: 2+3=5 Zyklen
; Total: 13 Zyklen für A,X,Y

; Alternativ: Zero Page ist schneller wenn häufig genutzt
; STA Save_A: 3, STX Save_X: 3, STY Save_Y: 3 → 9 Zyklen

; Stack-Pointer-Trick: Direkter Datenpuffer
; Setze SP auf $FF-N → nächste N PHA-Befehle landen im "Puffer"
          LDX #$C0         ; SP = $C0 (64 Bytes Puffer: $01C1-$01FF)
          TXS
          ; Jetzt: PHA schreibt in $01FF, $01FE, ... rückwärts
          ; Lesen mit TSX / LDA $0101,X / INX...

8.2 Indirekte Dispatch-Tabellen

; Game-State-Machine: schneller Dispatch ohne if-else-Ketten

GameLoop:
          LDA GameState     ; 0=Titel, 1=Spiel, 2=GameOver, 3=Pause
          ASL               ; × 2 für Word-Tabelle
          TAX
          JMP (StateTable,X) ; 5 Zyklen — direkter Sprung!

StateTable:
          !word DoTitleScreen
          !word DoGameplay
          !word DoGameOver
          !word DoPause

DoTitleScreen:
          ; ... Titelbildschirm-Logik ...
          RTS ; (oder JMP GameLoop)

DoGameplay:
          ; ... Spiel-Logik ...
          RTS

8.3 Inline-Subroutinen (JSR-Elimination)

; JSR + RTS kostet: 6 + 6 = 12 Zyklen Overhead!
; Für kleine, oft aufgerufene Routinen: Inline-Code

; Schlecht: Winzige Subroutine mit Overhead
DoubleA:  ASL  : RTS       ; 2+6=8 Zyklen für den Aufruf
          JSR DoubleA      ; 6 Zyklen → 14 total für 2 Zyklen Arbeit!

; Gut: Direkt inline
          ASL              ; 2 Zyklen — nur die Arbeit selbst

; ACME Makro für wiederverwendbaren Inline-Code:
!macro DoubleA {
          ASL
}
          +DoubleA         ; Expandiert zu: ASL — kein Overhead!

8.4 16-Bit-Arithmetik optimieren

; 16-Bit Addition (häufige Operation):
; Standard:
          LDA Val16Lo      ; 3
          CLC              ; 2
          ADC Add16Lo      ; 3
          STA Val16Lo      ; 3
          LDA Val16Hi      ; 3
          ADC Add16Hi      ; 3
          STA Val16Hi      ; 3
; Total: 20 Zyklen

; Wenn Add16 eine Konstante: Unmittelbar statt Laden
; Z.B. +1 (INC-Methode):
          INC Val16Lo      ; 5 (ZP: 5 statt 6)
          BNE .done        ; 3/2 → meist 2 (kein Überlauf)
          INC Val16Hi      ; selten ausgeführt
.done:                     ; Total: 7 Zyklen im Normalfall!

; Für häufige +256 Addition (High-Byte nur):
          INC Addr+1       ; 6 Zyklen: Nur High-Byte inkrementieren
; Äquivalent zu: 16-Bit-Pointer um 256 vorwärts bewegen

Übung 8.1 — Cycle-Counting-Protokoll

Nimm die Clear-Screen-Routine aus Stufe 03 (256-Byte-Schleife mit LDA #$20 + STA $0400,X). Zähle die exakten Zyklen. Optimiere sie mit Loop Unrolling ×8 und miss den Unterschied in Rasterzeilen. Lege den Test in den VBlank und vergleiche.

Übung 8.2 — Sinustabellen-Bobber

Erstelle eine Routine, die 8 Sprites in einem "Bob"-Muster auf einer Sinuskurve auf und ab bewegt. Sprite 0 liest aus SinTable[Frame], Sprite 1 aus SinTable[Frame+32], Sprite 2 aus SinTable[Frame+64], usw. Der Frame-Zähler wird jeden VBlank um 2 erhöht. Ziel: flüssige 50-Hz-Bewegung ohne Ruckeln.

Übung 8.3 — Selbstmodifizierender Scroller

Implementiere einen Text-Scroller, der via SMC die Quell-Adresse im Kopierer patcht. Der Text ist 256 Zeichen lang und wird zyklisch auf die Bildschirmzeile 24 kopiert. Nutze keine Zero-Page-Pointer — nur direktes Adress-Patching im Kopierer-Code.

Übung 8.4 — LAX/DCP Umschreiben

Schreibe eine Kollisions-Prüfung (Sprite vs. Sprite per Bounding Box) zunächst mit Standard-Opcodes, dann mit LAX, DCP und AXS. Dokumentiere den Zyklenspareffekt pro Vergleich und die Gesamtersparnis bei 8×8 Sprite-Paaren.
