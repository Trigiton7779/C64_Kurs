1. VS Code als C64-IDE einrichten

Visual Studio Code hat sich als bevorzugte Entwicklungsumgebung für C64-Assembler etabliert. Mit den richtigen Extensions, Tasks und einer Launch-Konfiguration wird es zu einer vollwertigen IDE: Syntax-Highlighting, automatisches Bauen auf Tastendruck und direktes Starten in VICE.

1.1 Empfohlene Extensions

  • ACME Cross Assembler — Syntax-Highlighting für .asm-Dateien, ACME-spezifische Direktiven hervorgehoben
  • C64 VICE Debugger — Direkte VICE-Integration: Breakpoints aus VS Code setzen, in VICE landen
  • EditorConfig — Einheitliche Einstellungen für Tab-Größe und Zeilenenden im Team

1.2 Workspace-Einstellungen (.vscode/settings.json)

// .vscode/settings.json — C64-Projekteinstellungen { "files.associations": { "*.asm": "acme-assembly", "*.inc": "acme-assembly" }, "editor.tabSize": 8, "editor.insertSpaces": false, "editor.rulers": [40, 80], "files.eol": "\n", "editor.wordWrap": "off" }

1.3 Build-Task (tasks.json)

Der Build-Task kompiliert das Projekt mit ACME und zeigt Fehler direkt im Problems-Panel an:

// .vscode/tasks.json { "version": "2.0.0", "tasks": [ { "label": "Build C64 (ACME)", "type": "shell", "command": "acme", "args": [ "-f", "cbm", "-o", "build/${fileBasenameNoExtension}.prg", "--report", "build/report.txt", "${file}" ], "group": { "kind": "build", "isDefault": true }, "problemMatcher": { "owner": "acme", "fileLocation": "absolute", "pattern": { "regexp": "^(.*):(\\d+): (Error|Warning): (.*)$", "file": 1, "line": 2, "severity": 3, "message": 4 } } }, { "label": "Build & Run in VICE", "dependsOn": "Build C64 (ACME)", "type": "shell", "command": "x64sc", "args": ["-autostart", "build/${fileBasenameNoExtension}.prg"] } ] }

1.4 ACME-Snippets in VS Code

Snippets beschleunigen die tägliche Arbeit. Eine .vscode/c64.code-snippets-Datei mit den häufigsten Patterns:

// .vscode/c64.code-snippets { "BASIC Stub": { "prefix": "basicstub", "body": [ "* = $0801", "!byte $0b,$08,$e0,$07,$9e,$32,$30,$36,$31,$00,$00,$00", "* = $0810 ; Startadresse", "$1" ] }, "IRQ Handler": { "prefix": "irq", "body": [ "irq_handler:", " pha : txa : pha : tya : pha", " lda #\\$01 : sta \\$D019 ; IRQ quittieren", " ; --- IRQ-Code hier ---", " pla : tay : pla : tax : pla", " rti" ] } }
Drücke Ctrl+Shift+B für "Build" und Ctrl+Shift+P → Run Task → Build & Run für direktes Kompilieren und Starten in VICE. Mit einem Custom Keybinding lässt sich auch F5 für "Build & Run" belegen.

2. Makefile für C64-Projekte

Ein Makefile definiert, wie das Projekt gebaut wird — unabhängig von der IDE. Es ermöglicht make auf der Kommandozeile, in Scripts und in CI-Umgebungen. Für C64-Projekte verwaltet es ACME-Aufrufe, D64-Erstellung und optionale Exomizer-Komprimierung.

2.1 Vollständiges C64-Makefile

# C64 Projekt Makefile # Verwendung: make [Ziel] [PAL=1|NTSC=1] [DEBUG=1] # Beispiele: make → PAL-Build # make NTSC=1 run → NTSC-Build und in VICE starten # make dist → PAL + NTSC D64 erstellen ACME := acme C1541 := c1541 VICE := x64sc EXOMIZER:= exomizer SRC := src/main.asm BUILD := build TARGET := meinspiel # PAL/NTSC-Auswahl (Standard: PAL) DEFINES := ifeq ($(NTSC),1) DEFINES += -D NTSC=1 VARIANT := ntsc else DEFINES += -D PAL=1 VARIANT := pal endif ifeq ($(DEBUG),1) DEFINES += -D DEBUG=1 endif PRG := $(BUILD)/$(TARGET)_$(VARIANT).prg D64 := $(BUILD)/$(TARGET)_$(VARIANT).d64 .PHONY: all build run clean dist test all: build build: $(PRG) $(PRG): $(SRC) src/*.asm @mkdir -p $(BUILD) $(ACME) -f cbm $(DEFINES) \ --report $(BUILD)/report.txt \ --depfile $(BUILD)/deps.mk \ -o $@ $(SRC) @echo "✓ Build: $@" # Dependency-Tracking (regeneriert wenn !source-Dateien sich ändern) -include $(BUILD)/deps.mk run: build $(VICE) -autostart $(PRG) $(D64): $(PRG) $(C1541) -format "$(TARGET),00" d64 $@ \ -write $(PRG) "$(TARGET)" @echo "✓ D64: $@" dist: $(MAKE) PAL=1 $(BUILD)/$(TARGET)_pal.d64 $(MAKE) NTSC=1 $(BUILD)/$(TARGET)_ntsc.d64 @echo "✓ Dist-Paket fertig" clean: rm -rf $(BUILD)/ test: build ./test/run_tests.sh $(PRG)

2.2 PAL/NTSC-Defines in ACME verwenden

Die Makefile-Defines stehen im ACME-Code als Symbole zur Verfügung:

; In ACME: PAL/NTSC-bedingte Raster-Werte
!ifdef NTSC {
    RASTER_IRQ   = 230   ; NTSC: 263 Rasterzeilen
    FRAMES_PER_S = 60
} else {
    RASTER_IRQ   = 250   ; PAL: 312 Rasterzeilen
    FRAMES_PER_S = 50
}
Der ACME-Schalter --depfile build/deps.mk erzeugt automatisch eine Dependency-Datei: Wenn eine per !source eingebundene Datei sich ändert, weiß Make, dass main.asm neu gebaut werden muss. Ohne diesen Schalter würden Header-Änderungen ignoriert.

3. Exomizer — Datenkomprimierung für den C64

Exomizer ist der De-facto-Standard-Packer für C64-Programme. Er nutzt einen LZ77-Algorithmus, der speziell auf die Fähigkeiten des 6502-Dekomprimierers optimiert ist. Das Ergebnis: 40–60% Platzeinsparung bei typischen Spiel- und Demo-Daten, mit einem schnellen Dekomprimierer der oft nur 3–5 KB Overhead hat.

3.1 Was sich lohnt zu komprimieren

DatentypTypische EinsparungEmpfehlung
Bitmap-Grafik (Hires)35–55%Ja — hohe Redundanz in Bilddaten
SID-Musik-Daten30–50%Ja — wiederholende Patterns
Level-Tilemaps40–65%Ja — viele wiederholende Tiles
Sinustabellen / LUTs5–20%Wenig lohnend
Ausgeführter Code25–45%Nur für Loader/Intro, nicht für Kern-Code
Sprite-Daten (animiert)20–40%Ja, wenn viele Frames

3.2 Exomizer aufrufen

Exomizer hat zwei Hauptmodi für C64:

# Self-Extracting Executable (SFX) — Startet nach Dekomprimierung
exomizer sfx sys -t 64 -o output.prg input.prg

# Roh-komprimierte Daten (für manuelle Dekomprimierung)
exomizer raw -o bitmap.exo bitmap.bin

# SFX mit Startadresse (kein BASIC-Stub nötig)
exomizer sfx $0810 -t 64 -o packed.prg original.prg

3.3 Dekomprimierer-Integration in ACME

; exomizer_loader — Lädt komprimiertes Daten-File und dekomprimiert
; Der Exomizer-Dekruncher (decrunch.asm) wird als Binärdatei eingebunden
; Komprimierte Daten liegen bei PACKED_ADDR, Ziel bei TARGET_ADDR

PACKED_ADDR   = $C000   ; Komprimierte Daten: temporär hier laden
TARGET_ADDR   = $2000   ; Ziel: Bitmap-RAM
DECRUNCH_ADDR = $0900   ; Dekruncher liegt hier

; Dekruncher einbinden (Binärdatei erzeugt mit: exomizer level ...)
* = DECRUNCH_ADDR
!bin "decrunch.bin"

load_and_decrunch:
    ; Komprimierte Daten per KERNAL laden
    lda #1
    ldx #8               ; Gerät 8 (Floppy)
    tay
    jsr $FFBA            ; SETLFS
    lda #filename_end-filename
    ldx #<filename
    ldy #>filename
    jsr $FFBD            ; SETNAM
    lda #0               ; Load-Flag (kein Verify)
    ldx #<PACKED_ADDR
    ldy #>PACKED_ADDR
    jsr $FFD5            ; LOAD
    bcs load_error

    ; Dekomprimieren: Quelle = PACKED_ADDR, Ziel = TARGET_ADDR
    lda #<PACKED_ADDR
    sta $FD              ; Dekruncher-ZP: Quell-Zeiger
    lda #>PACKED_ADDR
    sta $FE
    jsr DECRUNCH_ADDR    ; Dekruncher aufrufen → entpackt nach TARGET_ADDR
    rts

filename:      !pet "bitmap.exo"
filename_end:
Für größte Effizienz: Komprimiere Grafik-Daten und Musik separat (nicht zusammen), da verschiedene Datentypen unterschiedliche Komprimierungsraten haben. Ein falsch komprimiertes File kann sogar größer werden als das Original — Exomizer warnt dann.

4. GitHub Actions CI für C64-Projekte

Continuous Integration für Retro-Entwicklung? Absolut sinnvoll. Bei jedem Push auf GitHub wird automatisch gebaut, und das fertige .d64-Image steht als Artifact zum Herunterladen bereit. Bei Release-Tags wird das Image direkt als GitHub-Release veröffentlicht.

git push
→
Ubuntu Runner
→
ACME kompilieren
→
D64 erstellen
→
Artifact upload
→
GitHub Release

4.1 Vollständiger GitHub-Actions-Workflow

# .github/workflows/build.yml
name: Build C64 Project

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]
  release:
    types: [ created ]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install ACME Assembler
        run: |
          sudo apt-get update -q
          sudo apt-get install -y acme

      - name: Install VICE (for c1541)
        run: |
          sudo apt-get install -y vice
          # Setzt erforderliche VICE ROM-Dateien
          mkdir -p ~/.config/vice
          cp /usr/share/vice/C64/*.vpl ~/.config/vice/ 2>/dev/null || true

      - name: Build PAL version
        run: make PAL=1 dist

      - name: Build NTSC version
        run: make NTSC=1 dist

      - name: Upload PAL Artifact
        uses: actions/upload-artifact@v4
        with:
          name: build-pal
          path: build/*_pal.d64
          retention-days: 30

      - name: Upload NTSC Artifact
        uses: actions/upload-artifact@v4
        with:
          name: build-ntsc
          path: build/*_ntsc.d64
          retention-days: 30

  release:
    needs: build
    if: github.event_name == 'release'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build all variants
        run: make dist

      - name: Upload Release Assets
        uses: softprops/action-gh-release@v1
        with:
          files: |
            build/*_pal.d64
            build/*_ntsc.d64
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
GitHub Actions stellt Ubuntu-Runner kostenlos für öffentliche Repositories zur Verfügung. Für private Repositories gibt es ein monatliches Frei-Kontingent von 2000 Minuten. Ein typischer C64-Build dauert unter 30 Sekunden.

5. Headless-VICE-Testing

VICE kann ohne grafisches Fenster laufen — im "Headless"-Modus. Das ermöglicht automatisierte Tests in CI-Umgebungen: Das Programm startet, läuft für eine definierte Anzahl Frames, und ein Screenshot wird als Referenzvergleich gespeichert.

5.1 VICE Headless-Modus

# VICE headless starten, Programm auto-starten, 300 Frames laufen
x64sc -headless -autostart meinspiel.prg \
      -limitcycles 300000 \
      -screenshot test/output.png \
      -1

# -headless: kein GUI-Fenster
# -limitcycles: nach N Zyklen beenden (300000 ≈ ~5 Sekunden PAL)
# -screenshot: Screenshot beim Beenden speichern

5.2 Screenshot-Vergleichs-Testskript

#!/bin/bash
# test/run_screenshot_test.sh — CI-Screenshot-Regression-Test

PRG="$1"
REFERENCE="test/reference.png"
OUTPUT="test/output.png"
DIFF_THRESHOLD=500  # Max. erlaubte Pixel-Differenz

# VICE headless 5 Sekunden laufen lassen
x64sc -headless -autostart "$PRG" \
      -limitcycles 300000 \
      -screenshot "$OUTPUT" \
      -1 2>/dev/null

# Prüfe ob Screenshot erstellt wurde
if [ ! -f "$OUTPUT" ]; then
    echo "FEHLER: Kein Screenshot erstellt (Programmabsturz?)"
    exit 1
fi

# Screenshot mit Referenz vergleichen (ImageMagick)
DIFF=$(compare -metric AE "$REFERENCE" "$OUTPUT" /dev/null 2>&1)
echo "Pixel-Differenz: $DIFF"

if [ "$DIFF" -gt "$DIFF_THRESHOLD" ]; then
    echo "FEHLER: Screenshot weicht um $DIFF Pixel ab (Grenze: $DIFF_THRESHOLD)"
    compare "$REFERENCE" "$OUTPUT" test/diff.png  # Diff-Bild erzeugen
    exit 1
fi

echo "OK: Screenshot-Test bestanden ($DIFF Pixel Differenz)"
exit 0

5.3 Grenzen des Headless-Testings

Test-TypHeadless-VICEEchter Hardware-Test
Programm startet ohne Crash✓ Zuverlässig✓
Grafik korrekt nach N Frames✓ Screenshot-Vergleich✓
Audio/SID-Ausgabe✗ Nicht testbar✓
Timing auf echter Hardware✗ Emulator weicht ab✓ Pflicht
Joystick/Tastatur~ Skriptbar via Monitor✓
PAL/NTSC-Unterschiede✓ VICE-Flag wechseln✓
Headless-Testing im Emulator ersetzt keinen Test auf echter Hardware. Was im Emulator funktioniert, muss auf einem echten C64 nicht korrekt laufen — besonders bei zeitkritischem Code, Raster-Effekten und SID-Sounds. CI-Tests sind Smoke-Tests, keine Garantie.

6. Multi-File-ACME-Projekte — Struktur und Konventionen

6.1 Empfohlene Projektstruktur

meinspiel/ ├── src/ │ ├── main.asm ← Einstiegspunkt, !source alle anderen │ ├── memory_map.asm ← ZP-Adressen, RAM-Layout, Konstanten │ ├── macros.asm ← Wiederverwendbare !macro Definitionen │ ├── irq.asm ← IRQ-Handler │ ├── player.asm ← Spieler-Logik │ ├── gfx.asm ← VIC-II Initialisierung, Rendering │ └── snd.asm ← SID-Musik-Player ├── assets/ │ ├── charset.bin ← CharPad-Export │ ├── sprites.bin ← SpritePad-Export │ └── music.bin ← GoatTracker-Export ├── build/ ← .gitignore: generierte Dateien ├── test/ ← Testskripte, Referenz-Screenshots ├── .github/workflows/ ← CI-Konfiguration ├── .vscode/ ← tasks.json, settings.json ├── Makefile └── README.md

6.2 memory_map.asm — Zentrales ZP-Register

Alle Zero-Page-Variablen werden zentral reserviert. Kein Modul darf ohne Eintrag hier ZP-Adressen vergeben — so vermeidet man ZP-Konflikte zwischen Modulen:

; memory_map.asm — Zentrales Zero-Page-Layout
; ALLE ZP-Adressen HIER reservieren — nie im Modul-Code

; ── System (nicht verwenden) ───────────────────────────────
; $00-$01: CPU-I/O-Port (für !source memory_map.asm)
; $02-$0F: Reserviert / von BASIC genutzt

; ── IRQ-System ─────────────────────────────────────────────
ZP_IRQ_COUNTER   = $10   ; Frame-Counter (1 Byte)
ZP_IRQ_FLAGS     = $11   ; IRQ-Event-Flags

; ── Spieler ────────────────────────────────────────────────
ZP_PLR_X         = $12   ; Spieler X-Position (16-Bit: $12/$13)
ZP_PLR_X_HI      = $13
ZP_PLR_Y         = $14
ZP_PLR_VY        = $15   ; Vertikale Geschwindigkeit (signed)
ZP_PLR_STATE     = $16   ; 0=Stehen, 1=Laufen, 2=Springen

; ── Grafik-Routinen ────────────────────────────────────────
ZP_GFX_PTR       = $18   ; Allg. 16-Bit-Zeiger für gfx.asm
ZP_GFX_PTR_HI    = $19

; ── REU-Transfer-Parameter ─────────────────────────────────
ZP_REU_C64_LO    = $20   ; Für reu_transfer Subroutine
ZP_REU_C64_HI    = $21
ZP_REU_ADDR_LO   = $22
ZP_REU_ADDR_MI   = $23
ZP_REU_ADDR_HI   = $24
ZP_REU_LEN_LO    = $25
ZP_REU_LEN_HI    = $26
ZP_REU_CMD       = $27

6.3 macros.asm — Bibliothek häufiger Makros

; macros.asm — Wiederverwendbare ACME-Makros
; Einbinden: !source "src/macros.asm" (nur einmal in main.asm)

; Warten bis VBlank (Rasterzeile > 250)
!macro wait_vblank {
    bit $D011        ; Bit 7 = Rasterzeile-Bit-8
    bpl *-3          ; Warten bis im oberen Raster-Bereich
    bit $D011
    bmi *-3          ; Warten bis zurück im sichtbaren Bereich
}

; Zero-Page-Variablen sichern und wiederherstellen (für IRQ-Handler)
!macro save_zp .from, .to {
    ldx #(.to - .from)
.loop:
    lda .from,x
    sta zp_save_buf,x
    dex
    bpl .loop
}

!macro restore_zp .from, .to {
    ldx #(.to - .from)
.loop:
    lda zp_save_buf,x
    sta .from,x
    dex
    bpl .loop
}

; Sprite an (X,Y) positionieren (X: 0-319, Y: 0-255)
!macro set_sprite_pos .num, .x, .y {
    lda #(.x AND $FF)
    sta $D000 + (.num * 2)
    lda #(.y AND $FF)
    sta $D001 + (.num * 2)
    !if (.x > 255) {
        lda $D010 : ora #(1 << .num) : sta $D010
    } else {
        lda $D010 : and #(255 - (1 << .num)) : sta $D010
    }
}

7. Release-Management — Versionierung und Distribution

7.1 Semantic Versioning für C64-Releases

Das klassische Semantic-Versioning-Schema MAJOR.MINOR.PATCH ist auch für C64-Projekte sinnvoll:

  • MAJOR — Grundlegende Änderung: neues Gameplay-Konzept, komplett neues Level-Design
  • MINOR — Neue Feature: neues Level, neue Waffe, Bug-Fix mit sichtbarem Effekt
  • PATCH — Bugfix: Absturz behoben, Timing-Korrektur, NTSC-Fix

Beispiel: v1.2.0-pal — Major-Release 1, Minor 2, Patch 0, PAL-Variante.

7.2 Git-Tags und GitHub-Releases

# Release erstellen
git tag -a v1.0.0 -m "Erste vollständige Version"
git push origin v1.0.0

# GitHub Actions baut automatisch und erstellt Release mit D64-Assets
# Nach dem Push erscheint unter Releases:
# meinspiel_pal.d64  (PAL-Version)
# meinspiel_ntsc.d64 (NTSC-Version)

7.3 CHANGES.txt im Scene-Stil

Ein CHANGES.txt im .NFO-Stil dokumentiert den Release-Verlauf und gehört in jedes D64-Image:

; CHANGES.TXT — Versionsverlauf JUMPMASTER64

v1.0.0 (2025-01)  Erstveröffentlichung auf CSDb
  + 5 Level, Endgegner, High-Score-Tabelle

v1.0.1 (2025-02)  Bugfix-Release
  * NTSC: Raster-IRQ-Timing korrigiert
  * Level 3: Kollisionsfehler an Plattform-Kante behoben

8. Walkthrough: Vollständige CI/CD-Pipeline auf GitHub

Schritt für Schritt vom leeren Repository zur automatischen D64-Generierung und GitHub-Release.

8.1 Repository-Struktur anlegen

Anlegen des Projekts mit der Verzeichnisstruktur aus Abschnitt 6.1. Die wichtigste Datei zuerst: .gitignore:

build/
*.o
*.prg
!test/reference.prg

8.2 Makefile und VS Code einchecken

Commit mit Makefile, tasks.json und settings.json. Ab jetzt funktioniert Ctrl+Shift+B für alle Teammitglieder.

8.3 GitHub Actions aktivieren

Die Datei .github/workflows/build.yml aus Abschnitt 4.1 einchecken. Beim ersten Push startet GitHub Actions automatisch. Im Tab "Actions" des GitHub-Repositories sieht man den Build-Status in Echtzeit.

8.4 Referenz-Screenshot für Tests erstellen

# Einmalig: Referenz-Screenshot erzeugen und einchecken
make build
x64sc -headless -autostart build/meinspiel_pal.prg \
      -limitcycles 300000 -screenshot test/reference.png -1
git add test/reference.png
git commit -m "chore: add CI reference screenshot"

8.5 Ersten offiziellen Release erstellen

# v1.0.0 taggen und pushen — Actions erstellt GitHub Release automatisch
git tag -a v1.0.0 -m "Erster offizieller Release"
git push origin v1.0.0

# Unter github.com/user/repo/releases erscheint danach:
# - meinspiel_pal.d64
# - meinspiel_ntsc.d64
# als herunterladbare Assets
Die gesamte Einrichtung (Repository, Makefile, VS Code, GitHub Actions) dauert für ein bestehendes Projekt ca. 2–3 Stunden. Das amortisiert sich sofort: Jeder Commit baut automatisch, und Release-Distribution ist von da an ein einziges git tag.

Übungsaufgaben

Übung 14.1 — VS Code vollständig einrichten

Richte VS Code für ein bestehendes C64-Projekt ein. Erstelle tasks.json mit Build-Task und "Build & Run"-Task. Stelle sicher, dass ACME-Fehler als Probleme im Problems-Panel erscheinen (problemMatcher). Definiere mindestens drei nützliche Code-Snippets: BASIC-Stub, IRQ-Handler, LDA/STA-Schleife.

Übung 14.2 — Makefile mit PAL/NTSC

Schreibe ein vollständiges Makefile für ein bestehendes Projekt. Füge PAL/NTSC-Targets hinzu, die unterschiedliche ACME-Defines setzen. Teste: make PAL=1 dist und make NTSC=1 dist erzeugen zwei verschiedene D64-Images. Prüfe mit VICE, dass der richtige Raster-Wert für jede Variante kompiliert wurde.

Übung 14.3 — Exomizer-Integration

Nehme ein bestehendes Projekt mit Bitmap-Daten oder Sinustabellen. Komprimiere die Datendateien einzeln mit Exomizer. Messe die tatsächliche Einsparung in Bytes und Prozent. Integriere den Dekruncher-Aufruf in den Loader. Vergleiche die Ladezeit via Rahmenfarben-Methode (mit und ohne Exomizer).
