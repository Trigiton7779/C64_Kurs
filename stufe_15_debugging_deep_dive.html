1. VICE Monitor — Kommando-Referenz im Detail

Der VICE Monitor ist das mächtigste Werkzeug des C64-Entwicklers. Er ist ein vollständiger Maschinensprache-Monitor direkt im Emulator — kein externes Tool, kein Compile-Upload-Cycle. Hier kann man das laufende Programm in Echtzeit inspizieren und verändern.

1.1 Den Monitor öffnen

  • Alt+H (oder Menu: Debug → Activate Monitor) — hält das Programm an und öffnet den Monitor
  • BRK-Opcode ($00) im Code — löst automatisch den Monitor aus wenn der Prozessor darauf trifft
  • TCP-Interface — VICE mit -remotemonitor starten, dann via telnet localhost 6510 verbinden (für IDE-Integration)

2. Breakpoints und Watchpoints — Präzise Kontrolle

2.1 Bedingte Breakpoints

Bedingte Breakpoints halten das Programm nur an, wenn eine Bedingung erfüllt ist. Das ist unverzichtbar wenn ein Bug nur unter bestimmten Umständen auftritt:

; Breakpoint: Hält nur an wenn A == $42 (C:$...) break 1200 if a==$42 ; Breakpoint: Hält an wenn X > 7 (Sprite-Index außerhalb Bereich) (C:$...) break 1300 if x>7 ; Breakpoint: Wenn Speicherstelle einen bestimmten Wert hat (C:$...) break 1400 if $12==$FF ; Watchpoint: Hält an wenn $D020 (Rahmenfarbe) beschrieben wird (C:$...) watch store D020 ; Nützlich: Rahmenfarben-Methode automatisiert → wann genau wird die Farbe gesetzt? ; Watchpoint auf Lesezugriff: Wer liest Rasterzeile $D012? (C:$...) watch load D012

2.2 Trace-Modus — Ausführungsprotokoll

Der Trace-Modus protokolliert jede ausgeführte Instruktion in einem Adressbereich. Das erzeugt viel Ausgabe, aber ermöglicht exakte Analyse einer Subroutine:

; Trace-Log für Scroller-Update-Routine (Adresse $1100-$1180) (C:$...) trace 1100 1180 ; Programm weiter ausführen (C:$...) x TRACE: .C:1100 A5 20 LDA $20 A:04 X:00 Y:00 TRACE: .C:1102 85 22 STA $22 A:04 X:00 Y:00 TRACE: .C:1104 E6 20 INC $20 A:04 X:00 Y:00 ... (eine Zeile pro Instruktion) ; Trace deaktivieren (C:$...) del 1
Trace erzeugt extrem viel Ausgabe. Am besten nutzen für kurze, verdächtige Routinen (<50 Instruktionen). Für längere Routinen besser: gezielter bedingter Breakpoint am verdächtigen Punkt.

3. CPU-Profiling — Hotspots finden

3.1 Die Rahmenfarben-Methode (inline)

Die schnellste Profiling-Methode: Setze $D020 (Rahmenfarbe) auf verschiedene Werte vor und nach einer Routine. Im laufenden VICE sieht man am Bildschirmrand farbige Streifen — je breiter (= mehr Rasterzeilen), desto mehr Zyklen braucht die Routine:

; Rahmenfarben-Profiling: Routine messen
lda #2 : sta $D020  ; Rahmen ROT — Routine beginnt

; ... die zu messende Routine ...
jsr meine_routine

lda #0 : sta $D020  ; Rahmen SCHWARZ — Routine ende

; Breite des roten Streifens im Rasterbalken = Zyklen/63 Rasterzeilen

3.2 cycle_measure Makro — CIA2-Timer-basierte Messung

Für genaue Zyklenangaben nutzt man CIA2-Timer A. Der Timer läuft rückwärts mit CPU-Takt. Differenz vor und nach der Routine ergibt die exakte Zyklenanzahl:

; cycle_measure_start — Startet CIA2-Timer für Zeitmessung
; cycle_measure_stop  — Stoppt Timer und gibt Ergebnis aus
; Ergebnis: monitor-Ausgabe via BRK mit Zyklenzahl in X/Y

CIA2_TA_LO  = $DD04
CIA2_TA_HI  = $DD05
CIA2_CRA    = $DD0E

cycle_measure_start:
    lda #$FF : sta CIA2_TA_LO  ; Timer auf $FFFF laden
    lda #$FF : sta CIA2_TA_HI
    lda #$01 : sta CIA2_CRA    ; Timer starten (Bit 0)
    rts

cycle_measure_stop:
    lda #0 : sta CIA2_CRA     ; Timer stoppen
    lda CIA2_TA_HI : sta measure_hi
    lda CIA2_TA_LO
    ; Zyklen = $FFFF - Timer-Wert
    eor #$FF : sta measure_lo
    lda measure_hi : eor #$FF : sta measure_hi
    ; Breakpoint: Monitor zeigt Zyklenzahl an
    brk                        ; Monitor öffnet sich; "r" zeigt A,X,Y

measure_lo: !byte 0
measure_hi: !byte 0

; Verwendung:
;   jsr cycle_measure_start
;   jsr meine_teure_routine
;   jsr cycle_measure_stop
; → Monitor öffnet sich → "m measure_lo" zeigt 16-Bit-Zyklenzahl
Faustformeln für PAL-Timing: 63 Zyklen pro Rasterzeile, 312 Rasterzeilen pro Frame, 985.248 Hz CPU-Takt. Ein Frame = 19.656 Zyklen. Eine Routine, die mehr als 3.000 Zyklen braucht, frisst über 15% eines Frames.

4. Badlines — Der unsichtbare Flaschenhals

Badlines sind die häufigste Ursache für unerklärlich langsamen Code und Raster-Jitter. Sie entstehen, weil der VIC-II regelmäßig Daten aus dem Speicher lesen muss — und dafür den 6510-Bus für 40 Zyklen sperrt.

4.1 Was ist eine Badline?

In jeder Rasterzeile, in der der VIC-II Character-Daten vom Speicher holen muss, stiehlt er dem 6510 exakt 40 Zyklen (Zyklen 15–54 der 63-Zyklus-Rasterzeile). Übrig bleiben nur 23 Zyklen für Programmcode.

4.2 Die Badline-Formel

Eine Rasterzeile ist eine Badline wenn:

  1. Die Rasterzeile liegt im sichtbaren Bildschirmbereich: $30 ≤ Rasterzeile ≤ $F7 (PAL)
  2. Die unteren 3 Bits der Rasterzeile stimmen mit YSCROLL überein: (Rasterzeile AND %111) == (YSCROLL AND %111)

Mit dem Standard-YSCROLL-Wert von 3 ($D011 Bits 0–2) sind also Rasterzeilen $33, $3B, $43, $4B, ... badlines — jede 8. Zeile.

Rasterzeilen-TypVerfügbare ZyklenZyklen gesperrt durch VIC
Normale Zeile (kein Sprite)63 Zyklen0
Badline (Normal-YSCROLL)23 Zyklen40 (DMA für Char-Daten)
Badline + 1 Sprite21 Zyklen42
Badline + 8 Sprites7 Zyklen56
VBlank (Zeilen 300–311)63 Zyklen0 (kein DMA)

4.3 FLD — Flexible Line Distance: Badlines eliminieren

FLD (Flexible Line Distance) ist eine Demo-Technik, die YSCROLL in jedem Raster-IRQ so ändert, dass die Badline-Bedingung nie erfüllt wird. Indem man YSCROLL von 3 auf 4, 5, 6, 7, 0, 1, 2, 3 wechselt (immer in der vorherigen Rasterzeile), schiebt man die Badline immer eine Zeile weiter — sie trifft nie genau.

; fld_disable — Verhindert Badlines in kritischen Rasterzeilen
; Muss im Raster-IRQ jeder betroffenen Zeile aufgerufen werden
; Prinzip: YSCROLL jede Zeile erhöhen → Bedingung nie erfüllt

VIC_CTRL1  = $D011
VIC_CTRL2  = $D016
RASTERLINE = $D012

fld_yscroll: !byte 3   ; Aktueller YSCROLL-Wert

fld_irq_handler:
    pha
    lda #$01 : sta $D019  ; IRQ quittieren

    ; YSCROLL erhöhen (modulo 8) → Badline verschiebt sich um 1
    lda fld_yscroll
    clc
    adc #1
    and #%00000111        ; Auf 0-7 begrenzen
    sta fld_yscroll

    ; In $D011 einschreiben (Bits 2-0 = YSCROLL, Bits 7-3 unverändert)
    lda VIC_CTRL1
    and #%11111000        ; YSCROLL-Bits löschen
    ora fld_yscroll
    sta VIC_CTRL1

    ; Nächste IRQ-Zeile setzen (eine Zeile tiefer)
    lda RASTERLINE
    clc
    adc #1
    sta RASTERLINE
    pla
    rti
FLD bewegt den Bildschirminhalt vertikal — jede YSCROLL-Änderung verschiebt den Text um ein Pixel. Das ist bei reinem FLD für Code-Laufzeit gewollt, kann aber den Screen unleserlich machen. In Spielen nutzt man FLD daher nur in Bereichen ohne Bildschirminhalt (Schwarzrand) oder kompensiert mit entsprechender Darstellung.

5. Page-Cross-Bugs — Versteckte Cycle-Kosten

5.1 Was ist ein Page-Cross?

Eine "Seite" (Page) im 6510-Adressraum ist ein 256-Byte-Block: $0000–$00FF ist Page 0, $0100–$01FF ist Page 1, usw. Einige Adressierungsarten müssen prüfen, ob bei der Adressberechnung eine Seitengrenze überschritten wird. Falls ja, kostet der Zugriff einen extra Zyklus.

5.2 Betroffene Instruktionen

InstruktionNormalfallMit Page-CrossMehrkosten
LDA abs,X / LDA abs,Y4 Zyklen5 Zyklen+1
LDX abs,Y / LDY abs,X4 Zyklen5 Zyklen+1
STA abs,X / STA abs,Y5 Zyklen5 Zyklen0 (immer 5!)
(ind),Y — LDA (ptr),Y5 Zyklen6 Zyklen+1
Branch (BNE, BEQ, ...)2/3 Zyklen4 Zyklen+1 (über Seitengrenze)
JMP (ind)5 ZyklenFalsches Ergebnis!Hardware-Bug $xxFF

5.3 Tabellen page-aligned ablegen

Der häufigste Page-Cross-Bug: eine Lookup-Tabelle (z. B. Sinustabelle) liegt so im Speicher, dass der Zugriff mit LDA sin_table,X bei hohen X-Werten die Seitengrenze überschreitet. Lösung: Tabelle auf eine 256-Byte-Grenze ausrichten.

; PROBLEM: Sinustabelle NICHT page-aligned — bei X > n kostet LDA sin_table,X 5 statt 4 Zyklen
* = $1050         ; Tabelle bei $1050 → endet bei $114F → überschreitet $1100!
sin_table:  !byte 128,140,...  ; 256 Bytes — aber $1050+$FF = $114F → Page-Cross bei X>$AF

; LÖSUNG: Tabelle explizit page-aligned ablegen
; Methode 1: Direkte Adresse wählen
* = $1100         ; Exakt auf Seitengrenze $xx00
sin_table:  !byte 128,140,...  ; $1100+$FF = $11FF → KEIN Page-Cross!

; Methode 2: ACME-Ausdruck für automatische Ausrichtung
* = ((*+255)/256)*256  ; Auf nächste 256er-Grenze aufrunden
sin_table:  !byte 128,140,...

; Zyklen-Vergleich (innerste Schleife, 256 Iterationen):
;   Ohne Page-Align: ~128 Page-Crosses × 1 Zyklus = 128 Mehrzyklen
;   Mit Page-Align:  0 Page-Crosses = 0 Mehrzyklen
; Bei 50 Hz: 128 × 50 = 6400 verschwendete Zyklen pro Sekunde

5.4 JMP ($xxFF) — Der 6502-Hardware-Bug

Wenn der Adress-Pointer für JMP (ind) genau an einer Seitengrenze liegt (z. B. JMP ($10FF)), liest der 6502/6510 das Low-Byte der Zieladresse korrekt von $10FF, aber das High-Byte fälschlicherweise von $1000 statt $1100. Dies ist ein bekannter Hardware-Bug des 6502, der im 6510 nicht behoben wurde.

Dispatch-Tabellen für indirekten JMP niemals so aufbauen, dass ein Tabelleneintrag bei $xxFF liegt. Tabelle immer page-aligned ablegen.

6. Stack-Overflow-Diagnose

6.1 Der 6510-Stack: Beschränkungen und Überlauf

Der Stack des 6510 liegt fest bei $0100–$01FF. Das Stack-Pointer-Register SP zeigt auf den nächsten freien Slot und wächst von $FF abwärts. Beim Start zeigt SP typischerweise auf $FF (Stack leer). Jeder JSR schiebt 2 Bytes (Return-Adresse) auf den Stack, jeder PHA 1 Byte. Wenn SP auf $00 gefallen ist und noch ein Push kommt, wraps der Zeiger auf $FF zurück und überschreibt ältere Stack-Einträge — Korruption der Rücksprungadressen.

6.2 Stack-Overflow erkennen

; stack_guard — Canary-Byte gegen Stack-Overflow
; Schreibt bekanntes Muster an unteres Stack-Ende
; IRQ-Handler prüft regelmäßig ob Canary intakt ist

STACK_CANARY_ADDR = $0101  ; Tiefstes erreichbares Stack-Byte
CANARY_BYTE       = $C3    ; Zufälliger Wert — unwahrscheinlich als Zufallswert

install_stack_guard:
    lda #CANARY_BYTE
    sta STACK_CANARY_ADDR
    rts

; Im IRQ-Handler aufrufen (oder periodisch im Hauptprogramm)
check_stack_guard:
    lda STACK_CANARY_ADDR
    cmp #CANARY_BYTE
    beq .stack_ok
    ; CANARY ÜBERSCHRIEBEN → Stack-Overflow aufgetreten!
    brk                    ; Monitor öffnen für Post-Mortem-Analyse
.stack_ok:
    rts

; Im Monitor: "r" zeigt SP — wenn SP < $10, ist der Stack kritisch voll
; "m 0100 01FF" zeigt kompletten Stack-Inhalt

6.3 Häufige Ursachen für Stack-Overflow

  • Fehlendes RTS: Subroutine springt per JMP aus einer Schleife heraus, ohne die Return-Adresse vom Stack zu nehmen. Bei jedem Aufruf wächst der Stack.
  • IRQ-Handler ohne RTI: Wenn ein IRQ-Handler mit RTS statt RTI beendet wird, werden 1 Byte (Statusregister) nie entfernt.
  • Tiefe Rekursion: Rekursiver Algorithmus ohne Tiefenbegrenzung.
  • PHA ohne PLA: In einem Codepfad (z. B. bei einer Verzweigung) wird ein PHA nicht mit PLA ausgeglichen.
Im VICE-Monitor: watch store 0100 010F setzt einen Watchpoint auf die untersten 16 Stack-Bytes. Sobald der Stack so weit gewachsen ist, hält der Monitor das Programm an — lange bevor echte Korruption eintritt.

7. Sprite-DMA-Cycle-Stealing

7.1 Wie Sprite-DMA Zyklen stiehlt

Der VIC-II muss Sprite-Daten aus dem Speicher lesen, bevor er sie darstellen kann. Pro aktivem Sprite werden 2 Zyklen pro Rasterzeile gestohlen — und das in zwei Zeilen: der Zeile, in der das Sprite anfängt sichtbar zu sein (p-access), und jeder Zeile innerhalb des Sprite-Bereichs.

8. Häufige Assembler-Fallen

8.1 Zero-Page vs. Absolute Adressierung

ACME wählt automatisch die kürzeste Adressierungsart: LDA $10 → Zero-Page (2 Bytes, 3 Zyklen). LDA $0010 → Absolut (3 Bytes, 4 Zyklen). Das ist meistens korrekt — aber manchmal will man explizit die absolute Variante, z. B. wenn man sichergehen will, dass eine Speicherstelle im I/O-Bereich angesprochen wird.

8.2 BCC/BCS-Verwirrung nach CMP

Nach CMP A, N: Carry wird gesetzt wenn A >= N (unsigned). Das verwirrt viele Programmierer, weil man intuitiv "Carry = Übertrag" denkt:

BedingungKorrekte Branch-InstruktionFalsch (häufiger Fehler)
A == NBEQ—
A != NBNE—
A < N (unsigned)BCC (Carry GELÖSCHT)BCS — falsch!
A >= N (unsigned)BCS (Carry GESETZT)BCC — falsch!
A <= N (unsigned)BEQ + BCC (zwei Branches)BCS — falsch!
A > N (unsigned)BNE + BCSBCC — falsch!

8.3 Dezimal-Modus (BCD) vergessen

SED aktiviert den BCD-Modus: Addition und Subtraktion arbeiten danach als binär kodierte Dezimalzahl. Das bleibt aktiv bis CLD. Wer nach SED vergisst, CLD aufzurufen, bekommt in allen nachfolgenden arithmetischen Operationen falsche Ergebnisse — auch im IRQ-Handler!

8.4 Branch-Reichweite überschritten

Bedingte Branches (BNE, BEQ, etc.) können nur ±127 Bytes weit springen. ACME gibt beim Kompilieren einen Fehler aus. Workaround:

; PROBLEM: BNE springt weiter als 127 Bytes
; LÖSUNG: Invertiere die Bedingung und springe zu einem nahen JMP
original_code:
    cmp #10
    bne far_target      ; FEHLER: zu weit weg

fixed_code:
    cmp #10
    beq .skip           ; Bedingung invertiert: gleich → überspringen
    jmp far_target      ; JMP hat volle 16-Bit-Reichweite
.skip:

9. Walkthrough: Buggy Platformer debuggen

Fünf absichtlich eingebaute Bugs werden ausschließlich mit VICE-Monitor-Mitteln diagnostiziert und behoben — kein Raten, nur Messung.

Das Walkthrough-Projekt ist ein einfacher Plattformer mit fünf versteckten Bugs. Das Ziel: jeden Bug mit dem VICE-Monitor diagnostizieren, ohne einfach den Code zu lesen und nach dem Fehler zu suchen. Die Debug-Methode steht im Vordergrund.

Bug 1 — Spieler fällt durch Plattformen

Symptom: Manchmal fällt der Spieler durch die Plattform, wenn er von oben darauf springt.

Diagnose: watch store ZP_PLR_VY — Wann und wie wird die Y-Geschwindigkeit auf 0 gesetzt? Trace auf die Kollisionsroutine zeigt: Der Branch für "Spieler landet" verwendet BCS statt BEQ nach dem Tile-Vergleich.

Fix: CMP #PLATFORM_TILE + BEQ (nicht BCS) für Landen-Erkennung.

Bug 2 — Score springt um 2 statt 1

Symptom: Beim Einsammeln eines Münze erhöht sich der Score immer um 2.

Diagnose: watch store ZP_SCORE — Breakpoint zeigt, dass score_increment zweimal pro Frame aufgerufen wird. Trace auf die Coin-Collision-Routine: JSR score_increment steht in zwei unabhängigen Code-Pfaden die beide in derselben Situation aktiv sind.

Fix: Zweiten JSR-Aufruf entfernen.

Bug 3 — Raster-Jitter trotz Double-IRQ

Symptom: Die Rasterbars zittern leicht, obwohl Double-IRQ implementiert ist.

Diagnose: watch store D020 — Zeitpunkt des Farbwechsels variiert. Prüfe mit Rahmenfarben-Methode: Der Jitter tritt nur in Rasterzeile $43 auf. Das ist eine Badline! Der Double-IRQ landet gelegentlich auf einer Badline und verliert die 40 Zyklen.

Fix: FLD-Routine in die betroffenen IRQ-Zeilen integrieren, oder IRQ-Zeile so wählen, dass sie keine Badline ist.

Bug 4 — Musik friert nach ~3 Minuten ein

Symptom: Nach etwa 3 Minuten Spielzeit friert die Musik ein, dann stürzt das Programm ab.

Diagnose: watch store 0100 0110 — Stack-Bereich überwachen. Nach ~3 Minuten trifft der Watchpoint an: SP ist auf $08 gefallen. Der Musik-Player ruft eine Hüllkurven-Subroutine rekursiv auf — bei bestimmten ADSR-Zuständen gibt es einen Codepfad ohne RTS.

Fix: stack_guard installieren, fehlenden RTS im Musik-Player-Codepfad ergänzen.

Bug 5 — Sprite verschwindet bei X > 255

Symptom: Ein Feind-Sprite wird unsichtbar sobald er den rechten Bildschirmrand überschreitet.

Diagnose: break sprite_update_x if x==0 — Monitor zeigt: $D010 (MSB der Sprite-X-Koordinaten) wird nie gesetzt. Die Sprite-Positions-Lookup-Tabelle liegt bei $1EF0 — bei Index $10 wird $1F00 gelesen: Page-Cross! Falscher Tabellenwert für X-MSB.

Fix: Sprite-Positions-Tabelle page-aligned ablegen (* = $1F00).

9.1 Lessons Learned — Die fünf Debug-Schritte

  1. Symptom beobachten: Genau beschreiben, wann und unter welchen Bedingungen der Bug auftritt.
  2. Hypothese aufstellen: Welche Codekomponente könnte verantwortlich sein?
  3. Messung durchführen: Watchpoint/Breakpoint/Trace auf die verdächtige Stelle setzen.
  4. Fix implementieren: Minimale Änderung — nur das beheben, was die Messung gezeigt hat.
  5. Verifikation: Watchpoint erneut setzen und prüfen, dass das fehlerhafte Verhalten ausbleibt.
Die häufigste Debugger-Falle ist "Raten statt Messen". Professionelle Debugging-Praxis bedeutet: Keine Codeänderung ohne vorherige Diagnose. Der VICE-Monitor gibt einem alle Werkzeuge dafür — nutze sie.

  • Du setzt einen Watchpoint auf einen Schreibzugriff (watch store $D020) und identifizierst welcher Code das Register beschreibt
  • Du erkennst eine Badline anhand der Formel und weißt exakt wie viele Zyklen sie dem 6510 stiehlt
  • Du implementierst FLD (Flexible Line Distance) um Badlines in zeitkritischen Rasterzeilen zu eliminieren
  • Du diagnostizierst einen Stack-Overflow mit SP-Monitoring und findest die Subroutine mit fehlendem RTS

  • Du berechnest das Cycle-Budget für eine Rasterzeile mit 4 aktiven Sprites und einer Badline

Übungsaufgaben

Übung 15.1 — Badline-Karte erstellen

Schreibe ein ACME-Programm, das alle sichtbaren Rasterzeilen ($30–$F7) durchläuft und für jede Zeile prüft, ob sie eine Badline ist (gegeben Standard-YSCROLL=3). Zeige Badlines als rote, normale Zeilen als grüne Punkte auf dem Bildschirm. Ändere dann YSCROLL auf 7 und prüfe welche Zeilen sich ändern.

Übung 15.2 — Page-Cross-Benchmark

Implementiere eine Sinustabellen-Lookup-Schleife (256 Iterationen: LDA sin_table,X). Lege die Tabelle zuerst nicht page-aligned ab. Messe die Ausführungszeit mit dem cycle_measure-Makro. Lege die Tabelle dann page-aligned ab und messe erneut. Dokumentiere die Differenz in Zyklen und berechne die Ersparnis pro Sekunde bei 50 Hz.

Übung 15.3 — Stack-Canary in der Praxis

Implementiere stack_guard in einem bestehenden Projekt. Löse dann absichtlich einen kontrollierten Stack-Overflow aus: Schreibe eine rekursive Subroutine ohne Tiefenbegrenzung und rufe sie auf. Beobachte wie der Canary-Check im VICE-Monitor anschlägt und wie das SP-Register aussieht. Protokolliere den Stack-Inhalt via m 0100 01FF.

Übung 15.4 — Den Buggy Platformer debuggen

Das Kapitel enthält fünf beschriebene Bugs. Implementiere alle fünf absichtlich in einem eigenen Plattformer-Gerüst (oder nutze das Beispielprojekt aus stufe_11). Debugge jeden Bug ausschließlich mit VICE-Monitor-Mitteln — keine Code-Inspektion ohne vorherige Messung. Halte jeden Debug-Schritt in einer Protokolldatei fest: Symptom, Watchpoint-Kommando, Monitor-Ausgabe, Fix.
